## Novos recursos no MySQL

### Tipo JSON

Quando muitos desenvolvedores usam bancos de dados relacionais para persistência de dados, muitas vezes sentem que falta flexibilidade ao armazenamento estruturado, porque todas as colunas e seus tipos de dados devem ser projetados antecipadamente. Durante o desenvolvimento e a mudança de negócios, se a estrutura da tabela precisar ser modificada, isso será definitivamente algo problemático e doloroso. A partir do MySQL 5.7, o MySQL introduziu suporte para o tipo de dados `JSON`. O MySQL 8.0 resolveu o gargalo de desempenho do registro JSON. Se o tipo JSON for bem usado, ele realmente quebrará a fronteira entre bancos de dados relacionais e bancos de dados não relacionais e trará mais conveniência às operações de persistência de dados.

Os dados JSON geralmente aparecem em duas formas:

1. Objeto JSON

```JSON
{"name": "Luo Hao", "tel": "13122335566", "QQ": "957658"}
```

2. Matriz JSON

```JSON
[1, 2, 3]
```

```JSON
[{"name": "Luo Hao", "tel": "13122335566"}, {"name": "Wang Dachui", "QQ": "123456"}]
```

Onde precisamos usar o tipo JSON? Vejamos um exemplo simples. Agora, muitos produtos oferecem suporte a várias formas de login para usuários, como número de telefone celular, WeChat, QQ, Sina Weibo e assim por diante. Mas, em geral, não pediremos aos usuários que forneçam todas essas informações. Se usarmos a forma de design tradicional, precisaremos projetar muitas colunas para corresponder a muitas formas de login, e também podemos precisar permitir que essas colunas tenham valores nulos. Obviamente esta não é uma escolha muito boa. Por outro lado, se o produto adicionar mais uma forma de login, devemos modificar a estrutura da tabela anterior, o que é ainda mais doloroso. Mas com o tipo JSON, o problema mencionado é resolvido facilmente. Podemos fazer o design mostrado abaixo.

```SQL
CREATE TABLE `tb_test`
(
`user_id`    bigint unsigned,
`login_info` json,
PRIMARY KEY (`user_id`)
);

INSERT INTO `tb_test` 
VALUES 
    (1, '{"tel": "13122335566", "QQ": "654321", "wechat": "jackfrued"}'),
    (2, '{"tel": "13599876543", "weibo": "wangdachui123"}');
```

Se precisar consultar o número de telefone e o ID do WeChat de um usuário, você pode usar o SQL mostrado abaixo.

```SQL
SELECT `user_id`
     , JSON_UNQUOTE(JSON_EXTRACT(`login_info`, '$.tel')) AS phone
     , JSON_UNQUOTE(JSON_EXTRACT(`login_info`, '$.wechat')) AS wechat
FROM `tb_test`;
```

```
+---------+-------------+-----------+
| user_id | phone       | wechat    |
+---------+-------------+-----------+
|       1 | 13122335566 | jackfrued |
|       2 | 13599876543 | NULL      |
+---------+-------------+-----------+
```

Como o MySQL suporta o tipo JSON, ele também fornece um conjunto correspondente de funções para lidar com dados JSON, como as funções `json_extract` e `json_unquote` usadas acima. Claro, o SQL acima também possui uma maneira mais conveniente de escrevê-lo:

```SQL
SELECT `user_id`
     , `login_info` ->> '$.tel' AS phone
     , `login_info` ->> '$.wechat' AS wechat
  FROM `tb_test`;
```

Vejamos outro exemplo. Se nosso produto precisar criar perfis de usuário adicionando tags e, em seguida, recomendar serviços de plataforma ou bens de consumo com base em perfis de usuário, também podemos usar o tipo JSON para salvar dados de perfil de usuário. O código de exemplo é mostrado abaixo.

Crie a tabela de tags:

```SQL
CREATE TABLE `tb_tags`
(
`tag_id`   int unsigned NOT NULL COMMENT 'Tag ID',
`tag_name` varchar(20)  NOT NULL COMMENT 'Tag name',
PRIMARY KEY (`tag_id`)
);

INSERT INTO `tb_tags` (`tag_id`, `tag_name`) 
VALUES
    (1, 'Born in the 1970s'),
    (2, 'Born in the 1980s'),
    (3, 'Born in the 1990s'),
    (4, 'Born in the 2000s'),
    (5, 'Likes sports'),
    (6, 'Highly educated'),
    (7, 'Middle class'),
    (8, 'Owns a house'),
    (9, 'Owns a car'),
    (10, 'Likes movies'),
    (11, 'Likes online shopping'),
    (12, 'Frequently orders takeout');
```

Armazene tags de usuário:

```SQL
CREATE TABLE `tb_users_tags`
(
`user_id`   bigint unsigned NOT NULL COMMENT 'User ID',
`user_tags` json            NOT NULL COMMENT 'User tags'
);

INSERT INTO `tb_users_tags`
VALUES
    (1, '[2, 6, 8, 10]'),
    (2, '[3, 10, 12]'),
    (3, '[3, 8, 9, 11]');
```

A seguir, vamos entender as partes inteligentes do tipo JSON por meio de um grupo de consultas.

1. Usuários que gostam de filmes (`10`):

```SQL
SELECT `user_id`
  FROM `tb_users_tags`
 WHERE 10 MEMBER OF (`user_tags`->'$');
```

2. Usuários que gostam de filmes e nasceram na década de 1980:

```SQL
SELECT `user_id`
  FROM `tb_users_tags`
 WHERE JSON_CONTAINS(`user_tags`->'$', '[2, 10]');
```

3. Usuários que são amantes do cinema, nascidos na década de 1980 ou na década de 1990:

```SQL
SELECT `user_id`
  FROM `tb_users_tags`
 WHERE JSON_OVERLAPS(user_tags->'$', '[2, 3, 10]');
```

> **Observação**: As consultas acima usam o predicado `member of` e duas funções JSON. `json_contains` pode verificar se uma matriz JSON contém os elementos especificados, enquanto `json_overlaps` pode verificar se uma matriz JSON tem uma parte sobreposta à matriz especificada.

### Funções de janela

A partir do MySQL 8.0, o MySQL suporta funções de janela. A maioria dos bancos de dados comerciais e alguns bancos de dados de código aberto já forneciam suporte para funções de janela. Alguns também as chamam de funções OLAP (processamento analítico online). Pelo nome podemos saber que estão relacionados com estatística e análise. Para ajudar todos a entender as funções das janelas, vamos primeiro falar sobre a ideia de janela.

Uma janela pode ser entendida como um conjunto de registros. Uma função de janela é uma função especial executada em um conjunto de registros que atende a alguma condição, e para cada registro a função é executada dentro desta janela. As funções de janela são fáceis de confundir com as funções agregadas de que falamos acima. A principal diferença entre eles é que as funções agregadas combinam muitos registros em um registro, enquanto as funções de janela são executadas para cada registro e o número de registros não muda após a execução. As funções da janela não são apenas algumas funções. Eles são um conjunto completo de sintaxe e as funções são apenas uma parte dessa sintaxe. A sintaxe básica é mostrada abaixo:

```SQL
<window_function> OVER (PARTITION BY <grouping_columns> ORDER BY <sorting_columns> ROWS BETWEEN ... AND ...)
<window_function> OVER (PARTITION BY <grouping_columns> ORDER BY <sorting_columns> RANGE BETWEEN ... AND ...)
```

Na sintaxe acima, a posição da função de janela pode ser preenchida pelos dois tipos de funções a seguir:

1. Funções de janela dedicadas, como `lead`, `lag`, `first_value`, `last_value`, `rank`, `dense_rank` e `row_number`.
2. Funções agregadas como `sum`, `avg`, `max`, `min` e `count`.

Abaixo damos alguns exemplos simples de uso de funções de janela. Usamos diretamente o banco de dados `hrs` criado na lição anterior.

Exemplo 1: Consulte funcionários classificados do 4º ao 6º lugar por salário.

```SQL
SELECT * 
  FROM (SELECT `ename`
             , `sal`
             , ROW_NUMBER() over (ORDER BY `sal` DESC) AS `rk`
	      FROM `tb_emp`) AS `temp`
 WHERE `rk` between 4 and 6;
```

> **Observação**: A função `row_number()` usada acima pode gerar um número de linha para cada registro. No trabalho real, pode ser substituído por `rank()` ou `dense_rank()` de acordo com a necessidade. A diferença entre os três pode ser entendida lendo a documentação oficial ou [este artigo](https://zhuanlan.zhihu.com/p/92654574). Nas versões do MySQL anteriores à 8, podemos concluir uma operação semelhante da seguinte maneira.

```SQL
select `rank`, `ename`, `sal` from (
       select @a:=@a+1 as `rank`, `ename`, `sal` 
       from `tb_emp`, (select @a:=0) as t1 order by `sal` desc
) as `temp` where `rank` between 4 and 6;
```

Exemplo 2: Consulte os nomes e nomes dos departamentos dos dois funcionários mais bem pagos em cada departamento.

```SQL
select `ename`, `sal`, `dname` 
from (
    select 
        `ename`, `sal`, `dno`,
        rank() over (partition by `dno` order by `sal` desc) as `rank`
    from `tb_emp`
) as `temp` natural join `tb_dept` where `rank`<=2;
```

> **Nota**: Nas versões do MySQL anteriores à 8, podemos concluir uma operação semelhante da seguinte maneira.

```SQL
select `ename`, `sal`, `dname` from `tb_emp` as `t1` 
natural join `tb_dept` 
where (
        select count(*) from `tb_emp` as `t2` 
        where `t1`.`dno`=`t2`.`dno` and `t2`.`sal`>`t1`.`sal` 
)<2 order by `dno` asc, `sal` desc;
```
