## Arquitetura RESTful e DRF Avançado

Além do método discutido na lição anterior, o uso de DRF para criar APIs de dados no estilo REST também pode ser feito com CBV, ou seja, visualizações baseadas em classes. A característica de criar APIs de dados com CBV é que o código é simples e a eficiência de desenvolvimento é alta, mas não é tão flexível quanto o FBV, ou seja, visualizações baseadas em funções. Isso ocorre porque, ao usar o FBV, o código que a função de visualização executa e os dados que ela retorna podem ser altamente personalizados. Abaixo, usamos a API de dados do assunto como exemplo para explicar como personalizar uma API de dados por meio da CBV.

### Usando CBV

#### Herdando de uma subclasse de `APIView`

Modifique `polls/views.py` no projeto anterior, remova a função de visualização `show_subjects` e adicione uma classe chamada `SubjectView`. Esta classe herda de `ListAPIView`. `ListAPIView` pode receber solicitações GET. Ele envolve o método `get` para obter uma lista de dados e retornar dados JSON. `ListAPIView` é uma subclasse de `APIView`. `APIView` possui muitas outras subclasses. Por exemplo, `CreateAPIView` pode oferecer suporte a solicitações POST, `UpdateAPIView` pode oferecer suporte a solicitações PUT e PATCH e `DestroyAPIView` pode oferecer suporte a solicitações DELETE. O código `SubjectView` é mostrado abaixo.

```python
from rest_framework.generics import ListAPIView


class SubjectView(ListAPIView):
    # Use queryset to specify how to get subject data
    queryset = Subject.objects.all()
    # Use serializer_class to specify how to serialize subject data
    serializer_class = SubjectSerializer
```

Como acabamos de dizer, como a classe pai `ListAPIView` de `SubjectView` já implementou o método `get` para lidar com solicitações GET para obter a lista de assuntos, só precisamos declarar como obter os dados do assunto e como serializar os dados do assunto. O primeiro é especificado pelo atributo `queryset` e o último é especificado pelo atributo `serializer_class`. Para usar o `SubjectView` acima, precisamos modificar o arquivo `urls.py`, conforme mostrado abaixo.

```python
urlpatterns = [
    path('api/subjects/', SubjectView.as_view()),
]
```

Claramente, esta forma é muito mais simples do que a forma FBV mencionada anteriormente.

#### Herdando de `ModelViewSet`

Se a API de dados para assuntos precisar suportar solicitações GET, POST, PUT, PATCH e DELETE, para que possa suportar a obtenção, criação, atualização e exclusão de recursos do assunto, uma maneira mais simples é herdar de `ModelViewSet` para gravar a classe de visualização do assunto. Modifique `polls/views.py` novamente, remova a classe `SubjectView` e adicione uma classe chamada `SubjectViewSet`. O código é mostrado abaixo.

```python
from rest_framework.viewsets import ModelViewSet


class SubjectViewSet(ModelViewSet):
    queryset = Subject.objects.all()
    serializer_class = SubjectSerializer
```

Verificando o código-fonte da classe `ModelViewSet`, podemos descobrir que esta classe possui 6 classes pai. As primeiras 5 classes pai implementam suporte para POST, que significa criar um assunto, GET, que significa obter o assunto especificado, PUT/PATCH, que significa atualizar um assunto, DELETE, que significa excluir um assunto, e GET, que significa obter a lista de assuntos. Os métodos correspondentes são `create`, `retrieve`, `update`, `destroy` e `list`. Como as classes pai de `ModelViewSet` já implementaram esses métodos, quase terminamos todo o conjunto de APIs de dados de assunto sem escrever muito código. O que precisamos fazer é apenas apontar como obter os dados, especificados através do atributo `queryset`, e como serializar os dados, especificados através do atributo `serializer_class`. Este ponto é igual ao anterior, que herda de uma subclasse de `APIView`.

```python
class ModelViewSet(mixins.CreateModelMixin,
                   mixins.RetrieveModelMixin,
                   mixins.UpdateModelMixin,
                   mixins.DestroyModelMixin,
                   mixins.ListModelMixin,
                   GenericViewSet):
    """
    A viewset that provides default `create()`, `retrieve()`, `update()`,
    `partial_update()`, `destroy()` and `list()` actions.
    """
    pass
```

Para usar o `SubjectViewSet` acima, o mapeamento de URL precisa ser feito no arquivo `urls.py`. Como `ModelViewSet` é como uma coleção de múltiplas funções de visualização, isso é diferente da forma de mapeamento de URL usada anteriormente. Precisamos primeiro criar um roteador e registrar `SubjectViewSet` por meio dele e, em seguida, adicionar os URLs gerados após o registro bem-sucedido à lista `urlpatterns` juntos. O código é mostrado abaixo.

```python
from rest_framework.routers import DefaultRouter

router = DefaultRouter()
router.register('api/subjects', SubjectViewSet)
urlpatterns += router.urls
```

Além da classe `ModelViewSet`, o DRF também fornece uma classe chamada `ReadOnlyModelViewSet`. Como o nome mostra, esta classe é uma coleção de visualizações somente leitura, o que significa que a API de dados personalizada por herança dela só pode suportar solicitações GET, ou seja, solicitações para obter um único recurso e uma lista de recursos.

### Paginação de dados

Ao usar solicitações GET para obter uma lista de recursos, geralmente não carregamos todos os dados de uma vez, a menos que a quantidade de dados seja realmente pequena. A maioria das operações que obtêm listas de recursos suportam paginação de dados, o que significa que podemos obter dados diferentes especificando o número da página, ou algo como o número da página, e o tamanho da página, o que significa quantos dados carregar de uma vez. Podemos implementar a paginação fatiando o objeto `QuerySet` e também podemos usar os objetos `Paginator` e `Page` agrupados pela estrutura Django. Ao usar DRF, podemos modificar `REST_FRAMEWORK` no arquivo de configuração do Django e configurar a classe de paginação padrão e o tamanho da página para implementar a paginação, conforme mostrado abaixo.

```python
REST_FRAMEWORK = {
    'PAGE_SIZE': 10,
    'DEFAULT_PAGINATION_CLASS': 'rest_framework.pagination.PageNumberPagination'
}
```

Além do paginador `PageNumberPagination` configurado acima, a DRF também fornece os paginadores `LimitOffsetPagination` e `CursorPagination`. Uma coisa que vale a pena mencionar é `CursorPagination`. Isso pode evitar a exposição da quantidade total de dados do site ao usar a paginação por número de página. Os leitores interessados ​​​​podem investigar por conta própria. Se não quiser usar as configurações de paginação padrão no arquivo de configuração, você pode adicionar um atributo `pagination_class` na classe de visualização para especificar outro paginador novamente. Normalmente, este atributo pode ser definido para um paginador personalizado, conforme mostrado abaixo.

```python
from rest_framework.pagination import PageNumberPagination


class CustomizedPagination(PageNumberPagination):
    # Default page size
    page_size = 5
    # Query parameter corresponding to page size
    page_size_query_param = 'size'
    # Maximum page size
    max_page_size = 50
```

```python
class SubjectView(ListAPIView):
    # Specify how to get data
    queryset = Subject.objects.all()
    # Specify how to serialize data
    serializer_class = SubjectSerializer
    # Specify how to paginate
    pagination_class = CustomizedPagination
```

Se não desejar a paginação de dados, você pode definir o atributo `pagination_class` como `None` para cancelar o paginador padrão.

### Filtragem de dados

Se você quiser usar o CBV para personalizar a API de dados para obter informações do professor, isso também pode ser feito herdando de `ListAPIView`. Mas como as informações dos professores precisam ser obtidas de acordo com a disciplina especificada, os dados dos professores precisam ser filtrados em vez de obter todos os dados dos professores diretamente. Caso queira obter o número da disciplina da requisição e filtrar os professores através do número da disciplina, você pode fazer isso substituindo o método `get_queryset`. O código é mostrado abaixo.

```python
class TeacherView(ListAPIView):
    serializer_class = TeacherSerializer

    def get_queryset(self):
        queryset = Teacher.objects.defer('subject')
        try:
            sno = self.request.GET.get('sno', '')
            queryset = queryset.filter(subject__no=sno)
            return queryset
        except ValueError:
            raise Http404('No teachers found.')
```

Além do método acima, também podemos usar a biblioteca de terceiros `django-filter` junto com DRF para implementar a filtragem de dados. Depois de usar `django-filter`, podemos oferecer suporte à filtragem de dados configurando o atributo `filter-backends` para a classe de visualização e especificando `DjangoFilterBackend`. Após a conclusão da configuração acima, podemos usar o atributo `filter_fields` ou o atributo `filterset_class` para especificar como filtrar os dados. Os leitores interessados ​​podem estudá-lo por conta própria.
