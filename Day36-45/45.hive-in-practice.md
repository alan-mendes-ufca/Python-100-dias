## Hive na prática

### Visão geral da colmeia

[Colmeia](https://hive.apache.org/) é uma ferramenta de data warehouse baseada em Hadoop que foi de código aberto pelo Facebook e agora é mantida pela Apache Software Foundation. É a solução de big data mais amplamente utilizada.

Simplificando, o Hive pode transformar consultas SQL em tarefas MapReduce ou Spark. Ele oferece suporte perfeito para SQL e pode fazer estatísticas de big data de maneira muito conveniente, de modo que pessoas que não podem usar Java ou Scala também possam usar a plataforma de big data e o sistema de arquivos distribuído para armazenar e processar dados.

<img src="res/what_hive_can_do.png" style="zoom:50%;">

<img src="res/HADOOP-ECOSYSTEM-Edureka.png">

As principais funções do Hive incluem:

1. Mapeamento de arquivos de dados estruturados para tabelas.
2. Fornecendo HiveQL, uma linguagem de consulta semelhante a SQL para análise em larga escala.
3. Executando em HDFS e mecanismos de execução distribuídos, como MapReduce, Spark e Tez.

As características do Hive são mostradas abaixo:

1. A sintaxe semelhante ao SQL reduz a barreira de aprendizagem.
2. Boa escalabilidade para dados em nível de PB.
3. Suporta vários formatos de armazenamento, como Texto, ORC, Parquet e Avro.
4. Funciona com diferentes mecanismos de execução.
5. Alta latência em comparação com bancos de dados OLTP, por isso é melhor para análises offline do que cargas de trabalho transacionais de baixa latência.

### Sintaxe Básica do Hive

1. Exclua um banco de dados.

```hive
DROP DATABASE IF EXISTS eshop CASCADE;
```

2. Crie um banco de dados.

```hive
CREATE DATABASE IF NOT EXISTS eshop;
```

3. Mude o contexto.

```hive
USE eshop;
```

4. Crie uma tabela externa.

```hive
CREATE EXTERNAL TABLE IF NOT EXISTS dim_user_info 
(
user_id           STRING,
user_name         STRING, 
sex               STRING,
age               INT,
city              STRING,
first_active_time STRING,
level             INT,
extra1            STRING,
extra2            MAP<STRING, STRING>
)
ROW FORMAT DELIMITED 
FIELDS TERMINATED BY '\t'
COLLECTION ITEMS TERMINATED BY ','
MAP KEYS TERMINATED BY ':'
LINES TERMINATED BY '\n'
STORED AS TEXTFILE;
```

5. Carregar dados.

```hive
LOAD DATA LOCAL INPATH '/home/hadoop/data/user_info/user_info.txt' OVERWRITE INTO TABLE dim_user_info;
LOAD DATA INPATH '/user/data/user_info.txt' OVERWRITE INTO TABLE dim_user_info;
```

6. Crie uma tabela particionada.

```hive
CREATE TABLE IF NOT EXISTS fact_user_trade 
(
user_name      STRING,
piece          INT,
price          DOUBLE,
pay_amount     DOUBLE,
goods_category STRING,
pay_time       BIGINT
)  
PARTITIONED BY (dt STRING)
ROW FORMAT DELIMITED
FIELDS TERMINATED BY '\t'
LINES TERMINATED BY '\n'
STORED AS TEXTFILE;
```

Habilite partições dinâmicas:

```hive
SET hive.exec.dynamic.partition=true;
SET hive.exec.dynamic.partition.mode=nonstrict;
SET hive.exec.max.dynamic.partitions=1000;
SET hive.exec.max.dynamic.partitions.pernode=10000;
```

Reparar partições:

```hive
MSCK REPAIR TABLE fact_user_trade;
```

### Exemplos de consulta

```hive
-- Turn on local mode auto judgment
SET hive.exec.mode.local.auto=true;
-- Set the number of MAPREDUCE files
SET hive.exec.mode.local.auto.input.files.max=128;
-- Set the size of MAPREDUCE files
SET hive.exec.mode.local.auto.input.bytes.max=134217728;
-- Set the number of Reduce tasks to 1
SET mapreduce.job.reduces=1;

-- Query the first 10 female users in Beijing
SELECT user_name 
  FROM dim_user_info
 WHERE city='beijing'
       AND sex='female'
 LIMIT 10;

-- Query the user name, purchase quantity, and payment amount of users who bought food goods on 2019-03-24
SELECT user_name
     , piece
     , pay_amount
  FROM fact_user_trade
 WHERE dt='2019-03-24'
       AND goods_category='food';

-- Count the total payment amount of user ELLA in 2018 and the number of days between the most recent and earliest consumption
SELECT SUM(pay_amount) AS total
     , DATEDIFF(
           MAX(FROM_UNIXTIME(pay_time, 'yyyy-MM-dd')),
           MIN(FROM_UNIXTIME(pay_time, 'yyyy-MM-dd'))
       ) AS gap_days
  FROM fact_user_trade
 WHERE YEAR(dt)='2018'
       AND user_name='ELLA';
```

### Agrupamento e agregação

```hive
SELECT goods_category
     , COUNT(DISTINCT user_name) AS total_user
     , SUM(pay_amount) AS total_pay
  FROM fact_user_trade
 WHERE dt BETWEEN '2019-01-01' AND '2019-04-30'
 GROUP BY goods_category;

SELECT user_name
     , SUM(pay_amount) AS total
  FROM fact_user_trade
 WHERE dt BETWEEN '2019-04-01' AND '2019-04-30'
 GROUP BY user_name
HAVING SUM(pay_amount) > 50000;

SELECT COUNT(t.user_name)
  FROM (SELECT user_name
             , COUNT(DISTINCT goods_category) AS total
          FROM fact_user_trade
         WHERE YEAR(dt)='2018'
         GROUP BY user_name
        HAVING COUNT(DISTINCT goods_category)>2) AS t;

SELECT user_name
     , SUM(pay_amount) AS total
  FROM fact_user_trade
 WHERE dt BETWEEN '2019-04-01' AND '2019-04-30'
 GROUP BY user_name
 ORDER BY total DESC
 LIMIT 5;
```

Estatísticas por segmento etário:

```hive
SELECT CASE WHEN age < 20 THEN 'Under 20' 
            WHEN age < 30 THEN 'Under 30' 
            WHEN age < 40 THEN 'Under 40' 
            ELSE '40 and above' 
        END AS age_seg, 
       COUNT(DISTINCT user_id) AS total 
  FROM dim_user_info 
 GROUP BY CASE WHEN age < 20 THEN 'Under 20' 
               WHEN age < 30 THEN 'Under 30' 
               WHEN age < 40 THEN 'Under 40' 
               ELSE '40 and above' 
           END;

SELECT age_seg,
       IF(marriage_status = 1, 'Married', 'Unmarried') AS marriage_status,
       COUNT(*) AS total
  FROM (SELECT CASE WHEN age < 20 THEN 'Under 20'
                    WHEN age < 30 THEN '20-30'
                    WHEN age < 40 THEN '30-40'
                    ELSE '40 and above'
                END AS age_seg,
               extra2['marriage_status'] AS marriage_status
          FROM dim_user_info
         WHERE TO_DATE(first_active_time) BETWEEN '2018-01-01' AND '2018-12-31'
       ) AS t
 WHERE age_seg in ('20-30', '30-40')
 GROUP BY age_seg, IF(marriage_status = 1, 'Married', 'Unmarried');
```

Colete categorias em matrizes ou mapas:

```hive
SELECT user_name,
       COLLECT_SET(goods_category) AS categories
  FROM fact_user_trade
 GROUP BY user_name;

SELECT user_name
     , CONCAT_WS(', ', COLLECT_SET(goods_category)) AS categories
  FROM fact_user_trade
 GROUP BY user_name;

SELECT user_name
     , STR_TO_MAP(CONCAT_WS(',', COLLECT_LIST(CONCAT(goods_category, ':', cnt)))) AS category_cnt_map
  FROM (SELECT user_name
             , goods_category
             , COUNT(*) AS cnt
          FROM fact_user_trade
         GROUP BY user_name, goods_category) AS t
 GROUP BY user_name;
```

### Vista lateral/explodir

```hive
CREATE OR REPLACE VIEW v_user_categories
AS
SELECT user_name
     , COLLECT_SET(goods_category) AS categories
  FROM fact_user_trade
 WHERE dt BETWEEN '2019-04-01' and '2019-04-30' 
 GROUP BY user_name;

CREATE OR REPLACE VIEW v_user_categories_map
AS
SELECT user_name
     , STR_TO_MAP(CONCAT_WS(',', COLLECT_LIST(CONCAT(goods_category, ':', cnt)))) AS category_cnt_map
  FROM (SELECT user_name
             , goods_category
             , COUNT(*) AS cnt
          FROM fact_user_trade
         WHERE dt BETWEEN '2019-04-01' AND '2019-04-30' 
         GROUP BY user_name, goods_category) AS t
 GROUP BY user_name;

SELECT user_name
     , category
  FROM v_user_categories
LATERAL VIEW EXPLODE(categories) t AS category;

SELECT user_name
     , category
     , cnt
  FROM v_user_categories_map
LATERAL VIEW EXPLODE(category_cnt_map) t AS category, cnt;
```

### Conceitos de criação de tabela

#### Tipos de dados

Os tipos de dados Hive geralmente se enquadram em categorias primitivas e complexas.

| Tipo | Descrição | Uso típico |
| --- | --- | --- |
| `TINYINT` | Inteiro assinado de 1 byte |   |
| `SMALLINT` | Inteiro assinado de 2 bytes |   |
| `INT` | Inteiro assinado de 4 bytes |   |
| `BIGINT` | Inteiro assinado de 8 bytes | IDs, contadores |
| `BOOLEAN` | `TRUE` / `FALSE` |   |
| `FLOAT` | Ponto flutuante de precisão única |   |
| `DOUBLE` | Ponto flutuante de dupla precisão |   |
| `DECIMAL(P,S)` | Decimal de precisão fixa | Dinheiro |
| `STRING` | String de comprimento variável | Texto, JSON |
| `CHAR` | String de comprimento fixo |   |
| `VARCHAR` | String com comprimento limitado |   |
| `BINARY` | Dados binários |   |
| `TIMESTAMP` | Carimbo de data e hora |   |
| `DATE` | Data |   |
| `INTERVAL` | Intervalo de tempo |   |
| `STRUCT` | Objeto composto | Cargas de evento |
| `MAP` | Coleta de valores-chave | Atributos |
| `ARRAY` | Coleção ordenada do mesmo tipo | Listas |

Exemplo:

```hive
CREATE TABLE complex
(
  c1 ARRAY<INT>,
  c2 MAP<STRING, INT>,
  c3 STRUCT<a:STRING, b:INT, c:DOUBLE>,
  c4 STRUCT<name:STRING, addrs:ARRAY<STRUCT<city:STRING, zipcode:STRING>>>
);

SELECT c1[0]
     , c2['key']
     , c3.b
     , c4.addrs[0].city
  FROM complex;
```

O Hive suporta conversão de tipo implícita e explícita:

```hive
SELECT CAST('123' AS INT);
SELECT CAST(3.14159 AS DECIMAL(5,2));
```

#### Tipos de tabela

| Tipo de tabela | Significado |
| --- | --- |
| Tabela gerenciada | O Hive gerencia metadados e dados |
| Mesa externa | O Hive gerencia apenas metadados; os dados permanecem onde estão |
| Tabela particionada | Os dados são divididos em partições baseadas em diretório |
| Mesa em balde | Os dados são criptografados em buckets fixos |
| Mesa temporária | Dados temporários com escopo de sessão |
| Visualizar | Tabela virtual baseada em uma consulta |

#### `ROW FORMAT`

As opções comuns incluem texto delimitado, `OpenCSVSerde`, `RegexSerDe`, `JsonSerDe`, `AvroSerDe`, `ParquetHiveSerDe` e `OrcSerde`.

> **Observação**: geralmente você especifica um SerDe explicitamente apenas ao usar um formato não padrão, como CSV, JSON ou análise baseada em Regex.

### Padrões de gravação de dados

Padrões de escrita comuns incluem:

- `INSERT INTO`: anexar dados
- `INSERT OVERWRITE`: substituir dados de destino
- `LOAD DATA`: mova os arquivos existentes para o local da tabela Hive
- `LOAD DATA LOCAL`: carregar do sistema de arquivos local
- `CREATE TABLE AS SELECT`: crie uma tabela a partir de um resultado de consulta
- `INSERT OVERWRITE DIRECTORY`: exportar resultados da consulta
- inserção de partição dinâmica

| `map_keys(map)` | Devolva todas as chaves | `map_keys(map('a',1,'b',2)) = ['a','b']` |
| `map_values(map)` | Retornar todos os valores | `map_values(map('a',1,'b',2)) = [1,2]` |
| `sort_array(array)` | Classificar uma matriz | `sort_array(array(3,1,2)) = [1,2,3]` |
| `array_contains(array, val)` | Verifique se existe um elemento | `array_contains(array('a','b'),'a') = true` |

#### Funções de janela

As funções de janela são comumente usadas para classificação, cálculos cumulativos e cálculos dentro de um grupo.

| Função | Descrição | Exemplo |
| --- | --- | --- |
| `row_number()` | Número da linha dentro de uma partição | `row_number() over(partition by dept order by salary desc)` |
| `rank()` | Classificação com lacunas nos empates | `rank() over(order by score desc)` |
| `dense_rank()` | Classificação sem lacunas nos empates | `dense_rank() over(order by score desc)` |
| `lag(col,n,default)` | Valor `n` linhas anteriores | `lag(salary,1,0) over(order by id)` |
| `lead(col,n,default)` | Valor `n` linhas após | `lead(salary,1,0) over(order by id)` |
| `first_value(col)` | Primeiro valor dentro da janela | `first_value(salary) over(order by id)` |
| `last_value(col)` | Último valor dentro da janela | `last_value(salary) over(order by id)` |
| `sum(col) over(...)` | Soma cumulativa em janela | `sum(sales) over(partition by region order by month)` |

#### Exemplos de aplicação

1. `FROM_UNIXTIME`: converte um carimbo de data/hora em uma sequência de data.

```hive
SELECT FROM_UNIXTIME(pay_time, 'yyyy-MM-dd hh:mm:ss')
  FROM fact_user_trade LIMIT 10;
```

2. `DATEDIFF`: calcula a diferença entre duas datas.

```hive
SELECT user_name, DATEDIFF('2019-4-1', to_date(firstactivetime))
  FROM dim_user_info LIMIT 10;
```

3. `IF`: retorna valores diferentes com base em uma condição.

```hive
SELECT sex
     , IF(level > 5, 'high', 'low') AS level_type
     , COUNT(DISTINCT user_id) AS total
  FROM dim_user_info
 GROUP BY sex, IF(level > 5, 'high', 'low');
```

4. `SUBSTR`: extraia uma substring.

```hive
SELECT SUBSTR(first_active_time, 1, 7) AS month
     , COUNT(DISTINCT user_id) AS total
  FROM dim_user_info
 GROUP BY substr(first_active_time, 1, 7);
```

5. `GET_JSON_OBJECT`: extraia o valor de uma chave especificada de uma string JSON, como `GET_JSON_OBJECT(info, '$.first_name')`.

```hive
SELECT GET_JSON_OBJECT(extra1, '$.phonebrand') AS phone
     , COUNT(DISTINCT user_id) AS total
  FROM user_info
 GROUP BY GET_JSON_OBJECT(extra1, '$.phonebrand');
```

> **Nota**: A função MySQL correspondente é denominada `JSON_EXTRACT`.

### Agrupamento e agregação

| Operação | Sintaxe de exemplo | Descrição | Características do resultado | Caso de uso típico |
| --- | --- | --- | --- | --- |
| `GROUP BY` | `SELECT region, product, SUM(amount) FROM sales GROUP BY region, product;` | Agrupar pelas colunas especificadas | Agregação de saídas para combinação de uma dimensão | Resumos unidimensionais ou de combinação fixa |
| `GROUPING SETS` | `SELECT region, product, SUM(amount) FROM sales GROUP BY GROUPING SETS ((region, product), (region), (product), ());` | Especifique várias combinações de agrupamento de uma só vez | Produz várias combinações escolhidas e pode incluir totais | Relatórios onde apenas as combinações selecionadas são importantes |
| `CUBE` | `SELECT region, product, SUM(amount) FROM sales GROUP BY CUBE(region, product);` | Gere automaticamente todas as combinações de dimensões | Produz todas as combinações de dimensões (`2^n`) | Análise multidimensional completa no estilo OLAP |

### Operações de Amostragem

| Estilo de amostragem | Sintaxe de exemplo | Significado central | aleatório | Repetível | Precisão/viés | Requisitos | Desempenho e notas | Uso típico |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| Amostragem física de balde | `SELECT * FROM t TABLESAMPLE(BUCKET 2 OUT OF 8);` | Escolha o segundo balde físico entre oito | Não | Estável | Segue o agrupamento de tabelas | A tabela deve ser agrupada | Lê apenas alguns arquivos, por isso é rápido | Amostragem proporcional em tabelas grandes |
| Amostragem lógica de intervalo | `SELECT * FROM t TABLESAMPLE(BUCKET 1 OUT OF 4 ON user_id);` | Hash `user_id` em quatro buckets lógicos e pegue o bucket um | Pseudo-aleatório | Estável para a mesma chave | Aproximadamente um quarto, afetado pela distorção | Nenhuma tabela agrupada é necessária | A distribuição depende da distorção chave | Divisões consistentes de treinamento/teste |
| Amostragem de Bernoulli | `SELECT * FROM t WHERE RAND() < 0.1;` | Faça uma amostragem independente de cada linha com 10% de probabilidade | Sim | Pode ser consertado com uma semente | Somente proporção esperada | Nenhum | Requer uma verificação completa | Down-amostragem exploratória |
| Tamanho fixo por grupo | `SELECT * FROM (SELECT *, ROW_NUMBER() OVER(PARTITION BY gid ORDER BY RAND()) rn FROM t) x WHERE rn <= 100;` | Escolha aleatoriamente um número fixo de linhas por grupo | Sim | Pode ser consertado com uma semente | Contagem igual de amostras por grupo | Suporte à função de janela | Duas varreduras mais classificação | Amostragem balanceada |
| Amostragem de taxa por grupo | `WHERE RAND() < CASE WHEN gid='A' THEN 0.2 ELSE 0.05 END` | Atribua diferentes taxas de amostragem a cada grupo | Sim | Pode ser consertado com uma semente | Proporção esperada com flutuação intragrupo | Nenhum | Verificação completa | Amostragem ponderada |
| Truncamento `LIMIT` | `SELECT * FROM t LIMIT 1000;` | Pegue as primeiras linhas `N`, não aleatórias | Não | Estável | Enviesado | Nenhum | Mais rápido, mas apenas para visualizações | Desenvolvimento e inspeção de campo |

### Operações de classificação

| Estilo de classificação | Sintaxe de exemplo | Significado central | Redutores | Ordenado globalmente | Comportamento de embaralhar/classificar | Desempenho e notas | Uso típico |
| --- | --- | --- | --- | --- | --- | --- | --- |
| `ORDER BY` | `SELECT * FROM t ORDER BY ts DESC;` | Classificação global | 1 | Sim | Todos os dados vão para um redutor | Mais lento e se torna um gargalo em grandes resultados | Pequenos conjuntos de resultados, exibição final |
| `ORDER BY ... LIMIT` | `SELECT * FROM t ORDER BY score DESC LIMIT 1000;` | Top-N globais | 1 | Sim | Ainda um redutor, mas `LIMIT` ajuda | Aceitável para consultas Top-N | Top-N globais |
| `SORT BY` | `SELECT * FROM t SORT BY ts DESC;` | Classifique apenas dentro de cada redutor | Múltiplo | Não | Cada redutor classifica sua própria partição | Mais rápido que `ORDER BY`, mas não classificado globalmente | Exportação paralela e classificação fragmentada |
| `DISTRIBUTE BY` | `SELECT * FROM t DISTRIBUTE BY key;` | Controlar como as linhas são enviadas para os redutores | Múltiplo | Não | As mesmas chaves vão para o mesmo redutor, sem classificação | Frequentemente emparelhado com `SORT BY` | Particionamento para trabalho posterior |
| `DISTRIBUTE BY + SORT BY` | `SELECT * FROM t DISTRIBUTE BY key SORT BY key, ts DESC;` | A mesma chave em um redutor e classificada dentro desse redutor | Múltiplo | Não, mas solicitado dentro da chave | As linhas da mesma chave permanecem juntas e são classificadas | Padrão comum e prático | Ordenação de série temporal por usuário ou produto |
| `CLUSTER BY` | `SELECT * FROM t CLUSTER BY key;` | Atalho para `DISTRIBUTE BY key` mais `SORT BY key` ascendente | Múltiplo | Não | Açúcar de sintaxe; não consigo escolher a direção | Simples, mas menos flexível | Pré-classificação para gravações ou junções agrupadas |

### Vista Lateral

`LATERAL VIEW` é um recurso de sintaxe do Hive muito importante. É usado principalmente em conjunto com funções de geração de tabela, como `EXPLODE`, para dividir uma linha em várias linhas. Os casos de uso mais comuns envolvem `ARRAY`, `MAP` e campos semelhantes a JSON, e também pode ser usado para transformar uma tabela larga em uma tabela estreita.

| Uso | Sintaxe de exemplo | Descrição | Caso de uso típico |
| --- | --- | --- | --- |
| `LATERAL VIEW explode(array)` | `SELECT order_id, item FROM orders LATERAL VIEW explode(items) t AS item;` | Divida uma matriz em várias linhas, um elemento por linha | Detalhes do pedido, registros de eventos |
| `LATERAL VIEW posexplode(array)` | `SELECT order_id, pos, item FROM orders LATERAL VIEW posexplode(items) t AS pos, item;` | Divida um array e mantenha os índices dos elementos | Análise de clickstream com pedido retido |
| `LATERAL VIEW explode(map)` | `SELECT user_id, k, v FROM user_tags LATERAL VIEW explode(tags) t AS k, v;` | Dividir um mapa em linhas `(key, value)` | Tags de usuário, pares de valores-chave de atributos |
| `LATERAL VIEW json_tuple(json, …)` | `SELECT log_id, ip, device FROM event_log LATERAL VIEW json_tuple(log_json,'ip','device') t AS ip, device;` | Extraia campos do JSON em colunas | Análise de log, dados semiestruturados |
| `LATERAL VIEW inline(array)` | `SELECT order_id, col1, col2 FROM orders LATERAL VIEW inline(item_structs) t AS col1,col2;` | Expanda uma matriz de estruturas em várias linhas e colunas | Linhas incorporadas com vários atributos |
| `LATERAL VIEW stack(n, …)` | `SELECT user_id, subject, score FROM user_score LATERAL VIEW stack(3,'math',math,'eng',english,'phy',physics) t AS subject, score;` | Transforme múltiplas colunas em múltiplas linhas | Transformações amplas para longas |

### HiveSQL x MySQL

| Recurso | Colmeia SQL | SQL MySQL | Diferença |
| --- | --- | --- | --- |
| Particionamento de tabela | `PARTITIONED BY` | Nenhum equivalente direto neste contexto do curso | As partições Hive são importantes para filtragem de grandes dados |
| Agrupamento de mesa | `CLUSTERED BY` | Não compatível | O Hive pode agrupar dados para otimizar operações em tabelas grandes |
| Formatos de armazenamento | `STORED AS` com ORC, Parquet e outros | Mecanismos orientados a linhas, como InnoDB | Hive oferece suporte a formatos colunares para análise |
| Mecanismo de consulta | MapReduce, Tez, Spark | Execução tradicional do mecanismo de banco de dados | Hive pode trocar mecanismos de execução |
| Tratamento de dados complexos | `LATERAL VIEW`, `MAP`, `ARRAY`, `STRUCT` | Nenhum suporte nativo comparável | O Hive lida melhor com dados aninhados e semiestruturados |
| Junte-se à otimização | Suporta estratégias como MapJoin | Junções relacionais padrão | O Hive inclui otimizações de junção específicas para big data |
| Suporte UDF | Suporte a UDF e UDTF | Suporte UDF e recursos do lado do banco de dados, como gatilhos | Hive enfatiza extensibilidade analítica |
| Otimização de consulta | Poda de partição, poda de coluna, inserção de partição dinâmica | Otimização baseada principalmente em índice | A otimização do Hive foi projetada para análises em grande escala |
| Índices | Apoiado, mas geralmente não encorajado | Recurso principal de desempenho | Os índices são muito mais importantes no MySQL |
| Tipos de dados | Inclui `ARRAY`, `MAP`, `STRUCT` | Principalmente tipos escalares | O Hive oferece suporte a tipos estruturados mais ricos |
| Visualizações materializadas | Suportado | Nenhum suporte nativo no uso típico do MySQL | O Hive pode pré-calcular e persistir resultados |

### Otimização de desempenho

Há muitas maneiras de otimizar o desempenho da consulta SQL do Hive, incluindo design de tabela, ajuste de consulta e opções de mecanismo de execução.

| Método de otimização | Descrição | Cenário típico |
| --- | --- | --- |
| Particionamento | Particione uma tabela por um campo como data e use a remoção de partição para verificar menos dados | Tabelas grandes geralmente filtradas por data, área e campos semelhantes |
| Agrupamento | Divida os dados em intervalos fixos usando um valor hash | Junções ou agregações frequentes em uma coluna específica |
| Poda de coluna | Consulte apenas as colunas que você precisa | Consultas que afetam apenas parte de uma tabela ampla |
| Poda de partição | Filtrar colunas de partição em consultas | Tabelas particionadas com filtros de partição |
| Formatos colunares como ORC / Parquet | Use formatos compactados e fáceis de analisar | Varredura e agregação em grande escala |
| MapJoin | Carregue uma pequena tabela na memória e complete a junção no lado do mapa | Junções de mesa pequena a mesa grande |
| Escolhendo o tipo de junção correto | Use sort-merge join ou bucketed join quando apropriado | Ambas as tabelas são agrupadas ou classificadas nas mesmas chaves |
| Escolhendo o mecanismo de execução certo | Use MapReduce, Tez ou Spark de acordo com a carga de trabalho | Varia de acordo com o tipo de consulta e o volume de dados |
| Inserção de partição dinâmica | Ativar `hive.exec.dynamic.partition=true` | Dados que precisam ser inseridos em partições dinamicamente |
| Reduza os dados digitalizados antecipadamente | Use `WHERE` para filtrar o mais cedo possível | Qualquer consulta em conjuntos de dados muito grandes |
| Executar em lotes | Use `LIMIT` ou `TABLESAMPLE` quando a precisão total não for necessária | Exploração de volumes de dados muito grandes |
| Visualizações materializadas | Pré-calcular e armazenar resultados de consultas repetidas | Consultas que são executadas repetidamente |
| Execução paralela | Habilite `hive.exec.parallel=true` quando apropriado | Várias tarefas que podem ser executadas juntas |
