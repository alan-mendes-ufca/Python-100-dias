## Problemas e soluções no desenvolvimento de projetos em equipe

Os termos desenvolvimento individual e desenvolvimento de equipe já devem soar familiares. Desenvolvimento individual significa que uma pessoa controla todas as partes de um produto. Desenvolvimento em equipe significa que várias pessoas se agrupam e completam o desenvolvimento de um produto. Para realizar o desenvolvimento da equipe são indispensáveis os seguintes pontos:

1. Gerencie e compartilhe eventos no processo de desenvolvimento, como quem concluiu o quê e em que horas.
2. Compartilhe resultados de trabalho, conhecimentos recém-adquiridos e habilidades técnicas dentro da equipe.
3. Gerencie as alterações nos resultados do trabalho para que os resultados existentes não sejam danificados, enquanto diferentes membros ainda possam trabalhar em paralelo no estado atual do projeto.
4. Prove que o software produzido pela equipe pode rodar normalmente a qualquer momento.
5. Use fluxos de trabalho automatizados para que os membros da equipe possam realizar o desenvolvimento, os testes e a implantação corretamente.

### Problemas comuns no desenvolvimento de equipes

Comparado com o desenvolvimento individual, o desenvolvimento da equipe facilmente enfrenta os seguintes problemas.

#### Problema 1: A comunicação tradicional não consegue estabelecer claramente a prioridade

Por exemplo, se a equipe se comunica por e-mail, muitas mensagens podem ocultar mensagens importantes. A situação das questões não pode ser gerida com clareza e torna-se difícil dizer quais os problemas que já foram resolvidos e quais os que ainda estão pendentes. Pesquisar mensagens antigas em formato de texto completo também é ineficiente.

Solução: use uma ferramenta de gerenciamento de defeitos.

#### Problema 2: Não há ambiente que possa ser usado para verificação

Por exemplo, depois de receber um relatório de falha do sistema de produção, pode demorar muito para reproduzir o ambiente de produção.

Solução: implementar entrega contínua.

#### Problema 3: as ramificações do projeto são gerenciadas por meio de diretórios de alias

Solução: implementar controle de versão.

#### Problema 4: Reconstruir o banco de dados é muito difícil

Por exemplo, a estrutura da tabela na produção e no desenvolvimento pode não ser consistente ou até mesmo a ordem das colunas de uma tabela pode ser diferente.

Solução: implementar controle de versão.

#### Problema 5: Os problemas não podem ser descobertos a menos que o sistema seja executado

Por exemplo, corrigir um bug pode introduzir outro bug ou causar regressão. O uso incorreto do sistema de controle de versão pode substituir as alterações de outra pessoa e diferentes modificações podem interferir umas nas outras. Se tais problemas não forem descobertos precocemente, rastreá-los meses depois torna-se muito problemático.

Solução: implementar integração contínua e construir e testar os resultados do trabalho da equipe de forma frequente e contínua.

#### Problema 6: o código de um membro substitui a correção de outro membro

Solução: implementar controle de versão.

#### Problema 7: A refatoração não pode ser realizada com segurança

Refatorar significa ajustar a estrutura interna do código sem alterar o resultado produzido, mas a refatoração ainda pode introduzir regressão.

Solução: grande número de testes reutilizáveis e integração contínua.

#### Problema 8: O tempo de correção de bugs não pode ser rastreado e as regressões não podem ser rastreadas

Solução: o controle de versão, o gerenciamento de defeitos e a integração contínua devem interagir entre si e, idealmente, também devem ser integrados a ferramentas de implantação automatizadas.

#### Problema 9: O processo de lançamento é muito complicado

Solução: implementar entrega contínua.

Depois de analisar as questões acima, podemos tirar uma conclusão simples: no desenvolvimento em equipe, o controle de versão, o gerenciamento de defeitos e a integração contínua são muito importantes e não podem ser omitidos.

### Controle de versão

Dos problemas acima, podemos chegar a uma conclusão simples: o controle de versão é o primeiro pré-requisito para o desenvolvimento da equipe. Devemos usar o controle de versão para gerenciar todos os tipos de informações produzidas durante o desenvolvimento do produto, incluindo:

1. Código.
2. Documentos de requisitos e design.
3. Esquemas de banco de dados e dados iniciais.
4. Arquivos de configuração.
5. Definições de dependência para bibliotecas.

#### Uma introdução ao Git

![](res/git-logo.png)

Git é um sistema de controle de versão distribuído de código aberto nascido em 2005. Foi originalmente desenvolvido por Linus Torvalds, o pai do Linux, para ajudar a gerenciar o desenvolvimento do kernel Linux. Ao contrário das ferramentas comuns de controle de versão, como o Subversion, o Git usa um modelo de controle de versão distribuído, de modo que o trabalho de controle de versão ainda pode ser feito mesmo sem o suporte de um servidor central.

Para as pessoas que usaram SVN, Git e SVN compartilham uma ideia importante: ambos abandonaram o antigo modelo de controle de versão baseado em bloqueio e adotaram um modelo mais eficiente baseado em mesclagem. Suas principais diferenças são:

1. O Git é distribuído, enquanto o SVN é centralizado e depende de um servidor central.
2. O Git armazena conteúdo como instantâneos de metadados, enquanto o SVN gerencia metadados de versão por meio de diretórios `.svn`.
3. As ramificações do Git são tratadas de maneira muito diferente das ramificações do SVN.
4. O Git não possui número de revisão global, embora as tags de versão possam ser mantidas.
5. O Git usa hashing SHA-1 para proteger a integridade do conteúdo, o que lhe dá fortes garantias contra corrupção do repositório.

Resumindo, **Git é realmente excelente!!!**

#### Instalando o Git

Você pode encontrar a página de download do Git para seu sistema operacional no [site oficial do Git](http://git-scm.com/) e instalá-lo. Instalar o Git no macOS e no Windows é muito simples. No Linux, se você deseja instalar a versão oficial mais recente, é recomendável compilá-la e instalá-la a partir do código-fonte oficial. As etapas abaixo usam o CentOS como exemplo.

Baixe o arquivo de código-fonte do Git.

```shell
wget https://mirrors.edge.kernel.org/pub/software/scm/git/git-2.23.0.tar.xz
```

Extraia o arquivo.

```shell
xz -d git-2.23.0.tar.xz
tar -xvf git-2.23.0.tar
```

Instale a biblioteca de dependência de baixo nível.

```shell
yum -y install libcurl-devel
```

> **Observação**: Sem esta biblioteca de dependência, os recursos de rede do Git não funcionarão.

Configure antes da instalação.

```shell
cd git-2.23.0
./configure --prefix=/usr/local
```

Construa e instale.

```shell
make && make install
```

Após a instalação ser bem-sucedida, você pode digitar o comando abaixo no terminal para verificar sua versão do Git.

```shell
git --version
```

Se você nunca usou o Git antes, vale a pena ler primeiro um guia para iniciantes e obter uma compreensão aproximada do fluxo de trabalho, como [git - o guia simples](http://www.bootcss.com/p/git-guide/).

#### Operações locais do Git

Você pode transformar uma pasta em um repositório Git com:

```shell
git init
```

Depois de terminar a operação acima, o diretório local fica assim: o lado esquerdo da figura abaixo é o seu diretório de trabalho e o lado direito é o seu repositório local. A parte intermediária é a área de teste entre o diretório de trabalho e o repositório local.

![](./res/git_repository.png)

> **Dica**: Se você usar `ls -la` para visualizar todos os arquivos, descobrirá que após executar o comando acima, há mais uma pasta oculta chamada `.git` no diretório. Este é o repositório Git local.

Depois disso, o diretório local inclui:

- o diretório de trabalho;
- a área de teste entre o diretório de trabalho e o repositório local;
- o repositório `.git` local.

Você pode adicionar um arquivo ou todos os arquivos à área de teste:

```shell
git add <file>
git add .
```

Você pode então inspecionar o estado da árvore de trabalho, da área de preparação e do repositório local:

```shell
git status
```

Se precisar restaurar as alterações na árvore de trabalho do estado de teste, você pode usar:

```shell
git restore <file>
git restore .
```

> **Dica**: Se não quiser adicionar um arquivo à área de teste, você pode seguir a dica e usar `git rm --cached <file>` para colocar o arquivo de volta no diretório de trabalho.

Se esta é a primeira vez que você usa o Git, você deve configurar seu nome de usuário e e-mail antes de confirmar:

```shell
git config --global user.name "jackfrued"
git config --global user.email "jackfrued@126.com"
```

> **Dica**: você pode usar `git config --list` para visualizar informações de configuração do Git.

Então você pode fazer o commit:

```shell
git commit -m 'description of this commit'
```

Você pode visualizar o histórico de commits com:

```shell
git log
git log --graph --oneline --abbrev-commit
```

#### Operações remotas do Git

Embora o Git possa funcionar localmente, um servidor Git central ainda é necessário na colaboração empresarial. As empresas podem usar serviços hospedados como GitHub ou construir seu próprio servidor Git privado, geralmente com GitLab.

Com um servidor Git instalado, as operações remotas permitem que você envie seu próprio trabalho e extraia trabalho de outros. Por exemplo:

```shell
git remote add origin git@github.com:your-org/python.git
git push -u origin master:master
git pull origin master
```

O ponto principal é simples: o controle de versão não é apenas uma ferramenta de backup de código. Em projetos de equipe, é a base para desenvolvimento paralelo, rastreabilidade, colaboração e liberação controlada.

#### Operações de ramificação Git

1. **Criar** e **trocar** ramificações. Os comandos abaixo criam uma ramificação chamada `dev` e alternam para ela.

```shell
git branch <branch-name>
git switch <branch-name>
```

ou

```shell
git switch -c <branch-name>
```

2. **Associe uma ramificação local a uma ramificação remota**.

```shell
git branch --set-upstream-to origin/develop
git branch --set-upstream-to origin/develop <branch-name>
git branch --track <branch-name> origin/develop
git branch --unset-upstream <branch-name>
```

3. **Mesclar ramificações**. Após a conclusão do desenvolvimento na ramificação `dev`, se quisermos mesclá-la em `master`, podemos voltar para `master` e usar `git merge`.

```shell
git switch master
git merge --no-ff dev
```

Por padrão, `git merge` usa mesclagem de avanço rápido. Se quiser manter o histórico da ramificação, você pode usar `--no-ff`. Se ocorrer um conflito, o Git solicitará que você corrija o conflito e faça commit novamente após o conflito ser resolvido.

4. **Rebase de ramificações**. Depois de muitas mesclagens, as ramificações podem ficar confusas e complicadas. Para resolver isso, podemos usar `git rebase`.

![](./res/git-rebase.png)

```shell
git rebase master
git switch master
git merge dev
```

5. **Excluir ramificações**.

```shell
git branch -d <branch-name>
git branch -D <branch-name>
git push origin --delete develop
```

#### Outras operações Git

1. `git fetch`: baixa todas as alterações do repositório remoto.
2. `git diff`: compara diferenças entre a árvore de trabalho, a área de teste, o repositório ou duas ramificações.
3. `git stash`: salva temporariamente as alterações na árvore de trabalho e na área de preparação.

```shell
git stash
git stash list
git stash pop
```

4. `git reset`: volta para uma versão especificada.

![](./res/git-reset.png)

5. `git cherry-pick`: escolhe um commit de outro branch e o traz para o branch atual como um novo commit.
6. `git revert`: reverte um commit.
7. `git tag`: geralmente é usado para visualizar ou adicionar uma tag.

#### Fluxo de trabalho Git

Como o Git é uma ferramenta essencial para o desenvolvimento de equipes, também deve haver um fluxo de trabalho padrão na colaboração em equipe. Caso contrário, mesmo que a ferramenta seja poderosa, os membros da equipe ainda trabalharão desordenadamente e os conflitos estarão por toda parte.

##### Fluxo do Github

1. Clone o código do servidor para o local.

```shell
git clone git@github.com:your-org/python.git
```

2. Crie e mude para seu próprio branch.

```shell
git switch -c <branch-name>
```

3. Desenvolva em sua própria ramificação e faça o controle de versão local lá.
4. Envie sua própria ramificação para o servidor.

```shell
git push origin <branch-name>
```

5. Crie uma pull request ou solicitação de mesclagem online, solicitando a mesclagem de seu trabalho em `master`.

![](./res/gitee-pull-request.png)

Essa estratégia de gerenciamento de ramificações é chamada de **github-flow** ou **PR flow**. É muito simples e fácil de entender. Seu ponto fraco também é óbvio: a ramificação `master` geralmente é o código online atual, mas às vezes o trabalho mesclado em `master` não está pronto para ser lançado imediatamente.

##### Fluxo Git

Além do github-flow, existe outra estratégia de gerenciamento de ramificações chamada **git-flow**. É também um fluxo de trabalho que muitas empresas estão dispostas a usar.

![](./res/git-flow.png)

Neste modelo, o projeto possui duas ramificações de longa duração, `master` e `develop`. As outras ramificações são ramificações de suporte temporárias, incluindo `feature`, `release` e `hotfix`.

![](./res/git-flow-detail.png)

1. Crie uma ramificação `feature` a partir de `develop` e, após a conclusão do trabalho, mescle-a novamente em `develop`.

```shell
git switch -c feature/user develop
git checkout develop
git merge --no-ff feature/user
git branch -d feature/user
git push origin develop
```

2. Crie uma ramificação `release` a partir de `develop` e, após a conclusão do lançamento, mescle-a novamente em `master` e `develop`.

```shell
git checkout -b release-0.1 develop
git push -u origin release-0.1
git checkout master
git merge --no-ff release-0.1
git push
git checkout develop
git merge --no-ff release-0.1
git push
git branch -d release-0.1
git push --delete release-0.1
git tag v0.1 master
git push --tags
```

3. Crie uma ramificação `hotfix` de `master` e, depois que o bug for corrigido, mescle-a de volta para `master` e `develop`.

```shell
git checkout -b hotfix-0.1.1 master
git push -u origin hotfix-0.1.1
git checkout master
git merge --no-ff hotfix-0.1.1
git push
git checkout develop
git merge --no-ff hotfix-0.1.1
git push
git branch -d hotfix-0.1.1
git push --delete hotfix-0.1.1
git tag v0.1.1 master
git push --tags
```

O Git-flow facilita o controle do status da ramificação, mas é muito mais complexo que o github-flow. Portanto, no trabalho real, as pessoas costumam usar ferramentas gráficas do Git ou ajudantes dedicados para simplificar o processo.

### Gerenciamento de defeitos

Sem boas ferramentas de gerenciamento de equipe, o progresso do projeto certamente se tornará difícil e o gerenciamento de tarefas será difícil. Um sistema de gerenciamento de defeitos pode resolver esses problemas. Normalmente, tal sistema tem as seguintes funções:

1. Gerenciamento de tarefas.
2. Histórico claro e pesquisável de problemas anteriores.
3. Gerenciamento unificado e compartilhamento de informações.
4. Geração de diferentes relatórios.
5. Integração com outros sistemas e expansibilidade.

#### ZenTao

[ZenTao](https://www.zentao.net/) é uma ferramenta profissional de gerenciamento de projetos muito usada na China. Não é apenas uma ferramenta de gerenciamento de defeitos. Ele fornece funções completas de gerenciamento do ciclo de vida do software e oferece suporte ao desenvolvimento ágil do Scrum.

O artigo usa CentOS Linux como exemplo para mostrar como instalar o ZenTao com o pacote multifuncional oficial. Fora da China, equipes também costumam adotar alternativas como GitLab Issues, Jira, Redmine ou Backlog:

```shell
cd /opt
wget http://dl.cnezsoft.com/zentao/pro8.5.2/ZenTaoPMS.pro8.5.2.zbox_64.tar.gz
gunzip ZenTaoPMS.pro8.5.2.zbox_64.tar.gz
tar -xvf ZenTaoPMS.pro8.5.2.zbox_64.tar
/opt/zbox/zbox -ap 8080 -mp 3307
/opt/zbox/zbox start
```

Após a instalação, você pode abrir o navegador e visitar o servidor para usar o ZenTao. A conta de administrador padrão é `admin` e a senha é `123456`.

#### GitLab

Plataformas de hospedagem de código e servidores Git privados, como o GitLab, também fornecem funções de gerenciamento de defeitos. Quando quisermos reportar um bug, podemos criar um novo ticket de problema.

![](./res/gitlab-new-issue.png)

Ao criar um problema, geralmente devemos incluir:

1. a versão do software e o ambiente onde o bug aparece;
2. etapas de reprodução estáveis;
3. o comportamento esperado e o comportamento real;
4. opcionalmente, nosso próprio palpite sobre o motivo do bug.

Algumas equipes ágeis também usam issues para gerenciar requisitos de produtos. Isso é chamado de desenvolvimento orientado a issues.

#### Outros produtos

Além do ZenTao e do GitLab, [JIRA](https://www.atlassian.com/software/jira), [Redmine](https://www.redmine.org/) e Backlog também são bons sistemas de gerenciamento de defeitos.

### Integração Contínua

Para produzir software de alta qualidade rapidamente, a integração contínua (CI) é uma parte muito importante do desenvolvimento da equipe. CI significa permitir que o computador repita automaticamente compilação, teste, geração de relatórios e trabalhos semelhantes inúmeras vezes. Através da CI, os desenvolvedores podem encontrar problemas mais cedo e reduzir os riscos causados por erros humanos.

CI reúne controle de versão, construção automatizada e teste de código e os transforma em um processo automatizado e colaborativo. Como repete frequentemente todo o processo de desenvolvimento, ajuda os desenvolvedores a descobrir problemas antecipadamente.

Entre as ferramentas de CI, Jenkins e [TravisCI](https://www.travis-ci.org/) são muito representativos. A figura abaixo mostra o painel de trabalho do Jenkins.

![](./res/jenkins_new_project.png)

Para uma linguagem como Python, o CI é usado com mais frequência para conectar o controle de versão, acionar testes automáticos e gerar relatórios. Funções semelhantes também podem ser concluídas configurando o **Webhook**.
