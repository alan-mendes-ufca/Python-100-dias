## A Deeper Look at Models

In the previous lesson, we mentioned that Django is a Web framework based on MVC architecture, and MVC pursues decoupling of "model" and "view". To say it more directly, "model" means data, so it is also usually called a "data model". In actual projects, data models are usually persisted through a database, and relational databases were and still are the first choice for persistence. Below, we use a voting project to explain knowledge points related to models.

The voting project works like this:

- the home page shows all subjects on an online education platform
- clicking a subject displays its teachers
- logged-in users can vote for or against a teacher
- unregistered users can sign up
- login and registration pages are also part of the project

MySQL is used as the backing database.

We first create a Django project named `vote` and add a virtual environment and dependencies for it. Next, under the project, create an app named `polls` and a folder named `templates` for saving template pages.

2. Create the subject and teacher tables:

```SQL
use vote;

create table `tb_subject`
(
    `no` integer auto_increment comment 'Subject ID',
    `name` varchar(50) not null comment 'Subject name',
    `intro` varchar(1000) not null default '' comment 'Subject introduction',
    `is_hot` boolean not null default 0 comment 'Whether it is a popular subject',
    primary key (`no`)
);

create table `tb_teacher`
(
    `no` integer auto_increment comment 'Teacher ID',
    `name` varchar(20) not null comment 'Teacher name',
    `sex` boolean not null default 1 comment 'Gender',
    `birth` date not null comment 'Date of birth',
    `intro` varchar(1000) not null default '' comment 'Introduction',
    `photo` varchar(255) not null default '' comment 'Photo',
    `gcount` integer not null default 0 comment 'Positive votes',
    `bcount` integer not null default 0 comment 'Negative votes',
    `sno` integer not null comment 'Subject ID',
    primary key (`no`),
    foreign key (`sno`) references `tb_subject` (`no`)
);
```

3. Install the dependency needed to connect to MySQL in the virtual environment:

```bash
pip install mysqlclient
```

If for some reason `mysqlclient` cannot be installed, you can use its replacement `pymysql`. `pymysql` is a Python library for connecting to MySQL written in pure Python, and it is easier to install successfully. But it needs extra code in the `__init__.py` file of the Django project folder and has compatibility issues with Django 2.2 and above, so using `pymysql` is more troublesome. It is strongly recommended that everyone try `mysqlclient` first.

5. Django provides ORM to solve the persistence problem. ORM is translated into Chinese as "object-relational mapping". Because Python is an object-oriented programming language, we use object models in Python programs to save data, while relational databases use relation models and save data with two-dimensional tables. These two models do not match. ORM is used to achieve **two-way conversion** between the object model and the relation model, so we do not need to write SQL statements and cursor operations in Python code, because ORM completes these automatically. Using Django's ORM, we can directly turn the subject table and teacher table created just now into model classes in Django.

You can generate model classes from existing tables with:

```bash
python manage.py inspectdb > polls/models.py
```

We can slightly adjust the automatically generated model classes. The code is shown below.

```python
from django.db import models


class Subject(models.Model):
    no = models.AutoField(primary_key=True, verbose_name='ID')
    name = models.CharField(max_length=50, verbose_name='Name')
    intro = models.CharField(max_length=1000, verbose_name='Introduction')
    is_hot = models.BooleanField(verbose_name='Is Popular')

    class Meta:
        managed = False
        db_table = 'tb_subject'


class Teacher(models.Model):
    no = models.AutoField(primary_key=True, verbose_name='ID')
    name = models.CharField(max_length=20, verbose_name='Name')
    sex = models.BooleanField(default=True, verbose_name='Gender')
    birth = models.DateField(verbose_name='Date of Birth')
    intro = models.CharField(max_length=1000, verbose_name='Introduction')
    photo = models.ImageField(max_length=255, verbose_name='Photo')
    good_count = models.IntegerField(default=0, db_column='gcount', verbose_name='Positive Votes')
    bad_count = models.IntegerField(default=0, db_column='bcount', verbose_name='Negative Votes')
    subject = models.ForeignKey(Subject, models.DO_NOTHING, db_column='sno')

    class Meta:
        managed = False
        db_table = 'tb_teacher'
```

> **Note**: Model classes directly or indirectly inherit from the `Model` class. Model classes correspond to two-dimensional tables in relational databases, model objects correspond to records in the table, and attributes of model objects correspond to fields in the table. If you do not fully understand the attribute definitions in the model classes above, you can look at the "model definition reference" content provided later in the article.

### Use ORM to Complete CRUD Operations on Models

Open a Django shell:

```bash
python manage.py shell
```

#### Create

```python
from polls.models import Subject

subject1 = Subject(name='Python Full-Stack Development', intro='A very popular subject', is_hot=True)
subject1.save()
subject2 = Subject(name='Full-Stack Software Testing', intro='A subject about automated testing', is_hot=False)
subject2.save()
subject3 = Subject(name='JavaEE Distributed Development', intro='Server-side development based on Java', is_hot=True)
subject3.save()
```

#### Delete

```python
subject = Subject.objects.get(no=2)
subject.delete()
```

#### Update

```python
subject = Subject.objects.get(no=1)
subject.name = 'Python and AI'
subject.save()
```

#### Query

```python
Subject.objects.all()
Subject.objects.filter(is_hot=True)
Subject.objects.get(no=1)
```

Django's ORM also supports ordering, filtering, aggregation, and related-object queries. In actual work, most CRUD work in a Django application is written in the model layer through these APIs instead of raw SQL.

For queries of multiple objects, Django returns a `QuerySet`. `QuerySet` uses lazy query, which means creating the `QuerySet` itself does not immediately trigger a database query. Only when the queryset is really used will Django send SQL and fetch the result. This is something you need to remember in actual development.

If you want to update multiple rows, you do not need to get model objects one by one and modify attributes one by one. You can use the `update()` method on a `QuerySet` to update multiple rows at one time.

6. Register model admin classes.

```python
from django.contrib import admin

from polls.models import Subject, Teacher


class SubjectModelAdmin(admin.ModelAdmin):
    list_display = ('no', 'name', 'intro', 'is_hot')
    search_fields = ('name', )
    ordering = ('no', )


class TeacherModelAdmin(admin.ModelAdmin):
    list_display = ('no', 'name', 'sex', 'birth', 'good_count', 'bad_count', 'subject')

    ordering = ('no', )
```

### Implement the Subject and Teacher Pages

1. Modify `polls/views.py` and write view functions to render the subject page and teacher page.

```python
from django.shortcuts import render, redirect

from polls.models import Subject, Teacher


def show_subjects(request):
    subjects = Subject.objects.all().order_by('no')
    return render(request, 'subjects.html', {'subjects': subjects})


def show_teachers(request):
    try:
        sno = int(request.GET.get('sno'))
        teachers = []
        if sno:
            subject = Subject.objects.only('name').get(no=sno)
            teachers = Teacher.objects.filter(subject=subject).order_by('no')
        return render(request, 'teachers.html', {
            'subject': subject,
            'teachers': teachers
        })
    except (ValueError, Subject.DoesNotExist):
        return redirect('/')
```
