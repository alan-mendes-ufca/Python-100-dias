## Using Django for Commercial Projects

> Note: some illustrations in the original lesson come from *Python Project Development in Practice* and *Mastering Django*. Both books contain excellent discussion of the Django framework and are worth reading if you want more depth.

This lesson is not a narrow tutorial. It is closer to a practical checklist of the knowledge needed in a real Django project.

### MVC Architecture

The next questions are:

1. Why use the MVC architecture?
2. What is the role of each part in the MVC architecture?

![](./res/mvc.png)

The key idea is **decoupling**. Models and views should not be tightly bound together. In Django’s own vocabulary, the corresponding structure is closer to MTV, but the same separation idea still matters: keep data, presentation, and control flow from becoming one tangled mess.

### HTTP Requests and Responses

The lesson then reviews the structure of HTTP.

- An HTTP request is made of: request line + request headers + blank line + optional message body.
- An HTTP response is made of: status line + response headers + blank line + message body.

![](./res/http-request.png)

![](./res/http-response.png)

The lesson expects familiarity with `HttpRequest` and `HttpResponse`.

Important `HttpRequest` attributes and methods include:

- `method`
- `path` / `get_full_path()`
- `scheme` / `is_secure()` / `get_host()` / `get_port()`
- `META` / `COOKIES`
- `GET` / `POST` / `FILES`
- `get_signed_cookie()`
- `is_ajax()`
- `body` / `content_type` / `encoding`

Middleware may also add:

- `session`
- `user`
- `site`

Important `HttpResponse` features include:

- `set_cookie()` / `set_signed_cookie()` / `delete_cookie()`
- mapping-style header access
- `charset`
- `content`
- `status_code`

The lesson also reviews the broad HTTP status-code classes:

- `1xx`: the request has been received and processing continues;
- `2xx`: the request succeeded;
- `3xx`: further action is needed to complete the request;
- `4xx`: the client made an invalid request;
- `5xx`: the server failed while handling the request.

It also touches on `JsonResponse` and simple response examples.

```python
>>> from django.http import HttpResponse, JsonResponse
>>>
>>> response = JsonResponse({'foo': 'bar'})
>>> response.content
>>>
>>> response = JsonResponse([1, 2, 3], safe=False)
>>> response.content
>>>
>>> response = HttpResponse(b'...')
>>> response['cotent-type'] = 'application/pdf'
>>> response['content-disposition'] = 'inline; filename="xyz.pdf"'
>>> response['content-disposition'] = 'attachment; filename="xyz.pdf"'
>>>
>>> response.set_signed_cookie('foo', 'bar', salt='')
>>> response.status_code = 200
```

### Data Models

This part asks several practical questions:

1. What should be kept in mind when designing relational tables?
2. How do you create model classes from tables through reverse engineering?
3. How do you create tables from model classes through forward engineering?
4. What is ORM and what problem does it solve?

Common migration and reverse-engineering commands include:

```shell
python manage.py makemigrations <appname>
python manage.py migrate
python manage.py inspectdb > <appname>/models.py
```

The lesson also reviews:

- database integrity, including entity integrity, referential integrity, and domain integrity;
- Django field options such as `null`, `blank`, `default`, `primary_key`, `db_index`, and `unique`;
- `ForeignKey` options such as `db_constraint`, `on_delete`, and `related_name`;
- model methods such as `save()`, `delete()`, `clean()`, and `full_clean()`;
- queryset methods such as `get()`, `all()`, `values()`, `count()`, `filter()`, `exclude()`, `aggregate()`, `annotate()`, `create()`, `update()`, and `raw()`;
- `Q` objects and `F` objects;
- raw SQL access through Django database connections;
- custom model managers.

The key point is that real Django work requires more than just defining a couple of model classes. You need to understand how ORM operations map onto relational design and what tradeoffs exist between convenience and performance.

### View Functions and Transactions

The lesson treats each user request or user story as a potential view-function design problem. It also highlights a deeper issue: one request may contain several persistent operations, and those operations may need to be designed as one atomic unit. That is where transaction boundaries appear.

It reviews the ACID properties of transactions:

- **Atomicity**
- **Consistency**
- **Isolation**
- **Durability**

It also explains transaction-isolation levels:

- `Read Uncommitted`
- `Read Committed`
- `Repeatable Read`
- `Serializable`

and reminds the reader that stronger isolation reduces concurrency problems but usually costs more performance.

The lesson also lists several concurrency problems:

- lost updates;
- dirty reads;
- non-repeatable reads;
- phantom reads.

For Django transaction control, it introduces:

```python
ATOMIC_REQUESTS = True
```

```python
@transaction.non_atomic_requests
@transaction.atomic
```

```python
with transaction.atomic():
    pass
```

and manual control through:

```python
transaction.commit()
transaction.rollback()
```

### URL Configuration

The URL section covers several practical points:

1. Some URLs can be made active only in debug mode.
2. Named capture groups can be used for path parameters.
3. URL configuration itself does not care about the HTTP method.
4. `path()` can specify parameter types more clearly than `url()` style regex capture.
5. `include()` and `namespace` help resolve naming conflicts.
6. Extra parameters can be passed to views through dictionaries in URL definitions.
7. `reverse()` and `{% url %}` support reverse URL resolution.

### Other Views and File Responses

This section explains MIME types and how to control how generated content is handled in the browser, such as `inline` versus `attachment`.

It also discusses generating CSV, Excel, PDF, and statistical reports, including the pattern of writing binary content into an `HttpResponse` with an in-memory buffer like `BytesIO`.

### Summary

This lesson is essentially a commercial Django project checklist. It combines web architecture, HTTP, ORM, transactions, URL routing, templates, and file responses into one field guide. The point is not memorizing isolated APIs. The point is understanding how these parts fit together in a real project where deployment, data integrity, performance, and maintainability all matter at the same time.
