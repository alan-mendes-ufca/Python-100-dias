## Tarefas Assíncronas e Tarefas Agendadas

Em uma aplicação Web, se uma solicitação executa uma operação demorada, ou o tempo de execução da solicitação não pode ser determinado, e para o usuário basta saber que o servidor recebeu a solicitação e não há necessidade de obter o resultado imediatamente, então esse tipo de operação deve ser tratado de forma assíncrona. Se **usar cache é o primeiro ponto-chave para otimizar o desempenho do site**, então tornar tarefas demoradas ou incertas **assíncronas é o segundo ponto-chave para otimizar o desempenho do site**. Simplesmente falando, **se algo puder ser feito mais tarde, não faça agora**.

Considere o envio de mensagens SMS e o upload de arquivos para armazenamento em nuvem, mencionados na lição anterior, como exemplos. A primeira é uma operação cujo tempo é incerto, pois como chamador não podemos saber o tempo de resposta da plataforma de terceiros. Esta última é uma operação demorada, pois se o arquivo for grande ou a plataforma de terceiros for instável, o upload pode demorar muito. Claramente, ambas as operações podem ser tratadas de forma assíncrona.

O **Celery** é uma fila de tarefas distribuída: a aplicação Django publica uma mensagem em um *broker*, e um ou mais *workers* a executam fora do ciclo HTTP. Nesta aula, Redis atua tanto como broker quanto como backend de resultados. O exemplo pressupõe o projeto `hellodjango` e o app `first` criados na aula 46, com o ambiente virtual ativo.

### Pré-requisitos e Redis local

Você precisa de Python com Django instalado, do ambiente virtual do projeto ativo e de um Redis acessível apenas localmente. Escolha uma das opções abaixo, não as duas.

Se Redis estiver instalado no sistema operacional, inicie-o em um terminal:

```bash
redis-server
```

Em outro terminal, confirme a conexão:

```bash
redis-cli ping
```

O segundo comando deve responder `PONG`. Como alternativa, com Docker instalado, execute Redis em um terminal separado:

```bash
docker run --rm --name django-celery-redis -p 127.0.0.1:6379:6379 redis:7.4-alpine
```

O vínculo explícito a `127.0.0.1` impede que a porta seja exposta na rede local. Para parar o container, use `Ctrl+C` nesse terminal.

Instale o Celery com o extra para Redis. A faixa de versões é a mesma declarada em [`requirements/web.txt`](../requirements/web.txt).

```bash
python -m pip install "celery[redis]>=5.6,<5.7"
python -m pip show celery redis
```

### Configurar Celery no projeto Django

Crie `hellodjango/celery.py`, ao lado de `settings.py`. O nome `hellodjango` deve ser substituído pelo nome do seu pacote de projeto se ele for diferente.

```python
import os

from celery import Celery


os.environ.setdefault("DJANGO_SETTINGS_MODULE", "hellodjango.settings")

app = Celery("hellodjango")
app.config_from_object("django.conf:settings", namespace="CELERY")
app.autodiscover_tasks()
```

Em `hellodjango/__init__.py`, importe a aplicação Celery para que ela seja carregada junto com o Django e para que `@shared_task` use essa configuração.

```python
from .celery import app as celery_app

__all__ = ("celery_app",)
```

No final de `hellodjango/settings.py`, adicione a configuração. As URLs padrão valem somente para o Redis local iniciado acima. Em produção, defina as duas variáveis de ambiente no gerenciador de segredos ou na configuração do serviço, nunca no código-fonte.

```python
import os

from celery.schedules import crontab


CELERY_BROKER_URL = os.environ.get(
    "CELERY_BROKER_URL", "redis://127.0.0.1:6379/0"
)
CELERY_RESULT_BACKEND = os.environ.get(
    "CELERY_RESULT_BACKEND", "redis://127.0.0.1:6379/1"
)
CELERY_ACCEPT_CONTENT = ["json"]
CELERY_TASK_SERIALIZER = "json"
CELERY_RESULT_SERIALIZER = "json"
CELERY_TASK_TRACK_STARTED = True
CELERY_TASK_SOFT_TIME_LIMIT = 25
CELERY_TASK_TIME_LIMIT = 30
CELERY_RESULT_EXPIRES = 3600
CELERY_TIMEZONE = TIME_ZONE

CELERY_BEAT_SCHEDULE = {
    "delete-expired-sessions-daily": {
        "task": "first.tasks.delete_expired_sessions",
        "schedule": crontab(hour=3, minute=0),
    },
}
```

Os limites de tempo evitam que uma tarefa fique executando indefinidamente; ajuste-os para cada operação real. O backend de resultados expira após uma hora para não acumular dados indefinidamente.

### Criar e chamar uma tarefa

Crie `first/tasks.py`. Tarefas devem receber argumentos que possam ser serializados em JSON, como números, textos, listas e identificadores. Não envie objetos de modelo Django, objetos de requisição ou segredos pela fila.

```python
from celery import shared_task
from django.contrib.sessions.models import Session
from django.utils import timezone


@shared_task
def calculate_total(numbers):
    if not all(isinstance(number, (int, float)) for number in numbers):
        raise ValueError("numbers deve conter somente valores numéricos")
    return sum(numbers)


@shared_task
def delete_expired_sessions():
    deleted, _ = Session.objects.filter(expire_date__lt=timezone.now()).delete()
    return deleted
```

`calculate_total` é uma tarefa curta e verificável; `delete_expired_sessions` é útil como exemplo de manutenção agendada. A segunda depende das migrações iniciais do Django, portanto execute `python manage.py migrate` se ainda não o fez.

Uma view pode enfileirar uma tarefa com `delay()` e responder imediatamente com HTTP 202. Em `first/views.py`, adicione:

```python
from celery.result import AsyncResult
from django.contrib.admin.views.decorators import staff_member_required
from django.http import JsonResponse
from django.views.decorators.http import require_POST

from hellodjango.celery import app as celery_app
from .tasks import calculate_total


@require_POST
@staff_member_required
def start_total_calculation(request):
    result = calculate_total.delay([10, 20, 30])
    return JsonResponse({"task_id": result.id, "state": result.state}, status=202)


@staff_member_required
def task_status(request, task_id):
    result = AsyncResult(str(task_id), app=celery_app)
    response = {"task_id": str(task_id), "state": result.state}

    if result.successful():
        response["result"] = result.result
    elif result.failed():
        response["detail"] = "A tarefa falhou; consulte os logs do worker."

    return JsonResponse(response)
```

Em `hellodjango/urls.py`, registre as rotas:

```python
from django.contrib import admin
from django.urls import path

from first import views


urlpatterns = [
    path("admin/", admin.site.urls),
    path("tasks/total/", views.start_total_calculation, name="start-total"),
    path("tasks/<uuid:task_id>/", views.task_status, name="task-status"),
]
```

A view que inicia a tarefa aceita somente `POST` e exige que o usuário seja membro da equipe, preservando a proteção CSRF padrão do Django e impedindo que clientes anônimos encham a fila. A consulta de estado também é limitada a membros da equipe e não devolve a exceção nem o *traceback*. Em uma aplicação voltada a usuários finais, armazene a associação entre `task_id` e o usuário ou recurso dono da tarefa, valide essa autorização e aplique limite de taxa antes de enfileirar ou consultar tarefas.

Em aplicações reais, a tarefa normalmente recebe apenas o identificador de um pedido, arquivo ou relatório e busca os dados no worker. Se a tarefa for disparada após uma gravação em banco, agende-a somente após o *commit*:

```python
send_invoice.delay_on_commit(invoice.pk)
```

`delay_on_commit()` impede que o worker tente ler um registro que ainda não foi confirmado pela transação. Ele não retorna o identificador da tarefa; quando esse identificador for indispensável, use `transaction.on_commit()` para controlar o envio.

### Executar, observar resultados e tratar falhas

Em outro terminal, com o mesmo ambiente virtual ativo e no diretório que contém `manage.py`, inicie o worker:

```bash
celery -A hellodjango worker --loglevel=INFO
```

Em seguida, abra o shell Django em um terceiro terminal para testar a fila:

```bash
python manage.py shell
```

```python
from first.tasks import calculate_total

job = calculate_total.delay([10, 20, 30])
job.id
job.get(timeout=10)
```

O último comando retorna `60` quando o worker concluir. Use `get()` somente no shell administrativo ou em testes: nunca em uma view nem dentro de outra tarefa, pois isso bloqueia o processo e pode causar esgotamento ou impasse de workers.

Os estados mais comuns são `PENDING`, `STARTED`, `SUCCESS`, `FAILURE` e `RETRY`. A view `task_status` trata `SUCCESS` retornando o resultado simples e trata `FAILURE` com uma mensagem genérica. `PENDING` também pode significar que o resultado expirou ou que o identificador é desconhecido, não apenas que a tarefa está aguardando.

Falhas transitórias, como indisponibilidade de uma API externa, devem usar tentativas limitadas e espera crescente; erros de validação, como uma lista com texto enviada a `calculate_total`, não devem ser repetidos. Um padrão para uma tarefa que chama serviço externo é:

```python
from celery import shared_task


@shared_task(
    autoretry_for=(ConnectionError, TimeoutError),
    retry_backoff=True,
    retry_jitter=True,
    retry_kwargs={"max_retries": 3},
)
def notify_partner(order_id):
    # Faça a chamada com timeout e torne a operação idempotente.
    ...
```

Registre contexto suficiente para diagnosticar falhas nos logs do worker, sem incluir tokens, senhas ou dados pessoais desnecessários. Configure alertas para falhas esgotadas.

### Agendar tarefas com Celery Beat

`CELERY_BEAT_SCHEDULE`, definido anteriormente, envia `delete_expired_sessions` todos os dias às 03:00 no fuso `TIME_ZONE` do Django. Inicie o agendador em um processo separado:

```bash
celery -A hellodjango beat --loglevel=INFO
```

Execute **apenas uma** instância de Beat para cada agenda, ou a tarefa poderá ser enviada em duplicidade. Beat apenas agenda mensagens; o worker continua necessário para executá-las. Não use `worker -B` em produção. Tarefas agendadas que podem demorar mais que seu intervalo também exigem trava distribuída ou outro mecanismo para impedir sobreposição.

Para editar agendas em tempo de execução e administrá-las pelo Django Admin, avalie `django-celery-beat`; para a agenda fixa desta aula, a configuração em `settings.py` é suficiente.

### Segurança e operação

- Mantenha Redis e o broker fora da internet pública. Em produção, use rede privada, firewall, ACLs/autenticação e TLS quando suportado.
- Passe URLs e credenciais do broker e do backend por variáveis de ambiente ou um gerenciador de segredos. Não registre essas URLs se elas contiverem credenciais e não as versione no repositório.
- Aceite somente JSON, como na configuração acima. Não habilite `pickle` para mensagens recebidas de clientes não totalmente confiáveis.
- Execute workers com usuário e permissões mínimos, limite acesso de rede e monitore logs, filas e falhas.
- Faça tarefas idempotentes: uma entrega pode ser repetida após falha, reinício ou timeout. Use chaves de idempotência ao chamar provedores externos.
- Mantenha resultados pelo menor tempo necessário e nunca exponha IDs de tarefas, resultados ou erros sem autenticação e autorização adequadas.

Consulte a [integração oficial de Celery com Django](https://docs.celeryq.dev/en/stable/django/first-steps-with-django.html), a documentação de [tarefas periódicas](https://docs.celeryq.dev/en/stable/userguide/periodic-tasks.html) e o guia de [segurança](https://docs.celeryq.dev/en/stable/userguide/security.html) ao levar esta configuração para produção.
