## Estratégia de testes: TDD, testes unitários e integração

Uma aplicação confiável não depende de um único tipo de teste. Testes unitários verificam regras pequenas e isoladas; testes de integração verificam a colaboração entre componentes; e testes ponta a ponta confirmam os fluxos mais importantes pelo ponto de vista do usuário. Cada nível encontra defeitos diferentes e tem um custo diferente para executar e manter.

Nesta aula, vamos usar `unittest`, que faz parte da biblioteca padrão do Python, e as ferramentas de teste já incluídas no Django. Portanto, os exemplos não exigem uma biblioteca de terceiros. Ao final, você saberá onde TDD, mocks, cobertura e integração contínua entram no processo de desenvolvimento.

### Objetivos de aprendizagem

Ao final da aula, você será capaz de:

1. Distinguir testes unitários, de integração e ponta a ponta.
2. Escrever e executar testes unitários com `unittest`.
3. Aplicar o ciclo TDD: red, green e refactor.
4. Organizar testes em uma aplicação Django.
5. Testar rotas, views e modelos usando o cliente e o banco temporário do Django.
6. Isolar integrações externas com mocks e interpretar falhas.
7. Medir cobertura e executar a suíte continuamente.

### A pirâmide de testes

Uma estratégia saudável costuma se parecer com uma pirâmide: muitos testes rápidos na base e poucos testes mais lentos no topo. Não é uma regra de quantidade fixa, mas uma forma de equilibrar confiança, velocidade e manutenção.

| Nível | O que verifica | Exemplo | Característica |
| --- | --- | --- | --- |
| Unitário | Uma regra ou componente isolado | cálculo de desconto | Rápido e determinístico |
| Integração | A colaboração entre componentes reais | view, URL e banco de testes | Mais realista, porém mais lento |
| Ponta a ponta | Um fluxo completo como o usuário o executa | login no navegador e criação de pedido | Maior confiança no fluxo, maior custo |

Uma falha em produção não deve levar automaticamente à criação de um teste ponta a ponta. Primeiro, pergunte qual é o menor nível que reproduz o defeito com fidelidade. Na maioria dos casos, uma regra de negócio cabe em um teste unitário; uma configuração de rota ou consulta ORM pede um teste de integração.

### O padrão Arrange-Act-Assert

Um teste deve ser pequeno, legível e verificar uma única regra de negócio. Uma forma útil de estruturá-lo é o padrão **Arrange-Act-Assert**:

1. **Arrange**: prepare dados e objetos necessários para o cenário.
2. **Act**: execute a função ou a ação que será testada.
3. **Assert**: verifique o resultado esperado.

Por exemplo, a função abaixo calcula o preço final, em centavos, depois de aplicar um desconto.

```python
# first/utils.py

def calcular_preco_com_desconto(preco_em_centavos, percentual_desconto):
    if preco_em_centavos < 0:
        raise ValueError('O preço não pode ser negativo.')
    if not 0 <= percentual_desconto <= 100:
        raise ValueError('O desconto deve estar entre 0 e 100.')
    return preco_em_centavos * (100 - percentual_desconto) // 100
```

O código acima contém regras que podem ser verificadas sem subir um servidor, acessar a rede ou criar um banco de dados. Isso é uma boa característica de uma unidade de código.

### Primeiro teste com `unittest`

No app `first` criado na Aula 46, substitua o conteúdo de `first/tests.py` pelo código abaixo.

```python
import unittest

from .utils import calcular_preco_com_desconto


class CalcularPrecoComDescontoTests(unittest.TestCase):

    def test_aplica_desconto(self):
        # Arrange
        preco_em_centavos = 1_000

        # Act
        preco_final = calcular_preco_com_desconto(preco_em_centavos, 20)

        # Assert
        self.assertEqual(preco_final, 800)

    def test_aceita_desconto_zero(self):
        self.assertEqual(calcular_preco_com_desconto(1_000, 0), 1_000)

    def test_rejeita_preco_negativo(self):
        with self.assertRaises(ValueError):
            calcular_preco_com_desconto(-1, 10)

    def test_rejeita_desconto_fora_do_intervalo(self):
        with self.assertRaises(ValueError):
            calcular_preco_com_desconto(1_000, 101)
```

O método `assertEqual` compara o resultado obtido com o resultado esperado. Já `assertRaises` verifica que uma regra inválida gera a exceção esperada. Outros métodos úteis são:

| Método | Verificação |
| --- | --- |
| `assertTrue(valor)` | `valor` é verdadeiro |
| `assertFalse(valor)` | `valor` é falso |
| `assertIsNone(valor)` | `valor` é `None` |
| `assertIn(item, colecao)` | `item` pertence à coleção |
| `assertAlmostEqual(a, b)` | números de ponto flutuante são aproximadamente iguais |
| `assertRaises(Erro)` | o bloco produz a exceção esperada |

Os métodos de teste precisam começar com `test_`. Essa convenção permite que o executor os encontre automaticamente.

### Desenvolvimento orientado a testes (TDD)

TDD é uma prática de desenvolvimento, não apenas uma forma de verificar código já pronto. O ciclo é curto:

1. **Red**: escreva um teste que descreva um comportamento ainda inexistente e confirme que ele falha.
2. **Green**: implemente a menor quantidade de código necessária para fazê-lo passar.
3. **Refactor**: melhore nomes, estrutura ou duplicações sem mudar o comportamento; a suíte deve continuar verde.

Por exemplo, antes de implementar a validação de desconto, pode-se começar pelo caso mais simples:

```python
def test_aplica_desconto_de_vinte_por_cento(self):
    self.assertEqual(calcular_preco_com_desconto(1_000, 20), 800)
```

Na etapa **red**, esse teste falha porque a função ainda não existe ou não retorna `800`. Na etapa **green**, implemente apenas o necessário para passar. Depois, acrescente os casos de fronteira, como `0`, `100`, valores negativos e desconto acima de `100`, que aparecem no exemplo anterior. Por fim, refatore a função e os testes se houver repetição.

TDD não elimina a necessidade de explorar o problema, revisar código ou testar manualmente uma interface. Ele ajuda, porém, a transformar requisitos em exemplos executáveis e a evitar regressões durante a refatoração.

### Executando os testes

Na raiz do projeto Django, onde está `manage.py`, execute:

```bash
python manage.py test first
```

Para executar todos os testes do projeto, omita o nome do app:

```bash
python manage.py test
```

Uma execução bem-sucedida terá uma saída semelhante a esta:

```text
Found 4 test(s).
....
----------------------------------------------------------------------
Ran 4 tests in 0.001s

OK
```

Se uma asserção falhar, o Django mostra o nome do teste, o arquivo, a linha e a diferença entre o valor esperado e o valor obtido. Corrija a causa da falha; não altere a expectativa apenas para tornar a execução verde sem entender o comportamento desejado.

### Teste de integração: rota e view sem iniciar o servidor

O Django fornece um cliente de teste que simula requisições HTTP dentro do processo de teste. Ele não abre uma porta de rede e não exige executar `runserver`, mas integra a resolução de URL, o middleware necessário e a view. Por isso, este é um teste de integração leve, não um teste unitário puro.

Primeiro, dê um nome à rota criada na Aula 46.

```python
# hellodjango/urls.py

from django.contrib import admin
from django.urls import path

from first.views import show_index

urlpatterns = [
    path('admin/', admin.site.urls),
    path('hello/', show_index, name='hello'),
]
```

Adicione os testes da view ao mesmo arquivo `first/tests.py`.

```python
from django.test import SimpleTestCase
from django.urls import reverse


class ShowIndexTests(SimpleTestCase):

    def test_retorna_resposta_de_sucesso(self):
        response = self.client.get(reverse('hello'))

        self.assertEqual(response.status_code, 200)
        self.assertContains(response, "Today's recommended fruits are:")

    def test_rota_inexistente_retorna_404(self):
        response = self.client.get('/rota-inexistente/')

        self.assertEqual(response.status_code, 404)
```

`SimpleTestCase` é apropriado quando o teste não precisa consultar ou alterar o banco de dados. `reverse('hello')` obtém a URL pelo nome da rota e evita repetir o endereço `/hello/` em todos os testes.

### Teste de integração: modelos e banco de dados

Quando um teste acessa o banco de dados, use `django.test.TestCase`. O Django cria um banco exclusivo para a execução e o remove ao final. Nunca configure seus testes para usar o banco de produção.

Como exemplo, crie um modelo simples em `first/models.py`.

```python
from django.db import models


class Tarefa(models.Model):
    titulo = models.CharField(max_length=100)
    concluida = models.BooleanField(default=False)
```

Crie e aplique a migração antes de executar os testes:

```bash
python manage.py makemigrations first
python manage.py migrate
```

Depois, acrescente o teste abaixo a `first/tests.py`.

```python
from django.test import TestCase

from .models import Tarefa


class TarefaModelTests(TestCase):

    @classmethod
    def setUpTestData(cls):
        cls.tarefa = Tarefa.objects.create(titulo='Escrever testes')

    def test_cria_tarefa_nao_concluida(self):
        tarefa = Tarefa.objects.get(pk=self.tarefa.pk)

        self.assertEqual(tarefa.titulo, 'Escrever testes')
        self.assertFalse(tarefa.concluida)

    def test_altera_somente_a_tarefa_do_teste(self):
        self.tarefa.concluida = True
        self.tarefa.save()

        self.assertTrue(Tarefa.objects.get(pk=self.tarefa.pk).concluida)
```

`setUpTestData` é executado uma vez para a classe de testes e é útil para dados que não serão modificados. Quando cada teste precisa começar com objetos novos, use `setUp`, que é executado antes de cada método `test_`.

O `TestCase` restaura o banco entre os testes. Assim, a alteração feita em `test_altera_somente_a_tarefa_do_teste` não afeta outro método de teste. Essa independência é essencial: a ordem de execução dos testes não deve mudar o resultado.

Em testes com banco, valide regras observáveis: restrições, consultas, permissões e transações relevantes. Não teste detalhes que o próprio Django já garante, como o funcionamento interno de `CharField`.

### Mocks: isolando serviços externos

Uma integração com pagamento, e-mail, Redis ou uma API HTTP deve ter testes de integração próprios, em ambiente controlado. Um teste unitário não deve depender da disponibilidade, do custo nem dos dados de um serviço externo.

Para testar o comportamento do seu código sem realizar uma chamada real, injete a dependência e use `unittest.mock.Mock`.

```python
# first/services.py

def produto_disponivel(cliente_estoque, codigo):
    response = cliente_estoque.get(f'/produtos/{codigo}', timeout=3)
    response.raise_for_status()
    return response.json()['disponivel']
```

```python
import unittest
from unittest.mock import Mock

from .services import produto_disponivel


class ProdutoDisponivelTests(unittest.TestCase):

    def test_retorna_disponibilidade_da_resposta(self):
        cliente = Mock()
        cliente.get.return_value.json.return_value = {'disponivel': True}

        disponivel = produto_disponivel(cliente, 'ABC-123')

        self.assertTrue(disponivel)
        cliente.get.assert_called_once_with('/produtos/ABC-123', timeout=3)
        cliente.get.return_value.raise_for_status.assert_called_once_with()
```

O mock não testa a API de estoque. Ele testa o contrato entre `produto_disponivel` e o cliente recebido. Mantenha poucos mocks e prefira dependências explícitas: testar um conjunto grande de detalhes simulados torna os testes frágeis e pouco representativos.

### Testes ponta a ponta

Testes ponta a ponta (E2E) exercitam a aplicação pela mesma interface usada pelo usuário, geralmente com um navegador automatizado. Eles são apropriados para poucos fluxos críticos, como autenticação, checkout ou envio de um formulário importante.

Use-os com parcimônia: são mais lentos, exigem ambiente e dados previsíveis e podem falhar por fatores que não são defeitos da regra de negócio. Para uma API, um teste de integração que envie uma requisição ao cliente Django costuma ser suficiente; reserve o navegador para comportamentos que dependem de JavaScript, renderização ou interação real de página.

### Cobertura e execução contínua

Cobertura indica quais linhas ou ramos foram executados pela suíte. Ela ajuda a revelar áreas sem teste, mas uma porcentagem alta não prova que as asserções são úteis. Um teste que apenas chama uma função sem verificar o resultado aumenta a cobertura e oferece pouca proteção.

Instale a ferramenta de cobertura no ambiente de desenvolvimento e execute:

```bash
python -m pip install coverage
coverage run manage.py test
coverage report -m
```

Use a cobertura para fazer perguntas, não como uma meta cega: os caminhos de erro foram verificados? Os limites foram exercitados? A regra pode ser alterada sem que um teste falhe?

Antes de integrar código à ramificação principal, a suíte deve ser executada automaticamente pela integração contínua (CI). Um fluxo inicial é simples: instalar as dependências, executar `python manage.py test` e publicar o resultado. Com o projeto crescendo, a CI pode adicionar verificação de estilo, análise estática, cobertura e testes em mais de uma versão do Python.

### O que testar e o que evitar

Priorize regras com maior chance de regressão:

1. Valores válidos, inválidos e valores de limite.
2. Regras de negócio e cálculos.
3. Permissões, códigos HTTP e conteúdo essencial das respostas.
4. Comportamentos que já causaram defeitos no projeto.

Evite que um teste unitário dependa de serviços fora do seu controle, como uma API pública, e-mail, Redis ou um banco remoto. Para esses casos, isole a dependência com um objeto simulado (`unittest.mock`) ou execute um teste de integração separado e explicitamente configurado.

Também evite testar detalhes internos sem relevância para o usuário. Um teste deve descrever um comportamento observável, não apenas repetir a implementação atual.

### Exercícios

1. Acrescente um teste que confirme que `calcular_preco_com_desconto(1_000, 100)` retorna `0`.
2. Pratique TDD: escreva primeiro um teste que exija desconto de `50%`, confirme a falha e somente então implemente ou ajuste o código.
3. Crie uma rota que responda JSON e escreva um teste de integração que verifique o código `200` e os dados retornados.
4. Adicione ao modelo `Tarefa` um campo `data_criacao` com `auto_now_add=True` e teste se ele é preenchido ao criar uma tarefa.
5. Crie um serviço que receba um cliente HTTP como parâmetro e escreva um teste com `Mock` para a resposta de sucesso e outro para um erro HTTP.
6. Escolha um fluxo crítico do seu projeto e registre qual parte deve ter teste unitário, de integração e, se necessário, ponta a ponta. Justifique a escolha pelo custo e pelo risco.

### Resumo

Uma estratégia de testes combina níveis complementares: testes unitários rápidos para regras de negócio, testes de integração para a colaboração entre componentes e poucos testes ponta a ponta para fluxos críticos. TDD ajuda a construir essas verificações durante o desenvolvimento; mocks removem dependências externas dos testes unitários; e a CI impede que alterações sejam integradas sem executar a suíte. Execute `python manage.py test` antes de enviar alterações, especialmente após corrigir um defeito ou modificar uma regra de negócio. A próxima aula aborda a preparação de um projeto Django para entrar em produção; uma suíte de testes confiável é uma das verificações necessárias antes dessa etapa.
