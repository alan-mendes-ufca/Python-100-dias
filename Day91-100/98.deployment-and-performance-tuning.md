## Guia de implantação e entrada em operação do projeto

### Preparando-se para entrar ao vivo

1. Verifique o projeto antes de ir ao ar.

```shell
python manage.py check --deploy
```

2. Defina `DEBUG` como `False` e configure `ALLOWED_HOSTS`.

```python
DEBUG = False
ALLOWED_HOSTS = ['*']
```

3. Configuração relacionada à segurança.

```python
# How long to keep HTTPS connections
SECURE_HSTS_SECONDS = 3600
SECURE_HSTS_INCLUDE_SUBDOMAINS = True
SECURE_HSTS_PRELOAD = True

# Automatically redirect to secure connections
SECURE_SSL_REDIRECT = True

# Avoid letting the browser guess the content type by itself
SECURE_CONTENT_TYPE_NOSNIFF = True

# Avoid cross-site scripting attacks
SECURE_BROWSER_XSS_FILTER = True

# Cookies can only be sent through HTTPS
SESSION_COOKIE_SECURE = True
CSRF_COOKIE_SECURE = True

# Prevent clickjacking by changing the HTTP response header
# The current website does not allow loading with an <iframe> tag
X_FRAME_OPTIONS = 'DENY'
```

4. Coloque informações confidenciais em variáveis de ambiente ou arquivos.

```python
SECRET_KEY = os.environ['SECRET_KEY']

DB_USER = os.environ['DB_USER']
DB_PASS = os.environ['DB_PASS']

REDIS_AUTH = os.environ['REDIS_AUTH']
```

### Atualize o ambiente Python no servidor para 3.x

> Nota: Se precisar remover uma instalação anterior, basta excluir os arquivos e pastas relacionados.

1. Instale bibliotecas de dependência de baixo nível.

```shell
yum -y install zlib-devel bzip2-devel openssl-devel ncurses-devel sqlite-devel readline-devel tk-devel gdbm-devel libdb4-devel libpcap-devel xz-devel libffi-devel libxml2
```

2. Baixe o código-fonte do Python.

```shell
wget https://www.python.org/ftp/python/3.7.6/Python-3.7.6.tar.xz
```

3. Verifique o arquivo baixado.

```bash
md5sum Python-3.7.6.tar.xz
```

4. Descompacte e extraia.

```shell
xz -d Python-3.7.6.tar.xz
tar -xvf Python-3.7.6.tar
```

5. Execute a configuração antes da instalação e gere o `Makefile`.

```shell
cd Python-3.7.6
./configure --prefix=/usr/local/python37 --enable-optimizations
```

6. Construa e instale.

```shell
make && make install
```

7. Configure a variável de ambiente `PATH` e ative-a.

```shell
vim ~/.bash_profile
vim /etc/profile
```

```ini
export PATH=$PATH:/usr/local/python37/bin
```

```shell
source ~/.bash_profile
source /etc/profile
```

8. Registre um link virtual. Esta etapa não é obrigatória, mas geralmente é útil.

```shell
ln -s /usr/local/python37/bin/python3 /usr/bin/python3
```

9. Teste se o ambiente Python foi atualizado com sucesso. A instalação do Python 3 não deve danificar o Python 2 original.

```shell
python3 --version
python --version
```

### Estrutura de diretório do projeto

Suponha que a pasta do projeto seja `project`. Os cinco subdiretórios `code`, `conf`, `logs`, `stat` e `venv` são usados para salvar o código do projeto, arquivos de configuração, arquivos de log, recursos estáticos e o ambiente virtual. O subdiretório `cert` em `conf` armazena o certificado e a chave usados para HTTPS. O código do projeto em `code` pode ser retirado de um repositório de código por uma ferramenta de controle de versão. O ambiente virtual pode ser criado com ferramentas como `venv`, `virtualenv` e `pyenv`.

```text
project
├── code
│   └── fangtx
│       ├── api
│       ├── common
│       ├── fangtx
│       ├── forum
│       ├── rent
│       ├── user
│       ├── manage.py
│       ├── README.md
│       ├── static
│       └── templates
├── conf
│   ├── cert
│   ├── nginx.conf
│   └── uwsgi.ini
├── logs
├── stat
└── venv
```

### Configuração uWSGI

1. Instale `uWSGI`.

```shell
pip install uwsgi
```

2. Modifique o arquivo de configuração `uWSGI` `/root/project/conf/uwsgi.ini`.

```ini
[uwsgi]
# Base path
base=/root/project
# Project name
name=teamproject
# Daemon process
master=true
# Number of processes
processes=4
# Virtual environment
pythonhome=%(base)/venv
# Project path
chdir=%(base)/code/%(name)
# Python interpreter
pythonpath=%(pythonhome)/bin/python
# uwsgi file
module=%(name).wsgi
# Address and port for communication
socket=172.18.61.250:8000
# Log file path
logto=%(base)/logs/uwsgi.log
```

> Nota: Você pode primeiro alterar o item antes do sinal de igual para endereço de comunicação e porta de `socket` para `http` para teste. Se não houver problema, mude novamente para `socket` e use o Nginx para separar o conteúdo dinâmico do estático. Os recursos estáticos são tratados pelo Nginx e o conteúdo dinâmico é tratado pelo uWSGI.

3. Inicie o servidor.

```shell
nohup uwsgi --ini conf/uwsgi.ini &
```

### Configuração Nginx

1. Instale o Nginx.

```shell
yum -y install nginx
```

2. Modifique o arquivo de configuração global `/etc/nginx/nginx.conf`.

```nginx
user nginx;
worker_processes auto;
error_log /var/log/nginx/error.log;
pid /run/nginx.pid;
include /usr/share/nginx/modules/*.conf;

events {
    use epoll;
    worker_connections 1024;
}

http {
    log_format  main  '$remote_addr - $remote_user [$time_local] "$request" '
                      '$status $body_bytes_sent "$http_referer" '
                      '"$http_user_agent" "$http_x_forwarded_for"';
    access_log  /var/log/nginx/access.log  main;
    sendfile            on;
    tcp_nopush          on;
    tcp_nodelay         on;
    keepalive_timeout   30;
    types_hash_max_size 2048;
    include             /etc/nginx/mime.types;
    default_type        application/octet-stream;
    include /etc/nginx/conf.d/*.conf;
    include /root/project/conf/*.conf;
}
```

3. Edite o arquivo de configuração local `/root/project/conf/nginx.conf`.

```nginx
server {
    listen      80;
    server_name _;
    access_log /root/project/logs/access.log;
    error_log /root/project/logs/error.log;
    location / {
        include uwsgi_params;
        uwsgi_pass 172.18.61.250:8000;
    }
    location /static/ {
        alias /root/project/stat/;
        expires 30d;
    }
}
server {
    listen      443;
    server_name _;
    ssl         on;
    access_log /root/project/logs/access.log;
    error_log /root/project/logs/error.log;
    ssl_certificate     /root/project/conf/cert/214915882850706.pem;
    ssl_certificate_key /root/project/conf/cert/214915882850706.key;
    ssl_session_timeout 5m;
    ssl_ciphers ECDHE-RSA-AES128-GCM-SHA256:ECDHE:ECDH:AES:HIGH:!NULL:!aNULL:!MD5:!ADH:!RC4;
    ssl_protocols TLSv1 TLSv1.1 TLSv1.2;
    ssl_prefer_server_ciphers on;
    location / {
        include uwsgi_params;
        uwsgi_pass 172.18.61.250:8000;
    }
    location /static/ {
        alias /root/project/static/;
        expires 30d;
    }
}
```

Neste ponto, você pode iniciar o Nginx para acessar o aplicativo. Tanto HTTP quanto HTTPS podem funcionar normalmente. Se o Nginx já estiver em execução, após alterar o arquivo de configuração, você poderá reiniciá-lo com o seguinte comando.

4. Reinicie o servidor Nginx.

```shell
nginx -s reload
```

ou

```shell
systemctl restart nginx
```

> Nota: Você pode usar o comando `python manage.py collectstatic` em um projeto Django para coletar recursos estáticos em um diretório especificado. Para isso, basta adicionar `STATIC_ROOT` no `settings.py` do projeto.

#### Configuração de balanceamento de carga

Na configuração abaixo, o Nginx é usado para balanceamento de carga e fornece serviço de proxy reverso para outros três servidores Nginx criados por meio do Docker.

```shell
docker run -d -p 801:80 --name nginx1 nginx:latest
docker run -d -p 802:80 --name nginx2 nginx:latest
docker run -d -p 803:80 --name nginx3 nginx:latest
```

```nginx
user root;
worker_processes auto;
error_log /var/log/nginx/error.log;
pid /run/nginx.pid;

include /usr/share/nginx/modules/*.conf;

events {
    worker_connections 1024;
}

http {
    upstream xx {
        server 192.168.1.100 weight=2;
        server 192.168.1.101 weight=1;
        server 192.168.1.102 weight=1;
    }

    server {
        listen       80 default_server;
        listen       [::]:80 default_server;
        listen       443 ssl;
        listen       [::]:443 ssl;

        ssl on;
        access_log /root/project/logs/access.log;
        error_log /root/project/logs/error.log;
        ssl_certificate /root/project/conf/cert/214915882850706.pem;
        ssl_certificate_key /root/project/conf/cert/214915882850706.key;
        ssl_session_timeout 5m;
        ssl_ciphers ECDHE-RSA-AES128-GCM-SHA256:ECDHE:ECDH:AES:HIGH:!NULL:!aNULL:!MD5:!ADH:!RC4;
        ssl_protocols TLSv1 TLSv1.1 TLSv1.2;
        ssl_prefer_server_ciphers on;

        location / {
            proxy_set_header Host $host;
            proxy_set_header X-Forwarded-For $remote_addr;
            proxy_buffering off;
            proxy_pass http://fangtx;
        }
    }
}
```

> Nota: Quando o Nginx é configurado para balanceamento de carga, ele usa WRR, round robin ponderado, por padrão. Além disso, também suporta `ip_hash`, `fair` e `url_hash`. Ao configurar o módulo `upstream`, você também pode especificar valores de status do servidor, como `backup`, `down`, `fail_timeout`, `max_fails` e `weight`.

### Manter ativo

Quando o Nginx é usado para balanceamento de carga, a possibilidade de o próprio servidor de balanceamento de carga ficar inativo também deve ser considerada. Para isso, Keepalived pode ser usado para alternar entre os hosts de balanceamento de carga primário e de espera, para que o sistema possa manter alta disponibilidade. A configuração Keepalived é relativamente complexa e geralmente é realizada por equipes de operações especializadas.

### Replicação mestre-escravo MySQL

Abaixo, o Docker ainda é usado para mostrar como configurar a replicação mestre-escravo do MySQL. Primeiro prepare os arquivos de configuração do MySQL e os diretórios que salvam os dados do MySQL e os logs de tempo de execução. Em seguida, use o mapeamento de volume do Docker para especificar onde a configuração do contêiner, os dados e os arquivos de log são armazenados.

```text
root
└── mysql
    ├── master
    │   ├── conf
    │   └── data
    ├── slave-1
    │   ├── conf
    │   └── data
    ├── slave-2
    │   ├── conf
    │   └── data
    └── slave-3
        ├── conf
        └── data
```

1. Arquivo de configuração MySQL. Os arquivos de configuração mestre e escravo precisam de valores `server-id` diferentes.

```ini
[mysqld]
pid-file=/var/run/mysqld/mysqld.pid
socket=/var/run/mysqld/mysqld.sock
datadir=/var/lib/mysql
log-error=/var/log/mysql/error.log
server-id=1
log-bin=/var/log/mysql/mysql-bin.log
expire_logs_days=30
max_binlog_size=256M
symbolic-links=0
```

2. Crie e configure o mestre.

```shell
docker run -d -p 3306:3306 --name mysql-master \
-v /root/mysql/master/conf:/etc/mysql/mysql.conf.d \
-v /root/mysql/master/data:/var/lib/mysql \
-e MYSQL_ROOT_PASSWORD=123456 mysql:5.7

docker exec -it mysql-master /bin/bash
```

```sql
grant replication slave on *.* to 'slave'@'%' identified by 'iamslave';
flush privileges;
show master status;
```

O parâmetro `-v` usado acima significa mapear um volume de dados. O diretório antes dos dois pontos é um diretório na máquina host e o diretório após os dois pontos é um diretório dentro do contêiner. Isso equivale a montar o diretório da máquina host no contêiner.

3. Faça backup dos dados nas tabelas mestre, se necessário.

```sql
flush table with read lock;
```

```bash
mysqldump -u root -p 123456 -A -B > /root/backup/mysql/mybak$(date +"%Y%m%d%H%M%S").sql
```

```sql
unlock table;
```

4. Crie e configure os escravos.

```shell
docker run -d -p 3308:3306 --name mysql-slave-1 \
-v /root/mysql/slave-1/conf:/etc/mysql/mysql.conf.d \
-v /root/mysql/slave-1/data:/var/lib/mysql \
-e MYSQL_ROOT_PASSWORD=123456 \
--link mysql-master:mysql-master mysql:5.7

docker run -d -p 3309:3306 --name mysql-slave-2 \
-v /root/mysql/slave-2/conf:/etc/mysql/mysql.conf.d \
-v /root/mysql/slave-2/data:/var/lib/mysql \
-e MYSQL_ROOT_PASSWORD=123456 \
--link mysql-master:mysql-master mysql:5.7

docker run -d -p 3310:3306 --name mysql-slave-3 \
-v /root/mysql/slave-3/conf:/etc/mysql/mysql.conf.d \
-v /root/mysql/slave-3/data:/var/lib/mysql \
-e MYSQL_ROOT_PASSWORD=123456 \
--link mysql-master:mysql-master mysql:5.7
```

```sql
reset slave;
change master to master_host='mysql-master', master_user='slave', master_password='iamslave', master_log_file='mysql-bin.000003', master_log_pos=590;
start slave;
show slave status\G
```

Você pode configurar `slave2` e `slave3` da mesma maneira e, em seguida, construir um ambiente de replicação de um mestre e três escravos. O parâmetro `--link` utilizado na criação do container é utilizado para configurar o nome do host do container na rede.

Após a replicação ser configurada, as operações de gravação devem ser feitas no mestre e as operações de leitura devem ser feitas nos escravos. Por esse motivo, em um projeto Django, `DATABASE_ROUTERS` precisa ser configurado e uma classe de roteador mestre-escravo customizada é usada para implementar a separação leitura-gravação.

```python
DATABASE_ROUTERS = [
    'common.routers.MasterSlaveRouter',
]
```

```python
class MasterSlaveRouter(object):
    """Master-slave replication router."""

    @staticmethod
    def db_for_read(model, **hints):
        return random.choice(('slave1', 'slave2', 'slave3'))

    @staticmethod
    def db_for_write(model, **hints):
        return 'default'

    @staticmethod
    def allow_relation(obj1, obj2, **hints):
        return None

    @staticmethod
    def allow_migrate(db, app_label, model_name=None, **hints):
        return True
```

### Docker

Na verdade, a coisa mais problemática quando um projeto entra em operação é configurar o ambiente de tempo de execução do software. As diferenças entre ambientes trazem muitos problemas para instalação e implantação de software, e o Docker pode resolver esse problema. Docker foi introduzido em documentos anteriores. Aqui adicionamos alguns conhecimentos necessários.

1. Crie um arquivo de imagem.

Salve um contêiner como uma imagem:

```shell
docker commit -m "..." -a "jackfrued" <container-name> jackfrued/<image-name>
```

Construa uma imagem com um Dockerfile:

```dockerfile
FROM centos:latest
MAINTAINER jackfrued
RUN yum -y install gcc
RUN cd ~
RUN mkdir -p project/code
RUN mkdir -p project/logs
COPY ...
EXPOSE ...
CMD ~/init.sh
```

```shell
docker build -t jackfrued/<image-name> .
```

2. Importe e exporte imagens.

```shell
docker save -o <file-name>.tar <image-name>:<version>
docker load -i <file-name>.tar
```

3. Envie para o DockerHub.

```shell
docker tag <image-name>:<version> jackfrued/<name>
docker login
docker push jackfrued/<name>
```

4. Comunicação entre contêineres.

```shell
docker run --link <container-name>:<alias-name>
```

Se a implantação puder ser concluída dentro do Docker e o contêiner totalmente implantado puder ser compactado em um arquivo de imagem para distribuição e instalação, os problemas que podem surgir ao implantar o projeto em vários nós poderão ser resolvidos e toda a implantação poderá ser concluída em um curto espaço de tempo.

### Supervisor

[Supervisor](https://github.com/Supervisor/supervisor) é uma ferramenta de gerenciamento de processos escrita em Python. Ele pode iniciar, reiniciar e fechar processos de maneira conveniente em sistemas do tipo Unix. Atualmente, o Supervisor não oferece suporte direto ao Python 3, portanto, o Python 2 pode ser usado para instalar e executar o Supervisor e, em seguida, o Supervisor pode ser usado para gerenciar programas Python 3.

1. Instale o Supervisor.

```shell
virtualenv -p /usr/bin/python venv
source venv/bin/activate
pip install supervisor
```

2. Verifique o arquivo de configuração do Supervisor.

```shell
vim /etc/supervisord.conf
```

```ini
[include]
files = supervisord.d/*.ini
```

Isso mostra que a configuração de gerenciamento personalizada pode ser colocada no diretório `/etc/supervisord.d` e o nome do arquivo só precisa usar o sufixo `ini`.

3. Escreva seu próprio arquivo de configuração `fangtx.ini` e coloque-o em `/etc/supervisord.d`.

```ini
[program:project]
command=uwsgi --ini /root/project/conf/uwsgi.ini
stopsignal=QUIT
autostart=true
autorestart=true
redirect_stderr=true

[program:celery]
command=/root/project/venv/bin/celery -A fangtx worker
user=root
numprocs=1
stdout_logfile=/var/log/supervisor/celery.log
stderr_logfile=/var/log/supervisor/celery_error.log
autostart=true
autorestart=true
startsecs=10
killasgroup=true
priority=1000
```

4. Inicie o Supervisor.

```shell
supervisorctl -c /etc/supervisord.conf
```
