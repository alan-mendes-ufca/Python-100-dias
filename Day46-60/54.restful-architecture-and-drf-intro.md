## RESTful Architecture and DRF Basics

Turning software, platforms, and infrastructure into services is something many IT companies have been doing for a long time. This is what people often call SaaS (Software as a Service), PaaS (Platform as a Service), and IaaS (Infrastructure as a Service). There are many ways to implement service-oriented architecture (SOA), including RPC (Remote Procedure Call), Web Service, and REST. On the technical level, SOA is an **abstract, loosely coupled, coarse-grained software architecture**. On the business level, the core ideas of SOA are "**reuse**" and "**interoperability**". It turns system resources into operable and standard services, so these resources can be recombined and used again. Among the many ways to implement SOA, REST is considered the architecture most suitable for Internet applications, and an architecture that follows REST rules is often called a RESTful architecture.

### Overview of REST

The word REST was proposed by **Roy Thomas Fielding** in his doctoral dissertation in 2000. Roy was one of the main designers of the HTTP protocol versions 1.0 and 1.1, a main author of the Apache server software, and the first chairman of the Apache Foundation. In his doctoral dissertation, Roy gave the name REST to his architectural principles for Internet software. REST is short for **RE**presentational **S**tate **T**ransfer.

The "representation" here actually means the representation of a "resource". A resource is an entity on the network, or more directly, a piece of concrete information on the network. It can be a piece of text, an image, a song, or a service. We can use a URI (Uniform Resource Identifier) to point to a resource. To get this resource, we can access its URI. The URI is the unique identifier of the resource on the Internet. A resource can have many external forms of representation. The specific form used to show a resource is called its "representation". For example, text can be represented in `text/plain`, `text/html`, `text/xml`, `application/json`, and even binary form. Images can be represented in `image/jpeg` or `image/png`. A URI only stands for the entity of the resource, not its representation. Strictly speaking, the `.html` suffix at the end of some website addresses is unnecessary, because this suffix shows the format, and format belongs to the "representation" category. A URI should only represent the location of the "resource". Its exact representation should be specified in the HTTP request headers through the `Accept` and `Content-Type` fields. These two fields are the description of the "representation".

Visiting a website means an interaction process between the client and the server. In this process, data and state changes are surely involved. Web applications usually use HTTP as their communication protocol. If the client wants to operate the server, it must use an HTTP request to make the server perform "state transfer". Since this transfer is based on the representation layer, it is called "representational state transfer". The client uses the HTTP verbs GET, POST, PUT (or PATCH), and DELETE, which correspond to the four basic operations on resources. GET is used to get resources, POST is used to create resources, and can also be used to update resources, PUT (or PATCH) is used to update resources, and DELETE is used to delete resources.

Simply speaking, RESTful architecture means: "each URI represents one kind of resource, and the client uses the four HTTP verbs to operate on server resources, so the representation state of the resources is transferred".

When we design a Web application, if we need to provide resources to the client, we can use REST-style URIs. This is the first step to implement RESTful architecture. Of course, a real RESTful architecture is not only about URIs that look REST-style. More important are the two ideas "statelessness" and "idempotency". We will explain these two points in later lessons. The example below gives some URIs that match REST style, and you can use them as reference when designing URIs.

| Request Method (HTTP Verb) | URI                        | Meaning                                       |
| -------------------------- | -------------------------- | --------------------------------------------- |
| **GET**                    | `/students/`               | Get all students                              |
| **POST**                   | `/students/`               | Create a student                              |
| **GET**                    | `/students/ID/`            | Get the student information with the given ID |
| **PUT**                    | `/students/ID/`            | Update the student information with the given ID, giving all information |
| **PATCH**                  | `/students/ID/`            | Update the student information with the given ID, giving part of the information |
| **DELETE**                 | `/students/ID/`            | Delete the student information with the given ID |
| **GET**                    | `/students/ID/friends/`    | List all friends of the student with the given ID |
| **DELETE**                 | `/students/ID/friends/ID/` | Delete the friend with the given ID from the student with the given ID |

#### Writing Serializers

Frontend-backend separation development needs the backend to provide API data interfaces for the frontend and mobile side, and in most cases these API interfaces return JSON format data, so model objects need to be serialized. DRF wraps `Serializer` and `ModelSerializer` classes to implement serialization. By inheriting from `Serializer` or `ModelSerializer`, we can customize serializers and use them to turn objects into dictionaries, as shown below.

```python
from rest_framework import serializers


class SubjectSerializer(serializers.ModelSerializer):

    class Meta:
        model = Subject
        fields = '__all__'
```

The code above directly inherits from `ModelSerializer`, uses the `model` attribute of the `Meta` class to specify the model to serialize, and uses the `fields` attribute to specify the model fields that need to be serialized. Later, we can use this class in the view function to serialize the `Subject` model.

#### Writing View Functions

The DRF framework supports two ways to implement data APIs. One is FBV, meaning function-based views, and the other is CBV, meaning class-based views. Let us first see how to implement a data API with FBV. The code is shown below.

```python
from rest_framework.decorators import api_view
from rest_framework.response import Response


@api_view(('GET', ))
def show_subjects(request: HttpRequest) -> HttpResponse:
    subjects = Subject.objects.all().order_by('no')
    # Create the serializer object and specify the model to serialize
    serializer = SubjectSerializer(subjects, many=True)
    # Get the dictionaries corresponding to the model through the serializer's data property
    # and return JSON data by creating a Response object
    return Response(serializer.data)
```

#### Implementing the Teacher Information Data API

Write the serializer.

```python
class SubjectSimpleSerializer(serializers.ModelSerializer):

    class Meta:
        model = Subject
        fields = ('no', 'name')


class TeacherSerializer(serializers.ModelSerializer):

    class Meta:
        model = Teacher
        exclude = ('subject', )
```

Write the view function.

```python
@api_view(('GET', ))
def show_teachers(request: HttpRequest) -> HttpResponse:
    try:
        sno = int(request.GET.get('sno'))
        subject = Subject.objects.only('name').get(no=sno)
        teachers = Teacher.objects.filter(subject=subject).defer('subject').order_by('no')
        subject_seri = SubjectSimpleSerializer(subject)
        teacher_seri = TeacherSerializer(teachers, many=True)
        return Response({'subject': subject_seri.data, 'teachers': teacher_seri.data})
    except (TypeError, ValueError, Subject.DoesNotExist):
        return Response(status=404)
```

Configure URL mapping.

```python
urlpatterns = [

    path('api/teachers/', show_teachers),

]
```

### User Login with Frontend-Backend Separation

We mentioned before that HTTP is stateless. After one request ends, the connection is closed. When the server receives the next request, it does not know which user sent it. But for a Web application, state management is needed, so the server can know which user the HTTP request comes from, decide whether the user is allowed to make the request, and provide better service for the user. This process is what we often call **session management**.

The way we did session management, or user tracking, before was this: after the user logged in successfully, the server saved user-related data in a session object, then wrote the ID of the session object into the browser's cookie. On the next request, the HTTP request header carried the cookie data. The server read the `sessionid` from the cookie in the HTTP request header, found the matching session object according to this identifier, and then got the user data saved in the session before. As we just said, REST architecture is the architecture most suitable for Internet applications. It emphasizes the statelessness of HTTP, so the application can keep horizontal scalability. When concurrent visits increase, we can expand the system by adding new server nodes. Clearly, the user-tracking method based on session needs the server to save session objects. When new server nodes are added for horizontal scaling, session objects need to be copied and synchronized, and this is obviously very troublesome. There are two ways to solve this problem. One is to set up a cache server such as Redis, let multiple server nodes share the cache service, and place the session objects directly into the cache server. The other is to give up session-based user tracking and use **token-based user tracking**.

Token-based user tracking means that after the user logs in successfully, an identity marker is generated for the user and saved in browser local storage, such as localStorage, sessionStorage, or cookie. In this way, the server does not need to save user state, so horizontal scaling can be done easily. The specific flow of token-based user tracking is as follows:

1. When the user logs in, if login succeeds, generate a token for the user in some way. This token usually contains user identity, expiration time, and other information, and it needs to be encrypted and fingerprinted to avoid forging or tampering. The server returns the token to the frontend.
2. The frontend gets the token returned by the server and saves it in browser local storage. It can be saved in `localStorage` or `sessionStorage`. For a frontend project using Vue.js, it can also be managed through Vuex.
3. For a project that uses frontend routing, every time the frontend route jumps, it can first check whether there is a token in `localStorage`. If there is no token, jump to the login page.
4. Every time the frontend requests a backend data API, carry the token in the HTTP request header. The backend interface checks whether the request header has a token. If there is no token, or the token is invalid or expired, the server returns `401` in a unified way.
5. If the frontend receives HTTP response status code `401`, redirect to the login page.

From the description above, you have probably found that the most important part of token-based user tracking is generating a token as the user's identity marker when the user logs in successfully. There are many ways to generate a token. One more mature solution is JSON Web Token.

#### Overview of JWT

JSON Web Token is usually shortened to JWT. It is an open standard, RFC 7519. As RESTful architecture has become popular, more and more projects use JWT as a way to authenticate user identity. JWT is made by encoding three JSON objects, joining them together with `.`, and combining them into one string. These three JSON objects are the header, the payload, and the signature, as shown below.

![](res/json-web-token.png)

1. Header

    ```json
    {
      "alg": "HS256",
      "typ": "JWT"
    }
    ```

    In it, the `alg` property means the algorithm used for the signature, and the default is HMAC SHA256, written shortly as `HS256`. The `typ` property means the type of this token, and in JWT it is always written as `JWT`.

2. Payload

    The payload part is used to store the actual data that needs to be passed. The official JWT documentation defines seven optional fields:

    - `iss` - issuer
    - `exp` - expiration time
    - `sub` - subject
    - `aud` - audience
    - `nbf` - not before
    - `iat` - issued at
    - `jti` - ID

    Besides the fields officially defined, we can add custom fields according to the needs of the application, as shown below.

    ```json
    {
      "sub": "1234567890",
      "nickname": "jackfrued",
      "role": "admin"
    }
    ```

3. Signature

    The signature part generates a fingerprint for the first two parts to prevent data forgery and tampering. To create the signature, we first need to specify a secret key. This key is known only by the server and must not be leaked to users. Then, use the signature algorithm specified in the header, and the default is `HS256`, to generate the signature according to the formula below.

    ```python
    HS256(base64Encode(header) + '.' + base64Encode(payload), secret)
    ```

    After the signature is calculated, join the three parts, header, payload, and signature, into one string, and separate each part with `.`, then one JWT is generated.

#### Advantages and Disadvantages of JWT

The advantages of using JWT are very clear, including:

1. It is easier to implement horizontal scaling, because the token is saved in the browser and the server does not need to do state management.
2. It is easier to prevent CSRF attacks, because adding the token from `localStorage` or `sessionStorage` to the request header must be done by JavaScript code, instead of being automatically added to the request header.
3. It can prevent forging and tampering, because JWT has a signature. Forged or tampered tokens cannot pass signature verification and will be treated as invalid tokens.

Of course, no technology has only advantages and no disadvantages. JWT also has many disadvantages, and everyone should pay attention to them when using it. These disadvantages include:

1. It may suffer from XSS attacks, or cross-site scripting attacks, where malicious scripts are injected to run JavaScript code and get the user's token.
2. Before the token expires, an issued token cannot be revoked. To solve this problem, extra middleware and code are still needed.
3. JWT is the user's identity token. Once it is leaked, anyone can get all of that user's permissions. To reduce the risk after a token is stolen, the validity period of JWT should be set short. For some more important permissions, the user should be authenticated again in some other way when using them, for example by SMS verification code.
