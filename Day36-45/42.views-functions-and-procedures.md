### Visualizações

Uma visão é um objeto em um banco de dados relacional que combina o conjunto de resultados criado a partir de um grupo de instruções de consulta em um objeto de tabela de dados consultável. Simplificando, uma visualização é uma mesa virtual. Mas, diferentemente de uma tabela de dados, uma tabela de dados é uma estrutura de entidade, enquanto uma visualização é uma estrutura virtual. Você também pode entender uma visualização como uma instrução SQL salva no banco de dados e que recebe um nome.

O uso de visualizações pode trazer os seguintes benefícios:

1. Eles podem ocultar a tabela de dados físicos, de modo que programas externos não possam conhecer a estrutura real dos dados. Os visitantes podem usar parte da tabela em vez de toda a tabela, o que diminui o risco de ataque ao banco de dados.
2. Na maioria dos casos, as visualizações são somente leitura e programas externos não podem modificar dados diretamente por meio de visualizações.
3. Reutilize instruções SQL. Consultas altamente complexas podem ser agrupadas em uma tabela de visualização e os dados necessários podem ser retirados acessando diretamente a visualização. As visualizações também podem ser unidas a tabelas em consultas.
4. As visualizações podem retornar dados em um formato diferente da tabela de dados físicos. Os dados podem ser formatados quando a visualização é criada.

Crie uma visualização:

```SQL
create view `vw_emp_simple`
as
select  `eno`,
        `ename`,
        `job`,
        `dno`
  from  `tb_emp`;
```

> **Dica**: como uma visualização não contém dados, todo uso da visualização exige que a consulta subjacente seja executada novamente. Se a visualização for criada a partir de junções ou consultas aninhadas e se tornar bastante complexa, você poderá perceber que o desempenho cai drasticamente. Portanto, antes de usar uma visão complexa em uma aplicação real, você deve testá-la para ter certeza de que o desempenho é aceitável.

Com a visualização acima, podemos usar as instruções DCL da lição anterior para limitar alguns usuários à leitura de informações dos funcionários apenas por meio da visualização. Nesse caso, colunas sensíveis como salário (`sal`) e subsídio (`comm`) na tabela de funcionários não serão expostas. O código abaixo demonstra como consultar dados da visualização.

```SQL
select * from `vw_emp_simple`;
```

Como uma visualização é uma tabela virtual, os dados da visualização podem ser atualizados? Se uma visualização pode ser atualizada depende da situação. Os seguintes tipos de visualizações não podem ser atualizados:

1. Visualizações usando funções agregadas, `distinct`, `group by`, `having`, `union` ou `union all`.
2. Visualizações cuja cláusula `select` contém subconsultas.
3. Visualizações criadas sobre outras visualizações não atualizáveis.
4. Visualizações cuja cláusula `where` contém subconsultas que fazem referência a tabelas na cláusula `from`.

Solte uma visualização:

```SQL
drop view if exists `vw_emp_simple`;
```

> **Observação**: Se desejar atualizar uma visualização, você pode primeiro excluí-la com o comando acima ou usar `create or replace view` para atualizar a visualização.

Regras e limitações de visualizações:

1. As visualizações podem ser aninhadas e unidas a tabelas.
2. Uma visualização pode conter `order by`, mas um `order by` externo pode substituí-la.
3. As visualizações não podem usar índices e também não acionam a execução de gatilhos.

### Funções

As funções no MySQL são muito semelhantes às funções no Python, porque ambas são usadas para empacotar código que é relativamente independente em função e será reutilizado. Se devemos apontar uma diferença, uma delas é que as funções no MySQL podem executar instruções SQL. No exemplo abaixo, usamos uma função personalizada para truncar uma string muito longa.

A função a seguir trunca uma string longa:

```SQL
delimiter $$

create function fn_truncate_string(
    content varchar(10000),
    max_length int unsigned
) returns varchar(10000) no sql
begin
    declare result varchar(10000) default content;
    if char_length(content) > max_length then
        set result = left(content, max_length);
        set result = concat(result, '……');
    end if;
    return result;
end $$

delimiter ;
```

> **Nota 1**: `no sql` após a declaração da função diz que nenhuma instrução SQL é usada no corpo da função. Se o corpo da função precisar ler dados por meio de SQL, ele deverá ser declarado como `reads sql data`.
>
> **Nota 2**: O comando `delimiter` antes e depois da definição da função é usado para modificar o terminador. Como todas as instruções no corpo da função terminam com `;`, se não redefinirmos o terminador, quando `;` for atendido, o código será cortado e executado. Obviamente não é isso que queremos.

Chame a função em uma consulta:

```SQL
select fn_truncate_string('Walk with me through the streets of Chengdu until all the lights go out and we still do not stop', 10) as short_string;
```

```
+----------------+
| short_string   |
+----------------+
| Walk with m……  |
+----------------+
```

### Procedimentos

Um procedimento (também chamado de procedimento armazenado) é um conjunto de instruções SQL compiladas antecipadamente e armazenadas no banco de dados. Chamar um procedimento pode simplificar o trabalho dos desenvolvedores de aplicações, reduzir a comunicação com o servidor de banco de dados e também ajudar a melhorar o desempenho das operações de dados. Na verdade, até agora, todas as instruções SQL que usamos eram instruções únicas para uma ou mais tabelas, mas no desenvolvimento real frequentemente encontramos operações que precisam de várias instruções SQL para serem concluídas.

Por exemplo, quando um site de comércio eletrônico lida com um pedido de usuário, ele precisa realizar a seguinte série de processamento.

1. Verifique se existe estoque e se é suficiente.
2. Bloqueie o estoque e diminua o estoque disponível.
3. Caso o estoque seja insuficiente, entre em contato com os fornecedores ou gere registros de alerta.
4. Escreva logs e crie notificações de usuário independentemente de sucesso ou falha.

Podemos empacotar essas operações complexas por meio de um procedimento. Isto não só ajuda a garantir a consistência dos dados, mas também significa que se o negócio mudar no futuro, só precisamos de ajustar e modificar o procedimento. Para usuários que chamam o procedimento, o procedimento não expõe os detalhes das tabelas de dados, e executar um procedimento é muito mais rápido do que executar um grupo de SQL um por um.

O procedimento a seguir realiza um aumento salarial no banco de dados `hrs`:

- Departamento `10`: adicionar `300`
- Departamento `20`: adicionar `800`
- Departamento `30`: adicionar `500`

```SQL
delimiter $$

create procedure sp_upgrade_salary()
begin
    declare flag boolean default 1;
    declare continue handler for sqlexception set flag=0;

    start transaction;
    
    update tb_emp set sal=sal+300 where dno=10;
    update tb_emp set sal=sal+800 where dno=20;
    update tb_emp set sal=sal+500 where dno=30;

    if flag then
        commit;
    else
        rollback;
    end if;
end $$

delimiter ;
```

> **Observação**: O código do procedimento acima usa `start transaction` para iniciar um ambiente de transação. Para determinar se ocorreu uma exceção no código e, em seguida, confirmar ou reverter a transação, o procedimento acima define uma variável chamada `flag` e um manipulador de exceção. Se ocorrer uma exceção, `flag` será atribuído a `0`, e a estrutura da filial posterior decidirá se executará `commit` ou `rollback`.

Chame o procedimento:

```SQL
call sp_upgrade_salary();
```

Abandone o procedimento:

```SQL
drop procedure if exists sp_upgrade_salary;
```

Em um procedimento, podemos definir variáveis ​​e condições, usar instruções de ramificação e loop, operar nos resultados da consulta por meio de cursores e também usar o agendador de eventos. Não apresentaremos esses conteúdos aqui por enquanto. Embora tenhamos falado sobre muitas vantagens dos procedimentos, no desenvolvimento real, se os procedimentos forem usados ​​com frequência e uma grande quantidade de cálculos complexos forem colocados nos procedimentos, isso causará enorme pressão no servidor de banco de dados. O banco de dados muitas vezes já é o gargalo de desempenho, portanto, usar procedimentos sem dúvida está piorando as coisas. Portanto, para o desenvolvimento de produtos para a Internet, geralmente sugerimos que o banco de dados só faça um bom armazenamento e que cálculos e processamentos complexos sejam feitos por programas no servidor de aplicativos. Se o servidor de aplicativos ficar muito sobrecarregado, poderemos implantar mais facilmente vários servidores de aplicativos para compartilhar a pressão.

Se todos estiverem interessados ​​nas visualizações, funções, procedimentos e gatilhos dos quais não falamos aqui, basta ler um livro introdutório ao MySQL como *MySQL Crash Course* para uma compreensão geral, porque esses pontos de conhecimento podem não ser usados ​​em seu trabalho futuro, e aprendê-los pode ser apenas para entrevistas.

### Tópicos Adicionais

#### Formulários normais

A teoria da forma normal é a ideia orientadora para o projeto de tabelas bidimensionais em bancos de dados relacionais.

1. Primeira forma normal: cada valor da coluna deve ser atômico.
2. Segunda Forma Normal: todo atributo não-chave deve depender totalmente da chave.
3. Terceira Forma Normal: atributos não-chave devem depender apenas de chaves candidatas, e não uns dos outros.

> **Nota**: No trabalho real, por razões de eficiência, podemos muito provavelmente criar um design de forma antinormal ao projetar tabelas, ou seja, reduzimos deliberadamente o nível de forma normal e adicionamos dados redundantes para obter melhor desempenho.

#### Integridade de dados

1. Integridade da entidade: cada entidade é única.
   - `primary key` / `unique`
2. Integridade referencial: os relacionamentos não podem fazer referência a entidades que não existem.
   - `foreign key`
3. Integridade do domínio: os dados são válidos.
   - tipo de dados e comprimento
   - `not null`
   - `default`
   - `check`

     > **Note**: Before MySQL 8.x, check constraints did not work.

#### Consistência de dados

1. Transação: uma série de operações de leitura/gravação de banco de dados. Todas essas operações são bem-sucedidas ou falham.

2. Recursos ACID de uma transação
   - Atomicidade: uma transação é executada como um todo. As operações do banco de dados nele são todas executadas ou não são executadas
   - Consistência: uma transação deve garantir que o estado do banco de dados mude de um estado consistente para outro estado consistente
   - Isolamento: quando múltiplas transações são executadas ao mesmo tempo, a execução de uma transação não deve afetar a execução de outras transações
   - Durabilidade: as modificações feitas por uma transação confirmada devem ser mantidas permanentemente no banco de dados

3. Operações de transação no MySQL
   - Abra o ambiente de transação

     ```SQL
     start transaction
     ```

   - Confirme a transação

     ```SQL
     commit
     ```

   - Reverter a transação

     ```SQL
     rollback
     ```

4. Veja o nível de isolamento da transação

   ```SQL
   show variables like 'transaction_isolation';
   ```

   ```
   +-----------------------+-----------------+
   | Variable_name         | Value           |
   +-----------------------+-----------------+
   | transaction_isolation | REPEATABLE-READ |
   +-----------------------+-----------------+
   ```

Podemos ver que o nível de isolamento de transação padrão do MySQL é `REPEATABLE-READ`.

5. Modifique o nível de isolamento da transação da sessão atual

   ```SQL
   set session transaction isolation level read committed;
   ```

Visualize o nível de isolamento da transação novamente e o resultado é mostrado abaixo.

   ```
   +-----------------------+----------------+
   | Variable_name         | Value          |
   +-----------------------+----------------+
   | transaction_isolation | READ-COMMITTED |
   +-----------------------+----------------+
   ```

As transações em bancos de dados relacionais são um tópico muito amplo, porque quando múltiplas transações simultâneas acessam dados, pode haver três tipos de problemas de leitura, leitura suja, leitura não repetível e leitura fantasma, e dois tipos de problemas de atualização, primeira atualização perdida e segunda atualização perdida. Se você quiser entender esses cinco problemas, pode ler a questão 80 do [Coleção de perguntas da entrevista Java (parte 1)](https://blog.csdn.net/jackfrued/article/details/44921941). Para evitar esses problemas, os bancos de dados relacionais possuem mecanismos de bloqueio correspondentes. De acordo com diferentes objetos bloqueados, eles podem ser divididos em bloqueios em nível de tabela e bloqueios em nível de linha. De acordo com a relação de bloqueio das transações simultâneas, elas podem ser divididas em bloqueios compartilhados e bloqueios exclusivos. Mas usar bloqueios diretamente é muito problemático, então o banco de dados fornece aos usuários um mecanismo de bloqueio automático. Contanto que o usuário especifique um nível de isolamento de transação adequado, o banco de dados analisará as instruções SQL e, em seguida, adicionará bloqueios adequados aos recursos acessados ​​pela transação. Além disso, o banco de dados também manterá esses bloqueios e melhorará o desempenho do sistema por vários meios. Todas essas coisas são transparentes para os usuários. Se você deseja entender os detalhes das transações e bloqueios do MySQL, é recomendável ler o livro avançado [MySQL de alto desempenho](https://item.jd.com/11220393.html), que também é um livro clássico na área de banco de dados.

Deve ser explicado que o nível de isolamento das transações e a simultaneidade de acesso aos dados são opostos. Quanto maior for o nível de isolamento da transação, pior será a simultaneidade. Portanto, você precisa decidir qual nível de isolamento de transação usar de acordo com a aplicação específica. Não existe uma regra universal aqui.
