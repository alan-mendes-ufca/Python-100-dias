## Projeto de API de rede

Muitos aplicativos da web e aplicativos móveis agora usam um modelo de separação front-end-backend. Simplificando, o frontend ou cliente móvel interage com o backend por meio de APIs de rede, obtém os dados fornecidos pela interface e é responsável por renderizar a interface do usuário. API é a abreviação de Interface de Programação de Aplicativo. Uma API de rede geralmente se refere a um recurso que pode ser acessado por meio de uma URL. Através dessa URL, um cliente pode solicitar ao servidor que opere em um recurso e retorne o resultado. Você pode pensar desta forma: uma API de rede também é um tipo de encapsulamento. Para simplificar, ele esconde uma lógica de negócios complexa por trás de uma API simples.

Nesse sentido, uma API de rede também é uma espécie de encapsulamento: uma lógica de negócios complexa está escondida atrás de uma interface simples.

O formato geral de um URL é:

```text
protocol://username:password@host:port/path1/.../pathN/resource
```

> **Observação**: O nome de usuário, a senha, a porta e o caminho em um URL não são partes obrigatórias. Eles podem ser configurados de acordo com a necessidade real.

APIs de rede geralmente são acessadas por HTTP ou HTTPS. Uma das maiores vantagens das APIs baseadas em HTTP é que elas são simples de acessar e fáceis de usar em linguagens e aplicativos.

### Princípios de Design

#### A questão principal

Um princípio importante ao projetar APIs para clientes móveis ou PC é este: **projete a API em torno de entidades comerciais, não em torno de operações de interface de usuário**.

Se a API for projetada diretamente em torno das ações do usuário ou das funções da tela, à medida que os requisitos mudam e a interface muda, o desenvolvedor de back-end precisa continuar reescrevendo a API ou mantendo várias versões dela. Isso aumenta o custo de desenvolvimento e o custo de manutenção.

Uma abordagem melhor é tratar as entidades comerciais como recursos fornecidos pelo servidor e as URLs como identificadores desses recursos. Isso é mais simples e natural. Para ações de usuário relativamente complexas, uma “fachada” pode ser fornecida para combinar diversas interfaces quando necessário.

A lição dá um exemplo de APIs relacionadas a comentários de um site:

| API | Significado |
| --- | --- |
| `comments/show` | Obtenha a lista de comentários de uma postagem |
| `comments/by_me` | Receba meus próprios comentários |
| `comments/to_me` | Receba comentários enviados para mim |
| `comments/mentions` | Receba comentários que me mencionem |
| `comments/create` | Crie um comentário |
| `comments/destroy` | Excluir um comentário |
| `comments/reply` | Responder a um comentário |

Deve ser explicado que essas interfaces **não são APIs estilo REST**. REST é um estilo de arquitetura que muitas vezes é considerado muito adequado para aplicações de rede distribuída, mas ainda assim deve ser lido de forma crítica, pois algumas visões não são completamente corretas e alguns conteúdos podem até se contradizer.

As respostas da API geralmente são retornadas em **JSON** ou **XML**, embora o XML tenha caído em desuso. Para respostas JSON, a lição recomenda evitar valores `null` sempre que possível, porque uma vez que tais valores são mal tratados, eles podem trazer problemas desnecessários para desenvolvedores front-end e móveis, especialmente quando usam linguagens fortemente tipadas. Uma maneira de reduzir esse problema é impor restrições `not null` ou valores padrão razoáveis ​​no design do banco de dados.

#### Outros problemas

A lição também menciona várias questões práticas:

1. **Solicitações de atualização**. Projete uma API que os clientes chamem primeiro para que o servidor possa informar se uma atualização do aplicativo está disponível.
2. **Atualizações de versão**. O versionamento da API deve considerar a compatibilidade com versões anteriores. As versões novas e antigas devem permanecer acessíveis por um tempo. As informações de versão podem ser colocadas na URL ou em cabeçalhos HTTP.
3. **Dimensionamento da imagem**. Os clientes móveis podem precisar de vários tamanhos para a mesma imagem. Uma abordagem é passar parâmetros de tamanho ao buscar uma imagem. Uma abordagem melhor geralmente é usar armazenamento em nuvem ou um CDN que já suporte redimensionamento de imagem.

### Escrevendo documentação da API

A lição então usa uma API de comentários como exemplo para mostrar como a documentação da interface pode ser escrita.

Primeiro, define códigos de status globais:

| Código | Mensagem | Significado |
| --- | --- | --- |
| `10000` | Obtenha comentários com sucesso |    |
| `10001` | Criar comentário com sucesso |    |
| `10002` | Não é possível criar comentário | Falha na revisão de conteúdo |
| `10003` | O comentário foi excluído | O comentário foi removido antes de ser visualizado |

#### 1. Obtenha comentários do artigo

**OBTER** `/articles/{article-id}/comments/`

- Desenvolvedor: Wang Dachui
- Última atualização: 10 de agosto de 2018
- Etiqueta: `v1.0`
- Descrição: obtém todos os comentários do artigo especificado
- Nota de uso: retorna 20 registros por padrão e requer uma chave de identidade no cabeçalho da solicitação

Parâmetros de solicitação:

| Parâmetro | Tipo | Obrigatório | Localização | Significado |
| --- | --- | --- | --- | --- |
| `page` | inteiro | não | consulta | número da página, padrão `1` |
| `size` | inteiro | não | consulta | número de comentários por solicitação (`10~100`), padrão `20` |
| `key` | corda | sim | cabeçalho | chave de identidade do usuário |

Exemplo de resposta:

```json
{
    "code": 10000,
    "message": "Comments retrieved successfully",
    "page": 1,
    "size": 10,
    "totalPage": 35,
    "contents": [
        {
            "userId": 1700095,
            "nickname": "Wang Dachui",
            "pubDate": "July 31, 2018",
            "content": "Is the editor sick or what?"
        },
        {
            "userId": 1995322,
            "nickname": "Bai Yuanfang",
            "pubDate": "August 2, 2018",
            "content": "The person upstairs said it well."
        }
    ]
}
```

#### 2. Adicione um novo comentário ao artigo

**POST** `/articles/{article-id}/comments`

- Desenvolvedor: Wang Dachui
- Última atualização: 10 de agosto de 2018
- Etiqueta: `v1.0`
- Descrição: crie um comentário para o artigo especificado

Parâmetros de solicitação:

| Parâmetro | Tipo | Obrigatório | Localização | Significado |
| --- | --- | --- | --- | --- |
| `userId` | corda | sim | corpo | ID do usuário |
| `key` | corda | sim | cabeçalho | token de usuário |
| `content` | corda | sim | corpo | comentar conteúdo |

Exemplo de resposta:

```json
{
    "code": 10001,
    "message": "Comment created successfully",
    "comment": {
        "pubDate": "July 31, 2018",
        "content": "Is the editor sick or what?"
    }
}
```

O conselho prático final da lição é que, se você ainda não tem experiência em escrever documentação de API, plataformas online como RAP2 ou YAPI podem ajudá-lo a escrevê-la e gerenciá-la.

### Resumo

O ponto central desta lição é simples: uma API deve ser projetada em torno de recursos de negócios estáveis, e não de ações de interface frágeis. Um bom design de API de rede também inclui detalhes práticos, como controle de versão, prompts de atualização, manipulação de imagens, convenções de resposta e documentação cuidadosa. A documentação não é uma observação lateral após a interface; faz parte da própria interface.
