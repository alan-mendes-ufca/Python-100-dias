## Programação Simultânea em Python, Parte 1

Hoje em dia, os computadores que usamos já são multi-CPU ou multi-core, e os sistemas operacionais que usamos basicamente suportam "multitarefa". Isso nos permite executar vários programas ao mesmo tempo e também nos permite dividir um programa em várias subtarefas relativamente independentes e permitir que essas subtarefas sejam executadas em "paralelo" ou "simultaneamente", para que o tempo de execução do programa seja reduzido e os usuários também obtenham uma experiência melhor. Portanto, hoje, não importa qual linguagem de programação seja usada para desenvolvimento, a implementação de programação “paralela” ou “simultânea” já se tornou uma habilidade padrão para programadores. Para explicar como implementar programação “paralela” ou “concorrente” em programas Python, primeiro precisamos entender dois conceitos importantes: processo e thread.

### Threads e Processos

Quando executamos um programa através do sistema operacional, um ou mais processos são criados. Um processo é uma atividade em execução de um programa com certas funções independentes sobre uma coleção de dados. Simplificando, um processo é a unidade básica para a qual o sistema operacional aloca espaço de armazenamento. Cada processo possui seu próprio espaço de endereço, pilha de dados e outros dados auxiliares usados para rastrear a execução do processo. O sistema operacional gerencia a execução de todos os processos e aloca recursos a eles de maneira razoável. Um processo pode criar novos processos por meio de métodos como fork ou spawn para executar outras tarefas. Mas novos processos também têm seus próprios espaços de memória independentes, portanto, se dois processos quiserem compartilhar dados, eles deverão usar mecanismos de comunicação entre processos para fazê-lo, como pipes, sinais, soquetes e assim por diante.

Um processo também pode ter múltiplas linhas de execução. Simplificando, ele pode ter várias unidades de execução que podem obter agendamento da CPU, e isso é chamado de thread. Como os threads estão no mesmo processo, eles podem compartilhar o mesmo contexto. Portanto, em comparação com os processos, o compartilhamento de informações e a comunicação entre threads são mais fáceis. É claro que em um sistema de CPU de núcleo único, vários threads não podem ser executados ao mesmo tempo, porque em um momento apenas um thread pode obter a CPU. Vários threads alcançam simultaneidade compartilhando o tempo de execução da CPU.

Usar multithreading em programas geralmente traz benefícios óbvios. Os principais são melhorar o desempenho do programa e melhorar a experiência do usuário. Quase todos os softwares que usamos hoje usam multithreading. Isso pode ser confirmado pelas ferramentas de monitoramento de processos que acompanham o sistema, como Activity Monitor no macOS ou Task Manager no Windows, conforme mostrado abaixo.

Aqui, ainda precisamos enfatizar novamente dois conceitos: **simultaneidade** e **paralelismo**. **Simultaneidade** geralmente significa que em um momento apenas uma instrução pode ser executada, mas as instruções de vários threads são executadas por troca rápida. Por exemplo, um processador primeiro executa o thread A por um período de tempo, depois executa o thread B por um período de tempo e depois volta para o thread A por um período de tempo. Como o processador executa instruções e alterna muito rapidamente, as pessoas não conseguem sentir que o computador está alternando entre vários threads durante esse processo. Portanto, no nível macro, parece que vários threads estão sendo executados ao mesmo tempo, mas no nível micro, apenas um thread está realmente em execução. **Paralelismo** significa que em um determinado momento várias instruções estão sendo executadas em vários processadores ao mesmo tempo. O paralelismo deve depender de múltiplos processadores. Tanto no nível macro quanto no nível micro, vários threads podem realmente ser executados juntos ao mesmo tempo. Muitas vezes não separamos estritamente as palavras simultaneidade e paralelismo, então às vezes também tratamos multithreading, multiprocessamento e E/S assíncrona em Python como formas de implementar programação simultânea. Mas na verdade os dois primeiros também podem implementar programação paralela. Claro, há também o problema do Global Interpreter Lock, GIL, e discutiremos isso mais tarde.

### Programação Multithread

A classe `Thread` no módulo `threading` da biblioteca padrão do Python pode nos ajudar a implementar a programação multithread com muita facilidade. Usamos um exemplo de download de arquivos pela rede para comparar a diferença entre usar multithreading e não usar multithreading. O código é mostrado abaixo.

Baixando sem multithreading.

```python
import random
import time


def download(*, filename):
    start = time.time()
    print(f'Start downloading {filename}.')
    time.sleep(random.randint(3, 6))
    print(f'{filename} downloaded.')
    end = time.time()
    print(f'Download time: {end - start:.3f}s.')


def main():
    start = time.time()
    download(filename='Python-from-beginner-to-hospital.pdf')
    download(filename='MySQL-from-dropping-db-to-running-away.avi')
    download(filename='Linux-from-mastery-to-giving-up.mp4')
    end = time.time()
    print(f'Total time: {end - start:.3f}s.')


if __name__ == '__main__':
    main()
```

> **Nota**: O código acima não implementa realmente o download pela rede. Ele usa `time.sleep()` para dormir por um período de tempo para simular que o download de um arquivo precisa de algum tempo, o que é semelhante à situação real de download.

A execução do código acima pode obter resultados como o seguinte. Podemos ver que quando nosso programa tem apenas um thread em funcionamento, cada tarefa de download tem que esperar até que a tarefa de download anterior seja concluída antes de poder ser iniciada. Portanto o tempo total de execução do programa é a soma do tempo de execução das três tarefas de download.

```text

Python-from-beginner-to-hospital.pdf downloaded.
Download time: 3.005s.
Start downloading MySQL-from-dropping-db-to-running-away.avi.
MySQL-from-dropping-db-to-running-away.avi downloaded.
Download time: 5.006s.
Start downloading Linux-from-mastery-to-giving-up.mp4.
Linux-from-mastery-to-giving-up.mp4 downloaded.
Download time: 6.007s.
Total time: 14.018s.
```

Na verdade, não existe uma relação lógica de causa e efeito entre as três tarefas de download acima. Os três podem ser executados simultaneamente e a próxima tarefa de download não precisa esperar a conclusão da anterior. Por esse motivo, podemos reescrever o código acima usando programação multithread.

```python
import random
import time
from threading import Thread


def download(*, filename):
    start = time.time()
    print(f'Start downloading {filename}.')
    time.sleep(random.randint(3, 6))
    print(f'{filename} downloaded.')
    end = time.time()
    print(f'Download time: {end - start:.3f}s.')


def main():
    threads = [
        Thread(target=download, kwargs={'filename': 'Python-from-beginner-to-hospital.pdf'}),
        Thread(target=download, kwargs={'filename': 'MySQL-from-dropping-db-to-running-away.avi'}),
        Thread(target=download, kwargs={'filename': 'Linux-from-mastery-to-giving-up.mp4'})
    ]
    start = time.time()
    # Start three threads
    for thread in threads:
        thread.start()
    # Wait for the threads to end
    for thread in threads:
        thread.join()
    end = time.time()
    print(f'Total time: {end - start:.3f}s.')


if __name__ == '__main__':
    main()
```

O resultado de uma execução é mostrado abaixo.

```text

Start downloading MySQL-from-dropping-db-to-running-away.avi.
Start downloading Linux-from-mastery-to-giving-up.mp4.
MySQL-from-dropping-db-to-running-away.avi downloaded.
Download time: 3.005s.
Python-from-beginner-to-hospital.pdf downloaded.
Download time: 5.006s.
Linux-from-mastery-to-giving-up.mp4 downloaded.
Download time: 6.003s.
Total time: 6.004s.
```

A partir do resultado da execução acima, podemos descobrir que o tempo de execução de todo o programa é quase igual ao tempo de execução da tarefa de download que leva mais tempo. Isso significa que as três tarefas de download estão sendo executadas simultaneamente e uma não espera pela outra. Isto melhora claramente a eficiência de execução do programa. Simplificando, se houver unidades de execução em um programa que levam muito tempo e não houver uma relação lógica de causa e efeito entre elas, o que significa que a execução da unidade B não depende do resultado da execução da unidade A, então as unidades A e B podem ser colocadas em dois threads diferentes e executadas simultaneamente. Além de reduzir o tempo de espera de execução do programa, isso também pode trazer uma melhor experiência ao usuário, pois uma unidade bloqueada não faz com que o programa "pareça morto", pois ainda existem outras unidades no programa que podem continuar em execução.

#### Criando objetos Thread com a classe `Thread`

No código acima, podemos ver que podemos criar objetos de thread diretamente usando o construtor da classe `Thread`, e o método `start()` do objeto de thread pode iniciar um thread. Após o thread ser iniciado, ele executa a função especificada pelo parâmetro `target`, claro, com a premissa de que obtém o agendamento da CPU. Se a função de destino que o thread precisa executar possuir parâmetros, eles deverão ser especificados por meio do parâmetro `args`. Para argumentos de palavras-chave, eles podem ser passados por meio do parâmetro `kwargs`. O construtor da classe `Thread` também possui muitos outros parâmetros. Iremos explicá-los quando os encontrarmos mais tarde. Por enquanto, o que todos precisam dominar é `target`, `args` e `kwargs`.

#### Personalizando Threads Herdando a Classe `Thread`

Além da forma de criação de threads mostrada no código acima, também podemos customizar threads herdando da classe `Thread` e substituindo o método `run()`. O código específico é mostrado abaixo.

```python
import random
import time
from threading import Thread


class DownloadThread(Thread):

    def __init__(self, filename):
        self.filename = filename
        super().__init__()

    def run(self):
        start = time.time()
        print(f'Start downloading {self.filename}.')
        time.sleep(random.randint(3, 6))
        print(f'{self.filename} downloaded.')
        end = time.time()
        print(f'Download time: {end - start:.3f}s.')


def main():
    threads = [
        DownloadThread('Python-from-beginner-to-hospital.pdf'),
        DownloadThread('MySQL-from-dropping-db-to-running-away.avi'),
        DownloadThread('Linux-from-mastery-to-giving-up.mp4')
    ]
    start = time.time()
    # Start three threads
    for thread in threads:
        thread.start()
    # Wait for the threads to end
    for thread in threads:
        thread.join()
    end = time.time()
    print(f'Total time: {end - start:.3f}s.')


if __name__ == '__main__':
    main()
```

#### Usando um pool de threads

Também podemos colocar tarefas em vários threads para serem executadas usando um pool de threads. Usar threads por meio de um pool de threads deve ser a escolha ideal em programação multithread. Na verdade, criar e liberar threads traz custos relativamente altos e, frequentemente, criar e liberar threads geralmente não é uma boa escolha. Com um pool de threads, podemos preparar vários threads com antecedência. Durante o uso, não há necessidade de criar e liberar threads novamente por meio de código customizado. Podemos reutilizar diretamente os threads no pool de threads. O módulo `concurrent.futures` integrado do Python fornece suporte para pools de threads. O código é mostrado abaixo.

```python
import random
import time
from concurrent.futures import ThreadPoolExecutor
from threading import Thread


def download(*, filename):
    start = time.time()
    print(f'Start downloading {filename}.')
    time.sleep(random.randint(3, 6))
    print(f'{filename} downloaded.')
    end = time.time()
    print(f'Download time: {end - start:.3f}s.')


def main():
    with ThreadPoolExecutor(max_workers=4) as pool:
        filenames = ['Python-from-beginner-to-hospital.pdf', 'MySQL-from-dropping-db-to-running-away.avi', 'Linux-from-mastery-to-giving-up.mp4']
        start = time.time()
        for filename in filenames:
            pool.submit(download, filename=filename)
    end = time.time()
    print(f'Total time: {end - start:.3f}s.')


if __name__ == '__main__':
    main()
```

### Tópicos de Daemon

O chamado thread daemon é um thread de execução que não precisa ser mantido após o término do thread principal. O significado de "não precisa ser mantido" aqui é que os threads daemon são destruídos depois que todos os outros threads não-daemon terminam de ser executados. O que eles protegem são todos os threads não-daemon dentro do processo atual. Simplificando, os threads daemon morrem junto com o thread principal, e o ciclo de vida do thread principal é o ciclo de vida de um processo. Se isso não estiver claro, podemos examinar um simples trecho de código.

```python
import time
from threading import Thread


def display(content):
    while True:
        print(content, end='', flush=True)
        time.sleep(0.1)


def main():
    Thread(target=display, args=('Ping', )).start()
    Thread(target=display, args=('Pong', )).start()


if __name__ == '__main__':
    main()
```

> **Observação**: No código acima, definimos o parâmetro `flush` da função `print` como `True`. Isso ocorre porque se o valor do parâmetro `flush` for `False` e `print` não fizer processamento de quebra de linha, então a saída de conteúdo de cada `print` será colocada no buffer de saída do sistema operacional, até que o buffer seja preenchido pelo conteúdo de saída, e então o buffer será limpo e uma saída acontecerá. Este fenômeno é uma configuração feita pelo sistema operacional para reduzir interrupções de E/S e melhorar a utilização da CPU. Para permitir que o código produza interação intuitiva, definimos o parâmetro `flush` como `True`, forçando o buffer de saída a ser limpo em cada saída.

Depois que o código acima começar a ser executado, ele não irá parar, porque ambos os threads filhos têm loops infinitos, a menos que você pare o código manualmente. Porém, se, ao criar o objeto thread, o parâmetro denominado `daemon` for definido como `True`, os dois threads se tornarão threads daemon. Então, quando outros threads terminam, mesmo que haja um loop infinito, os dois threads daemon também morrem e não continuam a ser executados. O código é mostrado abaixo.

```python
import time
from threading import Thread


def display(content):
    while True:
        print(content, end='', flush=True)
        time.sleep(0.1)


def main():
    Thread(target=display, args=('Ping', ), daemon=True).start()
    Thread(target=display, args=('Pong', ), daemon=True).start()
    time.sleep(5)


if __name__ == '__main__':
    main()
```

No código acima, adicionamos uma linha, `time.sleep(5)`, no thread principal para deixá-lo hibernar por 5 segundos. Durante esse processo, os threads daemon que geram `Ping` e `Pong` continuam em execução, até que o thread principal termine após 5 segundos, e esses dois threads daemon também são destruídos e não continuam em execução.

### Contenção de recursos

Ao escrever código multithread, inevitavelmente encontraremos a situação em que vários threads competem pelo mesmo recurso, ou seja, pelo mesmo objeto. Nesta situação, se não houver um mecanismo adequado para proteger o recurso contestado, poderão acontecer situações inesperadas. O código abaixo cria threads `100` para transferir dinheiro para a mesma conta bancária, cujo saldo inicial é `0` yuan. Cada thread transfere `1` yuan. Em condições normais, o saldo final da nossa conta bancária deve ser `100` yuan. Mas ao executar o código abaixo, não podemos obter o resultado `100`.

```python
import time

from concurrent.futures import ThreadPoolExecutor


class Account(object):
    """Bank account"""

    def __init__(self):
        self.balance = 0.0

    def deposit(self, money):
        """Deposit money"""
        new_balance = self.balance + money
        time.sleep(0.01)
        self.balance = new_balance


def main():
    """Main function"""
    account = Account()
    with ThreadPoolExecutor(max_workers=16) as pool:
        for _ in range(100):
            pool.submit(account.deposit, 1)
    print(account.balance)


if __name__ == '__main__':
    main()
```

A classe `Account` no código acima representa uma conta bancária. Seu método `deposit` representa a ação de depósito. O parâmetro `money` representa a quantidade de dinheiro a depositar. Este método usa a função `time.sleep` para simular que a aceitação de um depósito leva algum tempo. Iniciamos threads `100` por meio do pool de threads para transferir dinheiro para uma conta, mas o código acima não pode ser executado para o resultado `100` que esperamos. Este é o problema de inconsistência de dados que pode ocorrer quando vários threads competem por um recurso. Observe a linha `14` no código acima. Quando vários threads são executados nesta linha, eles adicionam o valor do depósito ao mesmo saldo. Isto causa o fenômeno de "atualização perdida", significando que o resultado de uma modificação de dados anterior é substituído por uma modificação posterior, de modo que o resultado correto não pode ser obtido.

Para resolver o problema acima, podemos usar um mecanismo de bloqueio e proteger o código-chave que opera nos dados com um bloqueio. O módulo `threading` da biblioteca padrão do Python fornece as classes `Lock` e `RLock` para suportar mecanismos de bloqueio. Aqui não estudaremos profundamente a diferença entre eles. Recomenda-se que todos usem diretamente `RLock`. Em seguida, adicionamos um objeto de bloqueio à conta bancária e usamos o objeto de bloqueio para resolver o problema de "atualização perdida" que aconteceu ao depositar dinheiro agora há pouco. O código é mostrado abaixo.

```python
import time

from concurrent.futures import ThreadPoolExecutor
from threading import RLock


class Account(object):
    """Bank account"""

    def __init__(self):
        self.balance = 0.0
        self.lock = RLock()

    def deposit(self, money):
        # Acquire the lock
        self.lock.acquire()
        try:
            new_balance = self.balance + money
            time.sleep(0.01)
            self.balance = new_balance
        finally:
            # Release the lock
            self.lock.release()


def main():
    """Main function"""
    account = Account()
    with ThreadPoolExecutor(max_workers=16) as pool:
        for _ in range(100):
            pool.submit(account.deposit, 1)
    print(account.balance)


if __name__ == '__main__':
    main()
```

No código acima, as operações de aquisição e liberação do bloqueio também podem ser implementadas através da sintaxe de contexto. Usar a sintaxe de contexto torna o código mais simples e elegante, e essa também é a forma que recomendamos.

```python
import time

from concurrent.futures import ThreadPoolExecutor
from threading import RLock


class Account(object):
    """Bank account"""

    def __init__(self):
        self.balance = 0.0
        self.lock = RLock()

    def deposit(self, money):
        # Acquire and release the lock through context syntax
        with self.lock:
            new_balance = self.balance + money
            time.sleep(0.01)
            self.balance = new_balance


def main():
    """Main function"""
    account = Account()
    with ThreadPoolExecutor(max_workers=16) as pool:
        for _ in range(100):
            pool.submit(account.deposit, 1)
    print(account.balance)


if __name__ == '__main__':
    main()
```

Quando o saldo não é suficiente, os threads de retirada precisam parar e esperar até que os threads de depósito tenham depositado o dinheiro e, em seguida, tentar sacar novamente. ### O problema GIL

Se usarmos o interpretador Python oficial, geralmente chamado CPython, para executar programas Python, não podemos aumentar a utilização da CPU perto de 400% para uma CPU de 4 núcleos ou perto de 800% para uma CPU de 8 núcleos através de multithreading, porque o CPython é limitado pelo GIL, o Global Interpreter Lock, ao executar código. Mais especificamente, quando o CPython executa qualquer código, o thread correspondente deve primeiro obter o GIL. Então, a cada 100 instruções de bytecode, o CPython faz com que o thread que obteve o GIL libere ativamente o GIL, e só então os outros threads têm a chance de executar. Devido à existência do GIL, não importa quantos núcleos de CPU sua CPU tenha, o código Python que escrevemos não tem chance de realmente ser executado em paralelo.
