## Usando Django para projetos comerciais

> Nota: algumas ilustrações da lição original vêm de *Python Project Development in Practice* e *Mastering Django*. Ambos os livros contêm uma excelente discussão sobre o framework Django e valem a pena ser lidos se você quiser mais profundidade.

Esta lição não é um tutorial restrito. Está mais próximo de uma lista de verificação prática do conhecimento necessário em um projeto Django real.

### Solicitações e respostas HTTP

A lição então analisa a estrutura do HTTP.

- Uma solicitação HTTP é composta por: linha de solicitação + cabeçalhos de solicitação + linha em branco + corpo da mensagem opcional.
- Uma resposta HTTP é composta por: linha de status + cabeçalhos de resposta + linha em branco + corpo da mensagem.

![](./res/http-request.png)

![](./res/http-response.png)

A lição espera familiaridade com `HttpRequest` e `HttpResponse`.

Atributos e métodos importantes do `HttpRequest` incluem:

- `method`
- `path` / `get_full_path()`
- `scheme` / `is_secure()` / `get_host()` / `get_port()`
- `META` / `COOKIES`
- `GET` / `POST` / `FILES`
- `get_signed_cookie()`
- `is_ajax()`
- `body` / `content_type` / `encoding`

O middleware também pode adicionar:

- `session`
- `user`
- `site`

Os recursos importantes do `HttpResponse` incluem:

- `set_cookie()` / `set_signed_cookie()` / `delete_cookie()`
- acesso ao cabeçalho no estilo de mapeamento
- `charset`
- `content`
- `status_code`

A lição também analisa as amplas classes de código de status HTTP:

- `1xx`: a solicitação foi recebida e o processamento continua;
- `2xx`: a solicitação foi bem-sucedida;
- `3xx`: ações adicionais são necessárias para concluir a solicitação;
- `4xx`: o cliente fez uma solicitação inválida;
- `5xx`: o servidor falhou ao processar a solicitação.

Ele também aborda `JsonResponse` e exemplos de respostas simples.

```python
>>> from django.http import HttpResponse, JsonResponse
>>>
>>> response = JsonResponse({'foo': 'bar'})
>>> response.content
>>>
>>> response = JsonResponse([1, 2, 3], safe=False)
>>> response.content
>>>
>>> response = HttpResponse(b'...')
>>> response['cotent-type'] = 'application/pdf'
>>> response['content-disposition'] = 'inline; filename="xyz.pdf"'
>>> response['content-disposition'] = 'attachment; filename="xyz.pdf"'
>>>
>>> response.set_signed_cookie('foo', 'bar', salt='')
>>> response.status_code = 200
```

A lição também analisa:

O ponto principal é que o trabalho real do Django requer mais do que apenas definir algumas classes de modelo. Você precisa entender como as operações ORM são mapeadas no design relacional e quais compensações existem entre conveniência e desempenho.

### Ver funções e transações

A lição trata cada solicitação ou história de usuário como um possível problema de design de função de visualização. Também destaca uma questão mais profunda: uma solicitação pode conter diversas operações persistentes, e essas operações podem precisar ser projetadas como uma unidade atômica. É aí que aparecem os limites da transação.

Ele revisa as propriedades ACID das transações:

- **Atomicidade**
- **Consistência**
- **Isolamento**
- **Durabilidade**

Também explica os níveis de isolamento da transação:

- `Read Uncommitted`
- `Read Committed`
- `Repeatable Read`
- `Serializable`

e lembra ao leitor que um isolamento mais forte reduz problemas de simultaneidade, mas geralmente custa mais desempenho.

A lição também lista vários problemas de simultaneidade:

- atualizações perdidas;
- leituras sujas;
- leituras não repetíveis;
- leituras fantasmas.

Para controle de transações Django, introduz:

```python
ATOMIC_REQUESTS = True
```

```python
@transaction.non_atomic_requests
@transaction.atomic
```

```python
with transaction.atomic():
    pass
```

e controle manual através de:

```python
transaction.commit()
transaction.rollback()
```

### Configuração de URL

A seção URL cobre vários pontos práticos:

1. Alguns URLs podem ser ativados apenas no modo de depuração.
2. Grupos de captura nomeados podem ser usados ​​para parâmetros de caminho.
3. A configuração da URL em si não se preocupa com o método HTTP.
4. `path()` pode especificar tipos de parâmetros com mais clareza do que a captura de regex no estilo `url()`.
5. `include()` e `namespace` ajudam a resolver conflitos de nomenclatura.
6. Parâmetros extras podem ser passados ​​para visualizações por meio de dicionários em definições de URL.
7. `reverse()` e `{% url %}` suportam resolução de URL reversa.
