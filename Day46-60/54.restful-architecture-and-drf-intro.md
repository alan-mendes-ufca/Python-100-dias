## Arquitetura RESTful e noções básicas de DRF

Transformar software, plataformas e infraestrutura em serviços é algo que muitas empresas de TI vêm fazendo há muito tempo. Isso é o que as pessoas costumam chamar de SaaS (Software como Serviço), PaaS (Plataforma como Serviço) e IaaS (Infraestrutura como Serviço). Há muitas maneiras de implementar arquitetura orientada a serviços (SOA), incluindo RPC (Remote Procedure Call), Web Service e REST. No nível técnico, SOA é uma **arquitetura de software abstrata, fracamente acoplada e de granulação grossa**. No nível de negócios, as ideias centrais da SOA são "**reutilização**" e "**interoperabilidade**". Ele transforma os recursos do sistema em serviços operáveis e padrão, para que esses recursos possam ser recombinados e usados novamente. Entre as muitas maneiras de implementar SOA, REST é considerada a arquitetura mais adequada para aplicações de Internet, e uma arquitetura que segue regras REST é frequentemente chamada de arquitetura RESTful.

### Visão geral do REST

A palavra REST foi proposta por **Roy Thomas Fielding** em sua dissertação de doutorado em 2000. Roy foi um dos principais projetistas das versões 1.0 e 1.1 do protocolo HTTP, principal autor do software de servidor Apache e o primeiro presidente da Fundação Apache. Em sua tese de doutorado, Roy deu o nome REST aos seus princípios arquitetônicos para software de Internet. REST é a abreviação de **RE**apresentação **S**tate **T**transferência.

A “representação” aqui significa na verdade a representação de um “recurso”. Um recurso é uma entidade na rede ou, mais diretamente, uma informação concreta na rede. Pode ser um texto, uma imagem, uma música ou um serviço. Podemos usar um URI (Uniform Resource Identifier) para apontar para um recurso. Para obter este recurso, podemos acessar seu URI. O URI é o identificador exclusivo do recurso na Internet. Um recurso pode ter muitas formas externas de representação. A forma específica utilizada para mostrar um recurso é chamada de “representação”. Por exemplo, o texto pode ser representado em `text/plain`, `text/html`, `text/xml`, `application/json` e até mesmo em formato binário. As imagens podem ser representadas em `image/jpeg` ou `image/png`. Um URI representa apenas a entidade do recurso, não a sua representação. A rigor, o sufixo `.html` no final de alguns endereços de sites é desnecessário, pois esse sufixo mostra o formato, e o formato pertence à categoria "representação". Um URI deve representar apenas a localização do “recurso”. Sua representação exata deve ser especificada nos cabeçalhos das solicitações HTTP através dos campos `Accept` e `Content-Type`. Esses dois campos são a descrição da “representação”.

Visitar um site significa um processo de interação entre o cliente e o servidor. Neste processo, certamente estão envolvidas mudanças de dados e de estado. Os aplicativos da Web geralmente usam HTTP como protocolo de comunicação. Se o cliente quiser operar o servidor, ele deverá usar uma solicitação HTTP para fazer o servidor realizar a "transferência de estado". Como esta transferência é baseada na camada de representação, ela é chamada de “transferência de estado representacional”. O cliente usa os verbos HTTP GET, POST, PUT (ou PATCH) e DELETE, que correspondem às quatro operações básicas nos recursos. GET é usado para obter recursos, POST é usado para criar recursos e também pode ser usado para atualizar recursos, PUT (ou PATCH) é usado para atualizar recursos e DELETE é usado para excluir recursos.

Simplificando, arquitetura RESTful significa: “cada URI representa um tipo de recurso, e o cliente usa os quatro verbos HTTP para operar nos recursos do servidor, de modo que o estado de representação dos recursos é transferido”.

Quando projetamos uma aplicação Web, se precisarmos fornecer recursos ao cliente, podemos usar URIs estilo REST. Este é o primeiro passo para implementar a arquitetura RESTful. É claro que uma arquitetura RESTful real não envolve apenas URIs que parecem estilo REST. Mais importantes são as duas ideias “apatridia” e “idempotência”. Explicaremos esses dois pontos em lições posteriores. O exemplo abaixo fornece alguns URIs que correspondem ao estilo REST e você pode usá-los como referência ao projetar URIs.

| Método de solicitação (verbo HTTP) | URI | Significado |
| -------------------------- | -------------------------- | --------------------------------------------- |
| **PEGAR** | `/students/` | Receba todos os alunos |
| **PUBLICAR** | `/students/` | Crie um aluno |
| **PEGAR** | `/students/ID/` | Obtenha as informações do aluno com o ID fornecido |
| **COLOCAR** | `/students/ID/` | Atualizar as informações do aluno com o ID fornecido, fornecendo todas as informações |
| **CORREÇÃO** | `/students/ID/` | Atualizar as informações do aluno com o ID fornecido, fornecendo parte das informações |
| **EXCLUIR** | `/students/ID/` | Exclua as informações do aluno com o ID fornecido |
| **PEGAR** | `/students/ID/friends/` | Liste todos os amigos do aluno com o ID fornecido |
| **EXCLUIR** | `/students/ID/friends/ID/` | Exclua o amigo com o ID fornecido do aluno com o ID fornecido |

#### Escrevendo serializadores

O desenvolvimento da separação front-end-backend precisa que o back-end forneça interfaces de dados API para o front-end e o lado móvel e, na maioria dos casos, essas interfaces API retornam dados no formato JSON, portanto, os objetos do modelo precisam ser serializados. O DRF agrupa as classes `Serializer` e `ModelSerializer` para implementar a serialização. Herdando de `Serializer` ou `ModelSerializer`, podemos customizar serializadores e utilizá-los para transformar objetos em dicionários, conforme mostrado abaixo.

```python
from rest_framework import serializers


class SubjectSerializer(serializers.ModelSerializer):

    class Meta:
        model = Subject
        fields = '__all__'
```

O código acima herda diretamente de `ModelSerializer`, usa o atributo `model` da classe `Meta` para especificar o modelo a ser serializado e usa o atributo `fields` para especificar os campos do modelo que precisam ser serializados. Posteriormente, podemos usar esta classe na função de visualização para serializar o modelo `Subject`.

#### Escrevendo funções de visualização

A estrutura DRF oferece suporte a duas maneiras de implementar APIs de dados. Um é FBV, que significa visualizações baseadas em funções, e o outro é CBV, que significa visualizações baseadas em classes. Vamos primeiro ver como implementar uma API de dados com FBV. O código é mostrado abaixo.

```python
from rest_framework.decorators import api_view
from rest_framework.response import Response


@api_view(('GET', ))
def show_subjects(request: HttpRequest) -> HttpResponse:
    subjects = Subject.objects.all().order_by('no')
    # Create the serializer object and specify the model to serialize
    serializer = SubjectSerializer(subjects, many=True)
    # Get the dictionaries corresponding to the model through the serializer's data property
    # and return JSON data by creating a Response object
    return Response(serializer.data)
```

#### Implementando a API de dados de informações do professor

Escreva o serializador.

```python
class SubjectSimpleSerializer(serializers.ModelSerializer):

    class Meta:
        model = Subject
        fields = ('no', 'name')


class TeacherSerializer(serializers.ModelSerializer):

    class Meta:
        model = Teacher
        exclude = ('subject', )
```

Escreva a função de visualização.

```python
@api_view(('GET', ))
def show_teachers(request: HttpRequest) -> HttpResponse:
    try:
        sno = int(request.GET.get('sno'))
        subject = Subject.objects.only('name').get(no=sno)
        teachers = Teacher.objects.filter(subject=subject).defer('subject').order_by('no')
        subject_seri = SubjectSimpleSerializer(subject)
        teacher_seri = TeacherSerializer(teachers, many=True)
        return Response({'subject': subject_seri.data, 'teachers': teacher_seri.data})
    except (TypeError, ValueError, Subject.DoesNotExist):
        return Response(status=404)
```

Configurar mapeamento de URL.

```python
urlpatterns = [

    path('api/teachers/', show_teachers),

]
```

### Login de usuário com separação frontend-backend

Mencionamos antes que o HTTP não tem estado. Após o término de uma solicitação, a conexão é encerrada. Quando o servidor recebe a próxima solicitação, ele não sabe qual usuário a enviou. Mas para uma aplicação Web, o gerenciamento de estado é necessário, para que o servidor possa saber de qual usuário vem a solicitação HTTP, decidir se o usuário tem permissão para fazer a solicitação e fornecer um serviço melhor para o usuário. Esse processo é o que costumamos chamar de **gerenciamento de sessão**.

A maneira como fazíamos o gerenciamento de sessões, ou rastreamento de usuários, antes era esta: depois que o usuário efetuava login com sucesso, o servidor salvava os dados relacionados ao usuário em um objeto de sessão e, em seguida, escrevia o ID do objeto de sessão no cookie do navegador. Na solicitação seguinte, o cabeçalho da solicitação HTTP transportou os dados do cookie. O servidor leu o `sessionid` do cookie no cabeçalho da solicitação HTTP, encontrou o objeto de sessão correspondente de acordo com esse identificador e, em seguida, obteve os dados do usuário salvos na sessão anterior. Como acabamos de dizer, a arquitetura REST é a arquitetura mais adequada para aplicações da Internet. Ele enfatiza a ausência de estado do HTTP, para que o aplicativo possa manter a escalabilidade horizontal. Quando as visitas simultâneas aumentam, podemos expandir o sistema adicionando novos nós de servidor. Claramente, o método de rastreamento de usuários baseado em sessão precisa que o servidor salve os objetos da sessão. Quando novos nós de servidor são adicionados para escalabilidade horizontal, os objetos de sessão precisam ser copiados e sincronizados, e isso é obviamente muito problemático. Existem duas maneiras de resolver esse problema. Uma é configurar um servidor de cache como o Redis, permitir que vários nós do servidor compartilhem o serviço de cache e colocar os objetos de sessão diretamente no servidor de cache. A outra é desistir do rastreamento de usuários baseado em sessão e usar **rastreamento de usuários baseado em token**.

O rastreamento de usuário baseado em token significa que, após o login bem-sucedido do usuário, um marcador de identidade é gerado para o usuário e salvo no armazenamento local do navegador, como localStorage, sessionStorage ou cookie. Dessa forma, o servidor não precisa salvar o estado do usuário, portanto o dimensionamento horizontal pode ser feito facilmente. O fluxo específico de rastreamento de usuários baseado em token é o seguinte:

1. Quando o usuário fizer login, se o login for bem-sucedido, gere um token para o usuário de alguma forma. Esse token geralmente contém a identidade do usuário, o prazo de validade e outras informações, e precisa ser criptografado e ter impressões digitais para evitar falsificação ou adulteração. O servidor retorna o token para o frontend.
2. O frontend obtém o token retornado pelo servidor e o salva no armazenamento local do navegador. Ele pode ser salvo em `localStorage` ou `sessionStorage`. Para um projeto frontend usando Vue.js, ele também pode ser gerenciado através do Vuex.
3. Para um projeto que usa roteamento de front-end, sempre que a rota de front-end salta, ele pode primeiro verificar se há um token em `localStorage`. Se não houver token, vá para a página de login.
4. Cada vez que o front-end solicitar uma API de dados de back-end, carregue o token no cabeçalho da solicitação HTTP. A interface backend verifica se o cabeçalho da solicitação possui um token. Se não houver token, ou se o token for inválido ou expirado, o servidor retornará `401` de forma unificada.
5. Se o frontend receber o código de status de resposta HTTP `401`, redirecione para a página de login.

Pela descrição acima, você provavelmente descobriu que a parte mais importante do rastreamento de usuários baseado em token é gerar um token como marcador de identidade do usuário quando ele faz login com sucesso. Existem muitas maneiras de gerar um token. Uma solução mais madura é o JSON Web Token.

#### Visão geral do JWT

JSON Web Token geralmente é abreviado para JWT. É um padrão aberto, RFC 7519. À medida que a arquitetura RESTful se tornou popular, mais e mais projetos usam JWT como forma de autenticar a identidade do usuário. JWT é feito codificando três objetos JSON, unindo-os com `.` e combinando-os em uma string. Esses três objetos JSON são o cabeçalho, a carga útil e a assinatura, conforme mostrado abaixo.

![](res/json-web-token.png)

1. Cabeçalho

    ```json
    {
      "alg": "HS256",
      "typ": "JWT"
    }
    ```

    In it, the `alg` property means the algorithm used for the signature, and the default is HMAC SHA256, written shortly as `HS256`. The `typ` property means the type of this token, and in JWT it is always written as `JWT`.

2. Carga útil

    The payload part is used to store the actual data that needs to be passed. The official JWT documentation defines seven optional fields:

    - `iss` - issuer
    - `exp` - expiration time
    - `sub` - subject
    - `aud` - audience
    - `nbf` - not before
    - `iat` - issued at
    - `jti` - ID

    Besides the fields officially defined, we can add custom fields according to the needs of the application, as shown below.

    ```json
    {
      "sub": "1234567890",
      "nickname": "jackfrued",
      "role": "admin"
    }
    ```

3. Assinatura

    The signature part generates a fingerprint for the first two parts to prevent data forgery and tampering. To create the signature, we first need to specify a secret key. This key is known only by the server and must not be leaked to users. Then, use the signature algorithm specified in the header, and the default is `HS256`, to generate the signature according to the formula below.

    ```python
    HS256(base64Encode(header) + '.' + base64Encode(payload), secret)
    ```

    After the signature is calculated, join the three parts, header, payload, and signature, into one string, and separate each part with `.`, then one JWT is generated.

#### Vantagens e desvantagens do JWT

As vantagens de usar JWT são muito claras, incluindo:

1. É mais fácil implementar o escalonamento horizontal, pois o token é salvo no navegador e o servidor não precisa fazer gerenciamento de estado.
2. É mais fácil prevenir ataques CSRF, pois a adição do token de `localStorage` ou `sessionStorage` ao cabeçalho da solicitação deve ser feita por código JavaScript, em vez de ser adicionado automaticamente ao cabeçalho da solicitação.
3. Pode evitar falsificação e adulteração, porque o JWT possui uma assinatura. Os tokens forjados ou adulterados não podem passar na verificação de assinatura e serão tratados como tokens inválidos.

É claro que nenhuma tecnologia tem apenas vantagens e nenhuma desvantagem. O JWT também tem muitas desvantagens e todos devem prestar atenção a elas ao usá-lo. Essas desvantagens incluem:

1. Ele pode sofrer ataques XSS ou ataques de script entre sites, onde scripts maliciosos são injetados para executar código JavaScript e obter o token do usuário.
2. Antes de o token expirar, um token emitido não pode ser revogado. Para resolver esse problema, ainda são necessários middleware e código extras.
3. JWT é o token de identidade do usuário. Depois de vazar, qualquer pessoa pode obter todas as permissões desse usuário. Para reduzir o risco após o roubo de um token, o período de validade do JWT deve ser curto. Para algumas permissões mais importantes, o usuário deve ser autenticado novamente de alguma outra forma ao utilizá-las, por exemplo, por código de verificação SMS.

#### Usando PyJWT

No código Python, podemos usar a biblioteca de terceiros `PyJWT` para gerar e verificar o JWT. O comando abaixo instala `PyJWT`.


Gere um token.

```python
payload = {
    'exp': datetime.datetime.utcnow() + datetime.timedelta(days=1),
    'userid': 10001
}
token = jwt.encode(payload, settings.SECRET_KEY).decode()
```
