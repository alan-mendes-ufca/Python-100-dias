## Otimização de desempenho MySQL

### Princípios Básicos

Se você deseja que o MySQL tenha o melhor desempenho, existem três princípios básicos que devem ser seguidos.

1. Deixe o MySQL retornar ao seu dever básico de armazenamento. O banco de dados deve ser usado apenas para armazenamento de dados. Não deve fazer cálculos complexos ou carregar lógica de negócios. Armazenamento e computação devem ser mantidos separados.
2. Ao consultar dados, tente consultar uma única tabela sempre que possível e reduza consultas entre bancos de dados e junções excessivas.
3. Evite uma série de fatores comuns que prejudicam o desempenho, como transações grandes, instruções SQL gigantes, operações em lote enormes e campos grandes.

Esses assassinos de desempenho merecem um pouco mais de explicação:

- **Grandes transações** envolvem muitas etapas e muitas tabelas ou campos. Eles facilmente causam contenção de recursos e até mesmo impasses. Se ocorrer uma reversão, os recursos poderão permanecer ocupados por muito tempo.
- **Instruções SQL grandes** geralmente significam muitas junções de tabelas. Uma vez que uma consulta une mais de três tabelas, o MySQL normalmente consome muito mais recursos e o desempenho cai muito.
- **Lotes grandes** podem reduzir a sobrecarga de execução de uma instrução SQL por vez, mas devem ser testados com cuidado e executados fora dos horários de pico.
- **Campos grandes** como `blob` e `text` devem ser usados ​​o mínimo possível. Se forem necessários, é melhor separá-los da tabela principal de negócios e evitar recuperá-los e atualizá-los com frequência.

### Design de banco de dados e tabelas

Ao projetar bancos de dados e tabelas, a lição enfatiza os seguintes pontos:

1. Use InnoDB como mecanismo de armazenamento padrão e desative MyISAM.
2. Use `UTF8mb4` como conjunto de caracteres padrão.
3. Desative o comportamento do nome da tabela que diferencia maiúsculas de minúsculas configurando `lower_case_table_names=1` quando necessário.
4. Evite depender de procedimentos armazenados, gatilhos, visualizações e eventos para transportar lógica de negócios em sistemas de produção.
5. Mantenha o número de tabelas em uma instância de banco de dados sob controle.

Para tabelas InnoDB, a lição dá várias sugestões concretas:

1. Use uma chave primária inteira não assinada, geralmente com `auto_increment`, e não atualize a chave primária manualmente.
2. Adicione um `comment` a cada coluna.
3. Especifique explicitamente `engine` ao criar tabelas.
4. Certifique-se de que cada tabela tenha campos principais, como chave primária, hora de criação e hora de modificação.
5. Prefira campos `not null` com padrões razoáveis, porque `null` é mais difícil para o MySQL otimizar.
6. Mantenha o número de colunas em uma única tabela sob controle. Se houver muitos campos, considere a divisão vertical.
7. Evite `enum` e `set`.
8. Não armazene arquivos grandes diretamente como `blob` se salvar um caminho for suficiente.
9. Use `decimal` em vez de tipos de ponto flutuante para dinheiro e outros valores sensíveis à precisão.
10. Armazene endereços IP como `int unsigned` em vez de `char(15)`.

### Usando índices

O design do índice é uma das partes mais importantes da otimização do MySQL. A lição analisa vários princípios básicos:

1. As melhores colunas de índice não são necessariamente as colunas retornadas em `select`; geralmente são as colunas usadas nas condições `where` ou condições de junção.
2. Colunas com alta exclusividade geralmente produzem índices melhores do que colunas com muitos valores repetidos.
3. Para índices de string, geralmente é melhor criar um índice de prefixo com comprimento curto.
4. Um índice composto deve ser projetado tendo em mente a regra do prefixo mais à esquerda.
5. Não abuse dos índices. Eles ocupam espaço de armazenamento e também prejudicam o desempenho de gravação.
6. Tenha cuidado com situações que tornam os índices inválidos, como curingas iniciais em consultas `like` ou determinadas condições negativas.

### Procedimentos e particionamento

Os procedimentos armazenados podem simplificar o trabalho dos desenvolvedores de aplicativos e reduzir a comunicação entre o aplicativo e o servidor de banco de dados, para que possam ajudar a melhorar o desempenho nas operações de dados. Mas o conselho mais amplo da lição ainda é evitar inserir muita lógica de negócios no banco de dados.

MySQL também suporta particionamento. O particionamento pode ajudar a armazenar mais dados, melhorar alguns padrões de consulta, aumentar o rendimento e remover dados expirados com mais rapidez. A lição apresenta vários tipos de partição:

1. **Particionamento RANGE**

   ```sql
   CREATE TABLE tb_emp (
       eno INT NOT NULL,
       ename VARCHAR(20) NOT NULL,
       job VARCHAR(10) NOT NULL,
       hiredate DATE NOT NULL,
       dno INT NOT NULL
   )
   PARTITION BY RANGE(YEAR(hiredate)) (
       PARTITION p0 VALUES LESS THAN (1960),
       PARTITION p1 VALUES LESS THAN (1970),
       PARTITION p2 VALUES LESS THAN (1980),
       PARTITION p3 VALUES LESS THAN (1990),
       PARTITION p4 VALUES LESS THAN MAXVALUE
   );
   ```

### Otimização SQL

A lição fornece várias maneiras concretas de otimizar o SQL.

#### 1. Encontre SQL de baixa eficiência com o log de consulta lenta

Você pode inspecionar a configuração atual:

```sql
show variables like 'slow_query%';
show variables like 'long_query_time';
```

Exemplo de saída:

```sql
+---------------------------+----------------------------------+
| Variable_name             | Value                            |
+---------------------------+----------------------------------+
| slow_query_log            | OFF                              |
| slow_query_log_file       | /mysql/data/localhost-slow.log   |
+---------------------------+----------------------------------+

+-----------------+-----------+
| Variable_name   | Value     |
+-----------------+-----------+
| long_query_time | 10.000000 |
+-----------------+-----------+
```

Você pode criar e configurar o log de consulta lenta:

```bash
touch /var/log/mysqld-slow.log
chown mysql /var/log/mysqld-slow.log
```

```sql
set global slow_query_log_file='/var/log/mysqld-slow.log';
set global slow_query_log='ON';
set global long_query_time=1;
```

ou habilite-o diretamente no arquivo de configuração do MySQL.

#### 2. Use `EXPLAIN` para inspecionar planos de execução

Por exemplo:

```sql
explain select ename, job, sal from tb_emp where dno=20\G
```

Exemplo de saída:

```sql
*************************** 1. row ***************************
           id: 1
  select_type: SIMPLE
        table: tb_emp
         type: ref
possible_keys: fk_emp_dno
          key: fk_emp_dno
      key_len: 5
          ref: const
         rows: 7
        Extra: NULL
1 row in set (0.00 sec)
```

A lição destaca especialmente campos como:

- `select_type`
- `table`
- `type`
- `possible_keys`
- `key`
- `key_len`
- `rows`
- `extra`

Isso ajuda você a entender se a consulta está fazendo uma varredura completa da tabela, se os índices estão sendo usados ​​e quão cara a consulta provavelmente será.

#### 3. Use ferramentas de criação de perfil

Você pode inspecionar se a criação de perfil está ativada:

```sql
select @@have_profiling;
select @@profiling;
set profiling=1;
```

Então você pode executar o SQL e inspecionar o perfil:

```sql
show profiles;
show profile for query 1;
```

Exemplo de saída:

#### 4. Otimize as operações CRUD

A lição menciona várias ideias práticas:

- Para `insert`, inserir várias linhas em uma instrução geralmente é melhor do que enviar muitas inserções pequenas.
- Ao carregar dados de um arquivo de texto, `load data infile` costuma ser muito mais rápido do que inserções repetidas.
- `order by` pode ser otimizado quando a condição de classificação corresponde a um índice.
- Para `group by`, usar `order by null` pode evitar sobrecarga desnecessária de classificação.
- Algumas subconsultas podem ser substituídas por junções para melhor desempenho.
- As condições `or` só se beneficiam dos índices quando todas as condições relevantes são indexadas.
- Consultas de paginação como `limit 10000, 20` são caras. Um padrão mais eficiente é paginar uma subconsulta indexada e depois juntar-se novamente à tabela principal.

Por exemplo:

```sql
select * from tb_emp order by ename limit 10000, 20;
select * from tb_emp t1
inner join (
    select eno from tb_emp order by ename limit 10000, 20
) t2 on t1.eno = t2.eno;
```

A segunda forma geralmente é melhor se `ename` já tiver um índice.

### Resumo

A otimização de desempenho do MySQL não é um único truque. É uma combinação de disciplina básica de design, estrutura de tabela cuidadosa, indexação razoável, particionamento quando necessário e análise SQL disciplinada com ferramentas como log de consulta lenta, `EXPLAIN` e criação de perfil. O ponto principal da lição é simples: o bom desempenho vem do uso correto do banco de dados, e não da tentativa posterior de resgatar um design ruim.
