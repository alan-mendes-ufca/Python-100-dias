### Views

A view is an object in a relational database that combines the result set made from a group of query statements into a queryable data-table object. Simply speaking, a view is a virtual table. But unlike a data table, a data table is an entity structure, while a view is a virtual structure. You can also understand a view as an SQL statement saved in the database and given a name.

Using views can bring the following benefits:

1. They can hide the physical data table, so outside programs cannot know the real data structure. Visitors can use part of the table instead of the whole table, which lowers the risk of the database being attacked.
2. In most cases views are read-only, and outside programs cannot directly modify data through views.
3. Reuse SQL statements. Highly complex queries can be wrapped into a view table, and needed data can be taken out by directly accessing the view. Views can also be joined with tables in queries.
4. Views can return data in a different format from the physical data table. Data can be formatted when the view is created.

Create a view:

```SQL
create view `vw_emp_simple`
as
select  `eno`,
        `ename`,
        `job`,
        `dno`
  from  `tb_emp`;
```

> **Tip**: Because a view does not contain data, every use of the view requires the underlying query to be executed again. If the view is built from joins or nested queries and becomes fairly complex, you may find that performance drops sharply. So before using a complex view in a real application, you should test it to make sure the performance is acceptable.

With the view above, we can use the DCL statements from the previous lesson to limit some users to reading employee information only through the view. In that case, sensitive columns such as salary (`sal`) and allowance (`comm`) in the employee table will not be exposed. The code below demonstrates how to query data from the view.

```SQL
select * from `vw_emp_simple`;
```

Since a view is a virtual table, can the data in the view be updated? Whether a view can be updated depends on the situation. The following types of views cannot be updated:

1. Views using aggregate functions, `distinct`, `group by`, `having`, `union`, or `union all`.
2. Views whose `select` clause contains subqueries.
3. Views built on top of other non-updatable views.
4. Views whose `where` clause contains subqueries referencing tables in the `from` clause.

Drop a view:

```SQL
drop view if exists `vw_emp_simple`;
```

> **Note**: If you want to update a view, you can first delete the view with the command above, or use `create or replace view` to update the view.

Rules and limitations of views:

1. Views can be nested and joined with tables.
2. A view may contain `order by`, but an outer `order by` can override it.
3. Views cannot use indexes, and they also do not trigger triggers to run.

### Functions

Functions in MySQL are very similar to functions in Python, because both are used to package code that is relatively independent in function and will be reused. If we must point out a difference, one is that functions in MySQL can execute SQL statements. In the example below, we use a custom function to truncate an overlong string.

The following function truncates a long string:

```SQL
delimiter $$

create function fn_truncate_string(
    content varchar(10000),
    max_length int unsigned
) returns varchar(10000) no sql
begin
    declare result varchar(10000) default content;
    if char_length(content) > max_length then
        set result = left(content, max_length);
        set result = concat(result, '……');
    end if;
    return result;
end $$

delimiter ;
```

> **Note 1**: `no sql` after the function declaration says that no SQL statement is used in the function body. If the function body needs to read data through SQL, it should be declared as `reads sql data`.
>
> **Note 2**: The `delimiter` command before and after the function definition is used to modify the terminator. Because the statements in the function body all end with `;`, if we do not redefine the terminator, then when `;` is met, the code will be cut off and executed. Obviously this is not what we want.

Call the function in a query:

```SQL
select fn_truncate_string('Walk with me through the streets of Chengdu until all the lights go out and we still do not stop', 10) as short_string;
```

```
+----------------+
| short_string   |
+----------------+
| Walk with m……  |
+----------------+
```

### Procedures

A procedure (also called a stored procedure) is a set of SQL statements compiled in advance and stored in the database. Calling a procedure can simplify the work of application developers, reduce communication with the database server, and also help improve the performance of data operations. In fact, up to now, all the SQL statements we used were single statements for one or more tables, but in actual development we often meet operations that need multiple SQL statements to finish.

For example, when an e-commerce website handles a user order, it needs to do the following series of processing.

1. Check whether inventory exists and whether it is sufficient.
2. Lock inventory and decrement available stock.
3. If stock is insufficient, contact suppliers or generate warning records.
4. Write logs and create user notifications regardless of success or failure.

We can package these complex operations through a procedure. This not only helps guarantee data consistency, but also means that if the business changes in the future, we only need to adjust and modify the procedure. For users who call the procedure, the procedure does not expose the details of the data tables, and executing a procedure is much faster than executing a group of SQL one by one.

The following procedure performs a salary increase in the `hrs` database:

- Department `10`: add `300`
- Department `20`: add `800`
- Department `30`: add `500`

```SQL
delimiter $$

create procedure sp_upgrade_salary()
begin
    declare flag boolean default 1;
    declare continue handler for sqlexception set flag=0;

    start transaction;
    
    update tb_emp set sal=sal+300 where dno=10;
    update tb_emp set sal=sal+800 where dno=20;
    update tb_emp set sal=sal+500 where dno=30;

    if flag then
        commit;
    else
        rollback;
    end if;
end $$

delimiter ;
```

> **Note**: The procedure code above uses `start transaction` to start a transaction environment. To determine whether an exception happened in the code, and then commit or roll back the transaction, the procedure above defines a variable named `flag` and an exception handler. If an exception happens, `flag` will be assigned `0`, and the later branch structure will decide whether to execute `commit` or `rollback`.

Call the procedure:

```SQL
call sp_upgrade_salary();
```

Drop the procedure:

```SQL
drop procedure if exists sp_upgrade_salary;
```

In a procedure, we can define variables and conditions, use branching and loop statements, operate on query results through cursors, and also use the event scheduler. We will not introduce these contents here for now. Although we talked about many advantages of procedures, in actual development, if procedures are used frequently and a large amount of complex calculation is put into procedures, it will cause huge pressure on the database server. The database is often already the performance bottleneck, so using procedures is no doubt making things worse. Therefore, for internet product development, we usually suggest that the database should only do storage well, and complex calculation and processing should be done by programs on the application server. If the application server becomes too overloaded, we can more easily deploy multiple application servers to share the pressure.

If everyone is interested in the views, functions, procedures, and triggers we did not talk about here, it is enough to read an introductory MySQL book such as *MySQL Crash Course* for a general understanding, because these knowledge points may not be used in your future work, and learning them may only be for interviews.

### Additional Topics

#### Normal Forms

Normal-form theory is the guiding idea for designing two-dimensional tables in relational databases.

1. First Normal Form: every column value must be atomic.
2. Second Normal Form: every non-key attribute must fully depend on the key.
3. Third Normal Form: non-key attributes should depend only on candidate keys, not on each other.

> **Note**: In actual work, for efficiency reasons, we may very likely make an anti-normal-form design when designing tables, that is, we deliberately lower the normal-form level and add redundant data to get better performance.

#### Data Integrity

1. Entity integrity: each entity is unique.
   - `primary key` / `unique`
2. Referential integrity: relationships cannot reference entities that do not exist.
   - `foreign key`
3. Domain integrity: data is valid.
   - data type and length
   - `not null`
   - `default`
   - `check`

     > **Note**: Before MySQL 8.x, check constraints did not work.

#### Data Consistency

1. Transaction: a series of database read/write operations. These operations either all succeed or all fail.

2. ACID features of a transaction
   - Atomicity: a transaction is executed as a whole. The database operations in it are either all executed or not executed at all
   - Consistency: a transaction should make sure the database state changes from one consistent state to another consistent state
   - Isolation: when multiple transactions are executed at the same time, the execution of one transaction should not affect the execution of other transactions
   - Durability: modifications made by a committed transaction should be kept in the database permanently

3. Transaction operations in MySQL
   - Open the transaction environment

     ```SQL
     start transaction
     ```

   - Commit the transaction

     ```SQL
     commit
     ```

   - Roll back the transaction

     ```SQL
     rollback
     ```

4. View the transaction isolation level

   ```SQL
   show variables like 'transaction_isolation';
   ```

   ```
   +-----------------------+-----------------+
   | Variable_name         | Value           |
   +-----------------------+-----------------+
   | transaction_isolation | REPEATABLE-READ |
   +-----------------------+-----------------+
   ```

   We can see that MySQL's default transaction isolation level is `REPEATABLE-READ`.

5. Modify the transaction isolation level of the current session

   ```SQL
   set session transaction isolation level read committed;
   ```

   View the transaction isolation level again, and the result is shown below.

   ```
   +-----------------------+----------------+
   | Variable_name         | Value          |
   +-----------------------+----------------+
   | transaction_isolation | READ-COMMITTED |
   +-----------------------+----------------+
   ```

Transactions in relational databases are a very large topic, because when multiple concurrent transactions access data, there may be three kinds of read problems, dirty read, non-repeatable read, and phantom read, and two kinds of update problems, first lost update and second lost update. If you want to understand these five problems, you can read question 80 of [Java Interview Questions Collection (Part 1)](https://blog.csdn.net/jackfrued/article/details/44921941). In order to avoid these problems, relational databases have matching lock mechanisms underneath. According to different locked objects, they can be divided into table-level locks and row-level locks. According to the locking relationship of concurrent transactions, they can be divided into shared locks and exclusive locks. But using locks directly is very troublesome, so the database provides users with an automatic locking mechanism. As long as the user specifies a proper transaction isolation level, the database will analyze the SQL statements and then add suitable locks to the resources accessed by the transaction. In addition, the database will also maintain these locks and improve system performance by various means. All these things are transparent to users. If you want to understand the details of MySQL transactions and locks, it is recommended to read the advanced book [High Performance MySQL](https://item.jd.com/11220393.html), which is also a classic book in the database field.

The ANSI/ISO SQL 92 standard defines four levels of transaction isolation, as shown in the table below. It should be explained that transaction isolation level and concurrency of data access are opposed to each other. The higher the transaction isolation level is, the worse the concurrency is. So you need to decide which transaction isolation level to use according to the specific application. There is no universal rule here.
