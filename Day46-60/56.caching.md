## Usando cache

Normalmente, o gargalo de desempenho de uma aplicação Web aparece no banco de dados relacional. Quando o acesso simultâneo é grande, se todas as solicitações precisarem do banco de dados relacional para finalizar as operações de persistência de dados, o banco de dados certamente ficará sobrecarregado. Uma das maneiras mais importantes de otimizar o desempenho de aplicativos Web é usar o cache. Coloque antecipadamente esses conjuntos de dados que não são grandes, mas que são acessados com muita frequência, em um servidor de cache. Esta é novamente uma forma típica de trocar espaço por tempo. Normalmente, o servidor de cache coloca os dados diretamente na memória e usa estratégias de acesso a dados muito eficientes, como armazenamento de hash e pares de valores-chave, de modo que seu desempenho de leitura e gravação é muito melhor do que o de um banco de dados relacional. Por causa disso, podemos conectar um aplicativo Web a um servidor de cache para otimizar seu desempenho, e uma escolha muito boa é o Redis.

### Conectando um projeto Django ao Redis

Nas lições anteriores já apresentamos a instalação e uso do Redis, portanto não vamos repetir aqui. Se precisar conectar o Redis em um projeto Django, você pode usar a biblioteca de terceiros `django-redis`. Esta biblioteca de terceiros depende de outra biblioteca de terceiros chamada `redis`, que agrupa muitas operações do Redis.

Instale `django-redis`.

```bash
pip install django-redis
```

Modifique a configuração relacionada ao cache no arquivo de configurações do Django.

```python
CACHES = {
    'default': {
        # Specify connecting to Redis through django-redis
        'BACKEND': 'django_redis.cache.RedisCache',
        # URL of the Redis server
        'LOCATION': ['redis://1.2.3.4:6379/0', ],
        # Prefix of keys in Redis, to solve naming conflicts
        'KEY_PREFIX': 'vote',
        # Other configuration options
        'OPTIONS': {
            'CLIENT_CLASS': 'django_redis.client.DefaultClient',
            # Connection pool parameters, with several Redis connections prepared in advance
            'CONNECTION_POOL_KWARGS': {
                # Maximum connection count
                'max_connections': 512,
            },
            # Password for connecting to Redis
            'PASSWORD': 'foobared',
        }
    },
}
```

Neste ponto, nosso projeto Django já pode se conectar ao Redis. A seguir, vamos modificar o código do projeto e usar o Redis para fornecer serviço de cache para a API obter dados do assunto.

### Fornecendo serviço de cache para visualizações

#### Cache Declarativo

Cache declarativo significa não alterar o código original, mas usar decoradores, ou proxies, em Python para adicionar capacidade de cache ao código original. Para FBV, o código é mostrado abaixo.

```python
from django.views.decorators.cache import cache_page


@api_view(('GET', ))
@cache_page(timeout=86400, cache='default')
def show_subjects(request):
    """Get subject data"""
    queryset = Subject.objects.all()
    data = SubjectSerializer(queryset, many=True).data
    return Response({'code': 20000, 'subjects': data})
```

O código acima usa o decorador `cache_page` empacotado do Django para armazenar em cache o valor de retorno da função de visualização, ou seja, o objeto de resposta. O objetivo original de `cache_page` é armazenar em cache a página renderizada pela função de visualização. Para uma função de visualização que retorna dados JSON, isso é igual ao armazenamento em cache dos dados JSON. Ao usar o decorador `cache_page`, podemos passar o parâmetro `timeout` para especificar o tempo de expiração do cache e também podemos usar o parâmetro `cache` para especificar qual grupo de serviços de cache deve ser usado para armazenar dados em cache. Um projeto Django permite que vários grupos de serviços de cache sejam configurados no arquivo de configurações. O `cache='default'` acima especifica o uso do serviço de cache padrão, pois nas configurações anteriores configuramos apenas o serviço de cache denominado `default`. O valor de retorno da função de visualização será serializado em uma string de bytes e colocado no Redis. O tipo `str` no Redis pode receber strings de bytes. Também não precisamos lidar sozinhos com a serialização e desserialização de dados armazenados em cache, porque o decorador `cache_page` chama `RedisCache` na biblioteca `django-redis` para se conectar ao Redis. Esta classe usa `DefaultClient` para se conectar ao Redis e usa [serialização de picles](https://python3-cookbook.readthedocs.io/zh_CN/latest/c05/p21_serializing_python_objects.html). `django_redis.serializers.pickle.PickleSerializer` é a classe serializadora padrão.

Se o cache não contiver dados do assunto, quando a API de dados do assunto for acessada, nossa função de visualização enviará uma instrução SQL ao banco de dados executando `Subject.objects.all()` para obter os dados. O valor de retorno da função de visualização será armazenado em cache, portanto, quando esta função de visualização for solicitada na próxima vez, se o cache não tiver expirado, o valor de retorno da função de visualização poderá ser obtido diretamente do cache e não haverá necessidade de consultar o banco de dados novamente. Se você quiser entender o uso do cache, você pode configurar os logs do banco de dados ou usar Django-Debug-Toolbar para verificar. Quando a API de dados do assunto for acessada pela primeira vez, você verá a instrução SQL usada para consultar os dados do assunto. Quando os dados do assunto são obtidos novamente, nenhuma instrução SQL é enviada ao banco de dados, pois os dados podem ser obtidos diretamente do cache.

Para CBV, podemos usar o decorador do Django chamado `method_decorator` para colocar a função decoradora `cache_page` em um método da classe. O efeito é o mesmo do código acima. Uma coisa que todos deveriam prestar atenção é que o decorador `cache_page` não pode ser colocado diretamente na classe, pois é um decorador de funções. É por isso que o Django fornece `method_decorator` para resolver este problema. Claramente, `method_decorator` é um decorador para decorar métodos de classe.

```python
from django.utils.decorators import method_decorator
from django.views.decorators.cache import cache_page


@method_decorator(decorator=cache_page(timeout=86400, cache='default'), name='get')
class SubjectView(ListAPIView):
    """View class for getting subject data"""
    queryset = Subject.objects.all()
    serializer_class = SubjectSerializer
```

#### Cache Programático

Cache programático significa usar o serviço de cache por meio de código escrito por nós mesmos. Embora esta forma tenha um pouco mais de código, em comparação com o cache declarativo, é mais flexível na forma como utiliza e opera o cache, por isso é usado com mais frequência no desenvolvimento real. O código abaixo remove o decorador `cache_page` usado antes e obtém diretamente uma conexão Redis por meio da função `get_redis_connection` fornecida por `django-redis` para operar o Redis.

```python
def show_subjects(request):
    """Get subject data"""
    redis_cli = get_redis_connection()
    # First try to get subject data from the cache
    data = redis_cli.get('vote:polls:subjects')
    if data:
        # If subject data is got, do deserialization
        data = json.loads(data)
    else:
        # If subject data is not got from the cache, query the database
        queryset = Subject.objects.all()
        data = SubjectSerializer(queryset, many=True).data
        # Serialize the queried subject data and put it into the cache
        redis_cli.set('vote:polls:subjects', json.dumps(data), ex=86400)
    return Response({'code': 20000, 'subjects': data})
```

Deve ser explicado que o framework Django fornece duas variáveis prontas, `cache` e `caches`, para suportar operações de cache. O primeiro acessa o cache padrão, que é o cache denominado `default`, e o último pode obter o serviço de cache especificado por meio de indexação, por exemplo `caches['default']`. O envio de mensagens `get` e `set` para o objeto `cache` pode implementar leitura e gravação de cache, mas as operações desta forma são limitadas e não são tão flexíveis quanto a forma usada no código acima. Há outra coisa digna de nota. Como o objeto de conexão Redis passou por `get_redis_connection` pode enviar muitas operações para Redis, incluindo operações perigosas como `FLUSHDB` e `SHUTDOWN`, no desenvolvimento de projetos comerciais reais, as pessoas geralmente agrupam `django-redis` mais uma vez, por exemplo, agrupam uma classe de utilitário que fornece apenas os métodos de operação de cache necessários ao projeto, portanto, o possível risco de diretamente o uso de `get_redis_connection` é evitado. É claro que agrupar as operações de cache por nós mesmos também pode usar as formas "Read Through" e "Write Through" para atualizar o cache, e isso será apresentado a seguir.

### Problemas relacionados ao cache

#### Atualizando dados em cache

Ao usar o cache, uma questão que deve ser entendida claramente é: quando os dados mudam, como os dados no cache devem ser atualizados? Normalmente, a atualização do cache segue os seguintes padrões:

1. Padrão de cache à parte
2. Padrão de leitura/gravação
3. Escreva atrás do padrão de cache

A forma específica do primeiro padrão é: quando os dados são atualizados, primeiro atualize o banco de dados e depois exclua o cache. Observe que não devemos usar a forma de primeiro atualizar o banco de dados e depois atualizar o cache, e também não devemos usar a forma de primeiro excluir o cache e depois atualizar o banco de dados. Você mesmo pode pensar no motivo, especialmente no caso em que há operações simultâneas de leitura e gravação. Claro que a forma de primeiro atualizar o banco de dados e depois deletar o cache também tem risco teórico, mas a chance de acontecer algum problema é muito baixa, por isso muitos projetos utilizam essa forma.

Na primeira forma, o desenvolvedor que escreve o código de negócios precisa cuidar sozinho das operações em dois sistemas de armazenamento, o cache e o banco de dados relacional, de modo que o código se torna muito difícil de escrever. A ideia principal da segunda forma é transformar o sistema de armazenamento de back-end em um conjunto de código, e a manutenção do cache é envolvida nesse conjunto de código. Entre eles, Read Through significa atualizar o cache durante as operações de consulta. Ou seja, quando o cache se torna inválido, o próprio serviço de cache é responsável por carregar os dados, e isso é transparente para o lado da aplicação. Write Through significa que quando os dados forem atualizados, se o cache não for atingido, atualize o banco de dados diretamente e retorne. Se o cache for atingido, atualize-o primeiro e depois deixe o serviço de cache atualizar o banco de dados sozinho, de forma síncrona. Como dissemos há pouco, se você agrupar as operações Redis no projeto mais uma vez, poderá implementar os padrões Read Through e Write Through. Embora isso acrescente algum trabalho, certamente é algo que funciona uma vez e traz benefícios por muito tempo.

A terceira forma é que quando os dados são atualizados, apenas o cache é atualizado e o banco de dados não é atualizado. O serviço de cache então **atualiza o banco de dados de forma assíncrona em lotes**. Dessa forma, você pode melhorar muito o desempenho, mas o custo é abrir mão da **forte consistência** dos dados. A lógica de implementação da terceira via é relativamente complexa, pois precisa rastrear quais dados foram atualizados e, em seguida, atualizá-los para a camada persistente em lotes.

#### Penetração de cache

O cache é adicionado como uma camada intermediária para reduzir a pressão do banco de dados. Se visitantes mal-intencionados acessam frequentemente dados que não existem no cache, o cache perde o significado de existir. Instantaneamente, a pressão de todas as solicitações recai sobre o banco de dados. Isso faz com que o banco de dados sofra uma pressão enorme e pode até causar problemas de conexão. Isso é semelhante a um ataque distribuído de negação de serviço, ou DDoS. Uma maneira de resolver a penetração do cache é concordar que se uma consulta retornar um valor vazio, esse valor vazio também será armazenado em cache, mas um tempo limite curto deve ser definido para o cache do valor vazio, porque armazenar esse valor em cache ainda é um desperdício de espaço de cache. Outra maneira de resolver a penetração do cache é usar um filtro Bloom. Todos podem aprender de maneira específica sozinhos.

#### Divisão de cache

Em um projeto real, é possível que alguma chave de cache expire em um determinado momento, e justamente nesse momento chegue um grande número de solicitações simultâneas para essa chave. Essas solicitações não encontram os dados dessa chave no cache, portanto, obtêm os dados diretamente do banco de dados e os gravam de volta no cache. Neste momento, essas solicitações altamente simultâneas podem destruir instantaneamente o banco de dados. Este fenômeno é chamado de quebra de cache. Uma maneira mais comum de resolver a falha do cache é usar um bloqueio mutex. Simplificando, quando o cache se tornar inválido, não carregue imediatamente os dados do banco de dados. Primeiro defina um bloqueio mutex, como `setnx` no Redis. Somente a solicitação que definir com êxito o bloqueio mutex poderá executar a consulta, carregar dados do banco de dados e gravá-los no cache. Outras solicitações que não conseguem definir o bloqueio mutex podem primeiro dormir por um curto período e depois tentar novamente obter dados do cache. Se o cache ainda não tiver dados, repita a operação de bloqueio mutex mencionada anteriormente. O código de referência aproximado é mostrado abaixo.

```python
data = redis_cli.get(key)
while not data:
    if redis_cli.setnx('mutex', 'x'):
        redis.expire('mutex', timeout)
        data = db.query(...)
        redis.set(key, data)
        redis.delete('mutex')
    else:
        time.sleep(0.1)
        data = redis_cli.get(key)
```

#### Esconderijo de Avalanche

Avalanche de cache significa que quando os dados são colocados no cache, o mesmo tempo de expiração é usado, então o cache se torna inválido ao mesmo tempo em algum momento, todas as solicitações são encaminhadas para o banco de dados e o banco de dados recebe muita pressão instantaneamente e trava. A maneira de resolver a avalanche de cache também é bastante simples. Podemos adicionar um tempo aleatório além do tempo fixo de expiração do cache. Dessa forma, até certo ponto, podemos evitar que chaves diferentes expirem juntas ao mesmo tempo. Outra maneira é usar o cache multinível. Cada nível de cache possui um tempo de expiração diferente. Nesse caso, mesmo que todo o cache de um nível expire junto, o cache de outros níveis ainda pode fornecer dados, de modo que todas as solicitações não caiam no banco de dados.
