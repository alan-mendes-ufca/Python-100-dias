## MySQL Performance Optimization

### Basic Principles

If you want MySQL to give its best performance, there are three basic principles that should be followed.

1. Let MySQL return to its basic duty of storage. The database should only be used for data storage. It should not do complex computation or carry business logic. Storage and computation should be kept separate.
2. When querying data, try to query a single table whenever possible, and reduce cross-database queries and excessive joins.
3. Avoid a whole series of common performance killers, such as large transactions, giant SQL statements, huge batch operations, and large fields.

These performance killers deserve a little more explanation:

- **Large transactions** involve many steps and many tables or fields. They easily cause resource contention and even deadlocks. If a rollback happens, resources may stay occupied for too long.
- **Large SQL statements** often mean too many table joins. Once a query joins more than three tables, MySQL usually consumes much more resource and the performance drops a lot.
- **Large batches** can reduce the overhead of executing SQL one statement at a time, but they must be tested carefully and should be run during off-peak hours.
- **Large fields** such as `blob` and `text` should be used as little as possible. If they are necessary, it is better to separate them from the main business table and avoid frequent retrieval and update on them.

### Database and Table Design

When designing databases and tables, the lesson emphasizes the following points:

1. Use InnoDB as the default storage engine and disable MyISAM.
2. Use `UTF8mb4` as the default character set.
3. Turn off case-sensitive table-name behavior by setting `lower_case_table_names=1` when needed.
4. Avoid relying on stored procedures, triggers, views, and events to carry business logic in production systems.
5. Keep the number of tables in one database instance under control.

For InnoDB tables, the lesson gives several concrete suggestions:

1. Use an unsigned integer primary key, usually with `auto_increment`, and do not update the primary key manually.
2. Add a `comment` to every column.
3. Explicitly specify the `engine` when creating tables.
4. Make sure every table has core fields such as the primary key, creation time, and modification time.
5. Prefer `not null` fields with reasonable defaults, because `null` is harder for MySQL to optimize.
6. Keep the number of columns in a single table under control. If there are too many fields, consider vertical splitting.
7. Avoid `enum` and `set`.
8. Do not store large files directly as `blob` if saving a path is enough.
9. Use `decimal` instead of floating-point types for money and other precision-sensitive values.
10. Store IP addresses as `int unsigned` rather than `char(15)`.

### Using Indexes

Index design is one of the most important parts of MySQL optimization. The lesson reviews several basic principles:

1. The best index columns are not necessarily the columns returned in `select`; they are usually the columns used in `where` conditions or join conditions.
2. Columns with high uniqueness usually make better indexes than columns with many repeated values.
3. For string indexes, it is often better to create a prefix index with a short length.
4. A composite index should be designed with the leftmost-prefix rule in mind.
5. Do not overuse indexes. They take storage space and they also hurt write performance.
6. Be careful about situations that make indexes invalid, such as leading wildcards in `like` queries or certain negative conditions.

### Procedures and Partitioning

Stored procedures can simplify the work of application developers and reduce communication between the application and the database server, so they can help improve performance in data operations. But the broader advice of the lesson is still to avoid pushing too much business logic into the database.

MySQL also supports partitioning. Partitioning can help store more data, improve some query patterns, increase throughput, and remove expired data faster. The lesson introduces several partition types:

1. **RANGE partitioning**

   ```sql
   CREATE TABLE tb_emp (
       eno INT NOT NULL,
       ename VARCHAR(20) NOT NULL,
       job VARCHAR(10) NOT NULL,
       hiredate DATE NOT NULL,
       dno INT NOT NULL
   )
   PARTITION BY RANGE(YEAR(hiredate)) (
       PARTITION p0 VALUES LESS THAN (1960),
       PARTITION p1 VALUES LESS THAN (1970),
       PARTITION p2 VALUES LESS THAN (1980),
       PARTITION p3 VALUES LESS THAN (1990),
       PARTITION p4 VALUES LESS THAN MAXVALUE
   );
   ```

### SQL Optimization

The lesson gives several concrete ways to optimize SQL.

#### 1. Find low-efficiency SQL with the slow-query log

You can inspect the current configuration:

```sql
show variables like 'slow_query%';
show variables like 'long_query_time';
```

Example output:

```sql
+---------------------------+----------------------------------+
| Variable_name             | Value                            |
+---------------------------+----------------------------------+
| slow_query_log            | OFF                              |
| slow_query_log_file       | /mysql/data/localhost-slow.log   |
+---------------------------+----------------------------------+

+-----------------+-----------+
| Variable_name   | Value     |
+-----------------+-----------+
| long_query_time | 10.000000 |
+-----------------+-----------+
```

You can create and configure the slow-query log:

```bash
touch /var/log/mysqld-slow.log
chown mysql /var/log/mysqld-slow.log
```

```sql
set global slow_query_log_file='/var/log/mysqld-slow.log';
set global slow_query_log='ON';
set global long_query_time=1;
```

or enable it in the MySQL config file directly.

#### 2. Use `EXPLAIN` to inspect execution plans

For example:

```sql
explain select ename, job, sal from tb_emp where dno=20\G
```

Example output:

```sql
*************************** 1. row ***************************
           id: 1
  select_type: SIMPLE
        table: tb_emp
         type: ref
possible_keys: fk_emp_dno
          key: fk_emp_dno
      key_len: 5
          ref: const
         rows: 7
        Extra: NULL
1 row in set (0.00 sec)
```

The lesson especially points out fields such as:

- `select_type`
- `table`
- `type`
- `possible_keys`
- `key`
- `key_len`
- `rows`
- `extra`

These help you understand whether the query is doing a full table scan, whether indexes are being used, and how expensive the query is likely to be.

#### 3. Use profiling tools

You can inspect whether profiling is enabled:

```sql
select @@have_profiling;
select @@profiling;
set profiling=1;
```

Then you can run SQL and inspect the profile:

```sql
show profiles;
show profile for query 1;
```

Example output:

#### 4. Optimize CRUD operations

The lesson mentions several practical ideas:

- For `insert`, inserting multiple rows in one statement is usually better than sending many small inserts.
- If loading data from a text file, `load data infile` is often much faster than repeated inserts.
- `order by` can be optimized when the sorting condition matches an index.
- For `group by`, using `order by null` can avoid unnecessary sorting overhead.
- Some subqueries can be replaced with joins for better performance.
- `or` conditions only benefit from indexes when all relevant conditions are indexed.
- Pagination queries such as `limit 10000, 20` are expensive. A more efficient pattern is to page on an indexed subquery and then join back to the main table.

For example:

```sql
select * from tb_emp order by ename limit 10000, 20;
select * from tb_emp t1
inner join (
    select eno from tb_emp order by ename limit 10000, 20
) t2 on t1.eno = t2.eno;
```

The second form is usually better if `ename` already has an index.

### Summary

MySQL performance optimization is not one single trick. It is a combination of basic design discipline, careful table structure, reasonable indexing, partitioning where necessary, and disciplined SQL analysis with tools such as the slow-query log, `EXPLAIN`, and profiling. The lesson’s main point is simple: good performance comes from using the database in the right way, not from trying to rescue a bad design later.
