## An Overview of Web Data Collection

Crawlers, also often called web spiders, are robot programs, or automated script code, that automatically browse websites and get needed information according to certain rules. They are widely used in Internet search engines and data collection. Anyone who has used the Internet and a browser knows that, besides text information for users to read, a web page also contains some hyperlinks. A web crawler keeps getting the addresses of other pages on the Internet through the hyperlink information in web pages, and then keeps collecting data. Because of this, the process of collecting web data is like a crawler or spider wandering on the network, so it is vividly called a crawler or web spider.

### Application Areas of Crawlers

In an ideal situation, all ICPs, meaning Internet Content Providers, should provide API interfaces for their own websites to share the data that they allow other programs to get. In this case, crawler programs would not be needed at all. Well-known e-commerce platforms, such as Taobao and JD, and social platforms, such as Weibo and WeChat, all provide their own API interfaces, but these API interfaces usually limit both what data can be fetched and how often it can be fetched. For most companies, getting industry data and competitor data in time is one of the important parts of company survival. But for most companies, data is naturally a weak point. In this case, making proper use of crawlers to get data and pull out business value from it becomes very important.

The application areas of crawlers are actually very wide. Below we list part of them, and interested readers can explore the related content by themselves.

1. Search engines
2. News aggregation
3. Social applications
4. Public-opinion monitoring
5. Industry data

### Discussion of Crawler Legality

People often say, "If you write crawlers well, you may end up eating prison food well." So is writing crawler programs illegal? We can look at this question from the following angles.

1. The field of web crawlers is still in a pioneering stage. Although the Internet world has established some moral rules through its own rules of the game, namely the Robots protocol, the legal part is still being established and improved. That means this field is still a gray area for now.
2. If the crawler gets frontend data, meaning public information shown on web pages, rather than private and sensitive information from the website backend, there is less need to worry about legal restrictions, because at present the speed of development of the big-data industry chain is far greater than the speed of law improvement.
3. When crawling a website, the crawler needs to obey the Robots protocol and control the speed of data fetching. When using the data, the website's intellectual property rights must be respected. Since the Web 2.0 era, although much data on the Web is provided by users, website platforms have still paid operating costs. When users register and publish content, the platform usually has already got ownership, usage rights, and distribution rights over the data. If these rules are broken, the chance of losing in court is quite high.
4. Properly hiding your identity is necessary when writing crawler programs, and it is better not to let the other side prove that your crawler damaged other people's movable property, such as servers.
5. Do not open source or publicly show your crawler code on the public Internet, such as code hosting platforms. These actions usually bring unnecessary trouble to yourself.

#### The Robots Protocol

Most websites define a `robots.txt` file. This is a gentleman's agreement, and not a rule that all crawlers must obey. Below, taking Taobao's [`robots.txt`](http://www.taobao.com/robots.txt) file as an example, let us see what restrictions Taobao has for crawlers.

```text
User-agent: Baiduspider
Disallow: /

User-agent: baiduspider
Disallow: /
```

From the file above, we can see that Taobao forbids Baidu's crawler from crawling any of its resources. So when you search "Taobao" on Baidu, the text below the search result says that because the website's `robots.txt` file has restriction directives, the system cannot provide a content description for that page. As a search engine, Baidu at least on the surface obeyed Taobao's `robots.txt` protocol, so users cannot search Taobao's internal product information from Baidu.

### Hypertext Transfer Protocol, HTTP

Before starting the crawler discussion, let us review the Hypertext Transfer Protocol, HTTP, a little, because the content we see on web pages is usually the result of the browser running HTML, meaning HyperText Markup Language, and HTTP is the protocol that transfers HTML data. HTTP, like many other application-layer protocols, is built on top of TCP, meaning the Transmission Control Protocol. It uses the reliable transport service provided by TCP to implement data exchange in Web applications. According to the introduction on Wikipedia, the original purpose of designing HTTP was to provide a way to publish and receive HTML pages. That means this protocol is the carrier of the data transferred between browsers and Web servers. For detailed information about HTTP and its current development, everyone can read some introductory articles and references by themselves.

The picture below is the HTTP request and response messages that I captured when visiting the Baidu home page with Ethereal, the predecessor of WireShark, during my work at the Key Laboratory of Network Communication Technology of Sichuan Province. Because Ethereal captures the data that has passed through the network adapter, the protocol data from the physical link layer to the application layer can be seen clearly.

Figure 2. HTTP request

<img src="res/20210824003915.png" style="zoom:85%;">

An HTTP request is usually made up of four parts: request line, request headers, blank line, and message body. If there is no data to send to the server, the message body is not a required part. The request line contains the request method, such as GET or POST, the resource path, and the protocol version. The request headers are made up of several key-value pairs and contain information such as browser, encoding method, preferred language, and cache policy. After the request headers come the blank line and the message body.

An HTTP response is usually made up of four parts: response line, response headers, blank line, and message body. The message body is the data returned by the service. It may be an HTML page, JSON, binary data, and so on. The response line contains the protocol version and the response status code. #### Related Tools

Next, let us first introduce some helper tools for developing crawler programs. These tools will surely help you get twice the result with half the effort.

1. Chrome Developer Tools, the built-in developer tool in Google Chrome. The most commonly used modules of this tool are:
   Elements, used to view or modify HTML element properties, CSS properties, event listeners, and so on. CSS can be modified and shown immediately, which makes page debugging much easier.
   Console, used to run one-time code, view JavaScript objects, and view debug log or exception information. The console is actually an interactive environment for running JavaScript code.
   Sources, used to view the source code of the page's HTML, JavaScript, and CSS files. More importantly, it can debug JavaScript source code by adding breakpoints and stepping through the code.
   Network, used for HTTP requests, HTTP responses, and information related to network connections.
   Application, used to view browser local storage, background tasks, and other content. Local storage mainly includes Cookie, Local Storage, Session Storage, and so on.

### Basic Workflow of a Crawler

A basic crawler usually includes three parts: data collection, meaning page download, data processing, meaning page parsing, and data storage, meaning persisting useful information. Of course, more advanced crawlers use concurrent programming or distributed technology during data collection and processing, so they need the participation of a scheduler, which arranges threads or processes to run corresponding tasks, and backend management programs, which monitor the crawler's working state and check the data collection results.

<img src="res/20210824004107.png" style="zoom:62%;">

Generally speaking, the workflow of a crawler includes the following steps:

1. Set the crawling target, meaning the seed page or starting page, and get the page.
2. When the server cannot be accessed, try to download the page again according to the specified retry count.
3. When needed, set the user agent or hide the real IP, otherwise the page may not be accessible.
4. Do the necessary decoding work on the page that was got, and then grab the needed information.
5. In the page that was got, extract link information from the page in some way, such as a regular expression.
6. Process the links further, get the pages, and repeat the actions above.
7. Persist the useful information for later processing.
