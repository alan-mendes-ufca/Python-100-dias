## Teste de software e testes automatizados

### Visão geral dos testes de software

O teste de software é um processo usado para ajudar a identificar a correção, integridade, segurança e qualidade do software. Em outras palavras, é o processo de execução de um programa sob condições específicas para encontrar erros no programa, medir a qualidade do software e julgar se ele pode atender aos requisitos de projeto.

#### Métodos de teste

Teste de caixa preta: teste as funções de um aplicativo, não sua estrutura interna ou operação. O testador não precisa de conhecimento especial do código, estrutura interna ou linguagem de programação do aplicativo. O testador só precisa saber o que o sistema deve fazer. Ou seja, quando uma entrada específica é inserida, uma determinada saída deve ser produzida. Os casos de teste são projetados de acordo com as funções que o sistema aplicativo deve fornecer, com base em padrões, especificações ou requisitos. O testador escolhe entradas válidas e entradas inválidas para verificar se a saída está correta. Este método de teste é adequado para a maioria dos tipos de testes de software, como testes de integração e testes de sistema.

Teste de caixa branca: teste a estrutura interna ou operação de um aplicativo, não as funções do aplicativo. Nos testes de caixa branca, os casos de teste são projetados do ponto de vista da linguagem de programação. O testador insere dados para verificar o caminho do fluxo de dados dentro do programa e determina se a saída é adequada, o que é semelhante a testar nós em um circuito.

Devido aos limites de tempo e custo, uma das questões mais críticas em testes de software é: "**Entre todos os casos de teste possíveis, qual subconjunto pode encontrar mais erros?**" Portanto, quando os casos de teste são projetados, o teste de caixa branca presta atenção ao grau de cobertura lógica do programa, como cobertura de instruções, cobertura de condições e cobertura de ramificação. O teste de caixa preta pode usar métodos como particionamento de equivalência, análise de valor limite, análise gráfica de causa-efeito e adivinhação de erros para projetar casos de teste.

#### Tipos de teste (etapas)

Teste unitário: testar unidades de software. Seu objetivo é verificar a exatidão das unidades básicas de software. O objeto testado é a menor unidade no design de software, que geralmente é uma função.

Teste de integração: monte módulos de programa com uma estratégia de integração adequada e, em seguida, teste se as interfaces do sistema e as funções integradas estão corretas. Seu principal objetivo é verificar se as interfaces entre as unidades de software estão corretas. Os objetos testados no teste de integração são módulos que já passaram no teste unitário.

Teste de sistema: o teste de sistema inclui principalmente testes funcionais, testes de interface, testes de confiabilidade, testes de usabilidade e testes de desempenho.

Teste de regressão: atividades de teste realizadas para detectar erros introduzidos por alterações de código. O teste de regressão é uma tarefa importante durante o estágio de manutenção de software. Alguns estudos mostram que o custo gerado pelos testes de regressão pode representar mais de um terço do custo total do ciclo de vida do software.

#### Desenvolvimento Orientado a Testes (Testes Ágeis)

O desenvolvimento orientado a testes inclui as três etapas a seguir:

1. Escreva testes automatizados para novos recursos ou melhorias que ainda não foram implementadas.
2. Forneça a quantidade mínima de código que pode passar em todos os testes definidos.
3. Refatore o código para que atenda ao padrão de qualidade exigido.

O benefício do desenvolvimento orientado a testes é que ele pode prevenir efetivamente a regressão do software e produzir código de melhor qualidade. Além disso, o teste de aceitação deve ser feito pelo cliente. O cliente projeta testes de aceitação de acordo com cenários de uso e confirma objetivamente se a aplicação pode atender aos seus requisitos. Um recurso que pode passar em testes de unidade, ou mesmo em testes de sistema, ainda pode falhar no teste de aceitação do cliente.

#### Testando aplicativos de Internet e aplicativos móveis

Estratégias de teste para aplicações de internet:

1. Teste da camada de apresentação: teste de conteúdo, teste de estrutura do site e teste de ambiente do usuário, como navegadores e sistemas operacionais.
2. Testes da camada de negócios: desempenho, validação de dados, transações e serviços externos.
3. Testes de camada de persistência: tempo de resposta, integridade de dados e tolerância a falhas.

Estratégias de teste para aplicativos móveis:

1. Testes em dispositivos reais.
2. Teste baseado em emulador.

### Teste de Unidade (Módulo)

A biblioteca padrão do Python possui o módulo `unittest` para escrever testes unitários. Ao executar testes, é recomendado usar [pytest](https://docs.pytest.org/en/latest/) ou `nose2`. `pytest` é um executor de testes que pode pesquisar e executar testes automaticamente e também gera relatórios de erros detalhados.

Você pode instalar a biblioteca [luminárias de teste](https://pypi.org/project/testfixtures/) para ajudar nos testes de unidade. Ele combina muitos acessórios típicos e fornece módulos de função para criar diretórios, alterar a data do sistema e criar objetos simulados. Esses módulos podem nos ajudar a separar os testes unitários dos ambientes dos quais eles dependem. [zombar](https://pypi.org/project/mock/) é uma biblioteca que substitui os objetos dos quais o alvo de teste depende por objetos virtuais. Durante o teste, podemos especificar o valor de retorno de um objeto virtual quando ele é chamado e se uma exceção deve acontecer.

`tox` pode preparar convenientemente o ambiente necessário para executar testes. `tox` cria ambientes de teste em vários ambientes `virtualenv`, depois executa os testes nesses ambientes e mostra os resultados. Ele pode unificar as opções e variáveis de ambiente das ferramentas de teste, portanto, só precisamos executar o comando `tox` para concluir facilmente os testes necessários.

##### Rede - [Selenium](https://www.selenium.dev/documentation/)

Selenium é um grupo de ferramentas de teste orientadas por navegador usadas para automatizar testes funcionais e testes de integração de aplicativos da web. Como um usuário que usa um navegador, o Selenium pode realizar operações do mouse no navegador, digitar texto em formulários, verificar valores de formulários e assim por diante. Com esta capacidade, as operações manuais podem ser transformadas em operações automatizadas.

1. Vantagens do Selenium

    - O Selenium IDE pode gravar e reproduzir interações no navegador, o que é útil para explorar fluxos. Para uma suíte mantida no projeto, testes com WebDriver versionados junto do código são mais revisáveis e reutilizáveis.
   - Suporta muitos navegadores e sistemas operacionais.

2. Componentes do Selenium

    - [Selenium IDE](https://www.selenium.dev/selenium-ide/)
    - [Selenium WebDriver](https://www.selenium.dev/documentation/webdriver/)
    - [Selenium Grid](https://www.selenium.dev/documentation/grid/) para executar testes em paralelo em diferentes navegadores ou plataformas

3. Trabalhando em conjunto com ferramentas de integração contínua

Integração contínua significa integrar código na ramificação principal com frequência. Tem dois benefícios principais:

   - Erros podem ser encontrados rapidamente. Cada vez que uma pequena atualização é concluída, ela é integrada ao branch principal. Isso torna mais fácil encontrar erros rapidamente e também mais fácil localizá-los.
   - Evita que os galhos se afastem muito do galho principal. Se o código não for integrado com frequência, enquanto a ramificação principal continua mudando, a integração posterior se torna mais difícil e pode até falhar.

O objetivo da integração contínua é permitir que o produto seja iterado rapidamente, mantendo a alta qualidade. Sua principal medida é que antes que o código seja integrado à ramificação principal, ele deve passar por testes automatizados. Enquanto um caso de teste falhar, ele não poderá ser integrado.

Em integração contínua, mantenha os testes WebDriver no repositório e execute-os com o executor de testes do projeto, como `pytest`; não são necessários formatos de suíte HTML ou plugins específicos do Jenkins. O agente de CI deve disponibilizar o navegador, instalar as dependências e executar, por exemplo:

```bash
python -m pytest --junitxml=reports/junit.xml
```

Publique o relatório JUnit como artefato do trabalho. Quando for preciso testar navegadores ou plataformas em paralelo, conecte a suíte WebDriver a um Selenium Grid ou a um provedor compatível com WebDriver.

Além do Selenium, [WebTest](https://pypi.org/project/WebTest/), [Lasca](https://splinter.readthedocs.io/en/latest/) e [RobotFramework](https://robotframework.org/) também são opções para testes no lado da web. Entre eles, o WebTest pode enviar solicitações simuladas para aplicativos WSGI e obter os resultados, de modo que quase todos os aplicativos WSGI podem ser testados com ele. Splinter é um wrapper de segundo nível em torno do Selenium, por isso é mais conveniente e simples de usar.
