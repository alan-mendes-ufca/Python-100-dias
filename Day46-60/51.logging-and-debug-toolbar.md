## Log e barra de ferramentas de depuração

### Configurando o registro em log

Durante o desenvolvimento do projeto, é muito necessário mostrar informações de depuração suficientes para ajudar os desenvolvedores a depurar o código. Depois que o projeto fica online, também é muito necessário registrar avisos, erros e outras informações que aparecem durante a execução do sistema, para que pessoas relacionadas possam entender as condições de execução do sistema e manter o código. Ao mesmo tempo, a coleta de dados de registro também estabelece as bases para a operação digital de um site. Ao analisar os logs do sistema, podemos monitorar o tráfego do site e a distribuição do tráfego, e também podemos descobrir os hábitos de uso e padrões de comportamento dos usuários.

A seguir, vamos primeiro ver como configurar logs através do arquivo de configuração do Django. A configuração do log do Django pode basicamente seguir a documentação oficial e depois ser combinada com as necessidades reais do projeto. A maior parte desses conteúdos pode ser copiada da documentação oficial e depois ajustada um pouco. Algumas configurações de referência são fornecidas abaixo.

```python
LOGGING = {
    'version': 1,
    'disable_existing_loggers': False,
    'formatters': {
        'simple': {
            'format': '%(asctime)s %(module)s.%(funcName)s: %(message)s',
            'datefmt': '%Y-%m-%d %H:%M:%S',
        },
        'verbose': {
            'format': '%(asctime)s %(levelname)s [%(process)d-%(threadName)s] '
                      '%(module)s.%(funcName)s line %(lineno)d: %(message)s',
            'datefmt': '%Y-%m-%d %H:%M:%S',
        }
    },
    'filters': {
        'require_debug_true': {
            '()': 'django.utils.log.RequireDebugTrue',
        },
    },
    'handlers': {
        'console': {
            'class': 'logging.StreamHandler',
            'level': 'DEBUG',
            'filters': ['require_debug_true'],
            'formatter': 'simple',
        },
        'file1': {
            'class': 'logging.handlers.TimedRotatingFileHandler',
            'filename': 'access.log',
            'when': 'W0',
            'backupCount': 12,
            'formatter': 'simple',
            'level': 'INFO',
        },
        'file2': {
            'class': 'logging.handlers.TimedRotatingFileHandler',
            'filename': 'error.log',
            'when': 'D',
            'backupCount': 31,
            'formatter': 'verbose',
            'level': 'WARNING',
        },
    },
    'loggers': {
        'django': {
            'handlers': ['console', 'file1', 'file2'],
            'propagate': True,
            'level': 'DEBUG',
        },
    }
}
```

Você já deve ter notado que os `formatters` na configuração de log acima são **formatadores de log**. Eles significam como a saída do log é formatada. Os espaços reservados de formato significam:

1. `%(name)s` - o nome do registrador
2. `%(levelno)s` - a forma numérica do nível de log
3. `%(levelname)s` - o nome de texto do nível de log
4. `%(filename)s` - o nome do arquivo de origem que fez a chamada de registro
5. `%(pathname)s` - o nome do caminho do arquivo de origem que fez a chamada de registro
6. `%(funcName)s` – o nome da função que fez a chamada de registro
7. `%(module)s` - o nome do módulo que fez a chamada de registro
8. `%(lineno)s` - o número da linha que fez a chamada de registro
9. `%(created)s` - a hora em que o registro de log foi criado
10. `%(asctime)s` - data e hora
11. `%(msecs)s` - parte de milissegundos
12. `%(thread)d` - ID do encadeamento (inteiro)
13. `%(threadName)s` - nome do encadeamento
14. `%(process)d` - ID do processo (inteiro)

O `handlers` na configuração de log é usado para especificar **manipuladores de log**. Simplificando, eles especificam se os logs serão enviados para o console, para um arquivo ou para um servidor na rede. Os manipuladores disponíveis incluem:

1. `logging.StreamHandler(stream=None)` - pode enviar informações para qualquer objeto semelhante a arquivo, como `sys.stdout` ou `sys.stderr`
2. `logging.FileHandler(filename, mode='a', encoding=None, delay=False)` - grava mensagens de log em um arquivo
3. `logging.handlers.DatagramHandler(host, port)` - envia mensagens de log para um host de rede no host e porta especificados usando UDP
4. `logging.handlers.HTTPHandler(host, url)` - carrega mensagens de log para um servidor HTTP usando GET ou POST
5. `logging.handlers.RotatingFileHandler(filename, mode='a', maxBytes=0, backupCount=0, encoding=None, delay=False)` - grava mensagens de log em um arquivo e, se o tamanho do arquivo ultrapassar `maxBytes`, cria outro arquivo para registrar logs
6. `logging.handlers.SocketHandler(host, port)` - envia mensagens de log para um host de rede no host e porta especificados usando TCP
7. `logging.handlers.SMTPHandler(mailhost, fromaddr, toaddrs, subject, credentials=None, secure=None, timeout=1.0)` - envia logs para um endereço de e-mail especificado
8. `logging.MemoryHandler(capacity, flushLevel=ERROR, target=None, flushOnClose=True)` - gera logs para o buffer especificado na memória

Cada manipulador de log acima especifica uma propriedade chamada `level`. Significa o nível de log. Diferentes níveis de log mostram a seriedade das informações registradas no log. Python define seis níveis de log, e de baixo para alto são: `NOTSET`, `DEBUG`, `INFO`, `WARNING`, `ERROR` e `CRITICAL`.

Os **loggers** configurados no final são usados ​​para realmente gerar logs. Django fornece os registradores integrados mostrados abaixo:

1. `django` - todos os registradores de mensagens na hierarquia do Django
2. `django.request` - mensagens de log relacionadas ao tratamento de solicitações; As respostas 5xx são consideradas mensagens de erro e as respostas 4xx são consideradas mensagens de aviso
3. `django.server` - mensagens de log relacionadas às solicitações recebidas pelo servidor iniciado através de `runserver`; As respostas 5xx são consideradas erros, as respostas 4xx são registradas como avisos e todo o resto é registrado como `INFO`
4. `django.template` - mensagens de log relacionadas à renderização de modelo
5. `django.db.backends` - mensagens de log geradas pela interação com o banco de dados; se você quiser mostrar as instruções SQL executadas pela estrutura ORM, você pode usar este logger

O nível de log configurado no criador de logs pode não ser o nível de log final, porque também precisamos observar o nível de log configurado no manipulador. O nível mais alto dos dois é considerado o nível de log final.

### Barra de ferramentas de depuração do Django

Se você deseja depurar seu projeto Django, você realmente não deve perder a ferramenta chamada Django-Debug-Toolbar. É uma ferramenta indispensável para ajudar na depuração e otimização durante o desenvolvimento do projeto. Contanto que esteja configurado, você poderá ver convenientemente as informações de execução do projeto mostradas na tabela abaixo. Esta informação é muito importante para depurar o projeto e melhorar o desempenho da aplicação Web.

| Item | Descrição |
| ----------- | --------------------------------------- |
| Versões | Versão Django |
| Tempo | Tempo gasto pela visualização |
| Configurações | Valores definidos no arquivo de configuração |
| Cabeçalhos | Informações nos cabeçalhos de solicitação e resposta |
| Solicitar | Todos os tipos de variáveis ​​relacionadas à solicitação |
| Arquivos estáticos | Situação de carregamento de arquivo estático |
| Modelos | Informações relacionadas ao modelo |
| Cache | Usar cache |
| Sinais | Informações de sinal do Django integradas |
| Registro | Informações registradas |
| SQL | SQL enviado ao banco de dados e tempo de execução |

1. Instale Django-Debug-Toolbar.

```bash
pip install django-debug-toolbar
```

2. Configure-o alterando `settings.py`.

```python
INSTALLED_APPS = [
    'debug_toolbar',
]

MIDDLEWARE = [
    'debug_toolbar.middleware.DebugToolbarMiddleware',
]

DEBUG_TOOLBAR_CONFIG = {
    'JQUERY_URL': 'https://cdn.bootcss.com/jquery/3.3.1/jquery.min.js',
    'SHOW_COLLAPSED': True,
    'SHOW_TOOLBAR_CALLBACK': lambda x: True,
}
```

3. Configure-o alterando `urls.py`.

```python
if settings.DEBUG:
    import debug_toolbar
    urlpatterns.insert(0, path('__debug__/', include(debug_toolbar.urls)))
```

4. Após a configuração do Django-Debug-Toolbar, você verá uma barra de ferramentas de depuração no lado direito da página, conforme mostrado na imagem abaixo. Inclui as informações de depuração mencionadas acima, como tempo de execução, configurações do projeto, cabeçalhos de solicitação, SQL, recursos estáticos, modelos, cache e sinais, por isso é muito conveniente de visualizar.

### Otimização de ORM

Após a configuração do log ou do Django-Debug-Toolbar, podemos verificar a execução da função de visualização que exportou dados do professor para um relatório Excel anteriormente. O que nos importa aqui é a aparência real da consulta SQL gerada pela estrutura ORM. Acredito que o resultado aqui pode surpreendê-lo um pouco. Após executar `Teacher.objects.all()`, podemos notar que o SQL visto no console ou mostrado pelo Django-Debug-Toolbar fica assim:

```sql
SELECT `tb_teacher`.`no`, `tb_teacher`.`name`, `tb_teacher`.`detail`, `tb_teacher`.`photo`, `tb_teacher`.`good_count`, `tb_teacher`.`bad_count`, `tb_teacher`.`sno` FROM `tb_teacher`; args=()
SELECT `tb_subject`.`no`, `tb_subject`.`name`, `tb_subject`.`intro`, `tb_subject`.`create_date`, `tb_subject`.`is_hot` FROM `tb_subject` WHERE `tb_subject`.`no` = 101; args=(101,)
SELECT `tb_subject`.`no`, `tb_subject`.`name`, `tb_subject`.`intro`, `tb_subject`.`create_date`, `tb_subject`.`is_hot` FROM `tb_subject` WHERE `tb_subject`.`no` = 101; args=(101,)
SELECT `tb_subject`.`no`, `tb_subject`.`name`, `tb_subject`.`intro`, `tb_subject`.`create_date`, `tb_subject`.`is_hot` FROM `tb_subject` WHERE `tb_subject`.`no` = 101; args=(101,)
SELECT `tb_subject`.`no`, `tb_subject`.`name`, `tb_subject`.`intro`, `tb_subject`.`create_date`, `tb_subject`.`is_hot` FROM `tb_subject` WHERE `tb_subject`.`no` = 101; args=(101,)
SELECT `tb_subject`.`no`, `tb_subject`.`name`, `tb_subject`.`intro`, `tb_subject`.`create_date`, `tb_subject`.`is_hot` FROM `tb_subject` WHERE `tb_subject`.`no` = 103; args=(103,)
SELECT `tb_subject`.`no`, `tb_subject`.`name`, `tb_subject`.`intro`, `tb_subject`.`create_date`, `tb_subject`.`is_hot` FROM `tb_subject` WHERE `tb_subject`.`no` = 103; args=(103,)
```

Esse problema geralmente é chamado de problema de "consulta 1+N". Em alguns lugares, também é chamado de problema de “consulta N+1”. Originalmente, obter dados de professores precisava apenas de uma instrução SQL, mas como os professores estão relacionados a disciplinas, quando consultamos linhas `N` de dados de professores, a estrutura ORM do Django envia `N` mais instruções SQL ao banco de dados para consultar as informações da disciplina desses professores. Cada execução SQL tem um custo relativamente grande e também pressiona o servidor de banco de dados. Se pudermos terminar a consulta do professor e da disciplina em uma instrução SQL, essa será certamente a melhor maneira. Isso também é fácil de fazer e você provavelmente já pensou em como fazê-lo. Sim, podemos usar uma consulta join, mas como fazemos isso ao usar o framework ORM do Django? Para relacionamentos muitos-para-um, como professores e disciplinas no aplicativo de votação, podemos usar o método `select_related()` de `QuerySet` para carregar objetos relacionados. Para relacionamentos muitos-para-muitos, como pedidos e produtos em um site de comércio eletrônico, podemos usar o método `prefetch_related()` para carregar objetos relacionados.

Na função de visualização que exporta o relatório Excel do professor, podemos otimizar o código assim.

```python
queryset = Teacher.objects.all().select_related('subject')
```

Na verdade, a operação de consulta de dados de votos positivos e negativos do professor na função de visualização que usa ECharts para gerar o relatório de front-end também pode ser otimizada. Neste exemplo, precisamos apenas de três dados: o nome do professor, contagem positiva de votos e contagem negativa de votos. Mas por padrão, o SQL gerado consulta todos os campos da tabela professor. Podemos usar o método `only()` de `QuerySet` para especificar os atributos que precisamos consultar e também podemos usar o método `defer()` de `QuerySet` para especificar os atributos que não precisamos consultar no momento. Dessa forma, o SQL gerado usa projeção para especificar as colunas necessárias, melhorando o desempenho da consulta. O código é mostrado abaixo.

```python
queryset = Teacher.objects.all().only('name', 'good_count', 'bad_count')
```

Claro, se quisermos contar a contagem média de votos positivos e negativos dos professores em cada disciplina, também podemos fazer isso com a estrutura ORM do Django. O código é mostrado abaixo.

```python
queryset = Teacher.objects.values('subject').annotate(good=Avg('good_count'), bad=Avg('bad_count'))
```

Os elementos do `QuerySet` obtidos aqui são objetos de dicionário. Cada dicionário possui três grupos de pares de valores-chave, que são `subject` para o ID do assunto, `good` para a contagem de votos positivos e `bad` para a contagem de votos negativos. Se quiser obter o nome do assunto em vez do ID, você pode alterar o código assim.

```python
queryset = Teacher.objects.values('subject__name').annotate(good=Avg('good_count'), bad=Avg('bad_count'))
```

Portanto, podemos ver que o framework ORM do Django nos permite completar o agrupamento e agregar consultas em um banco de dados relacional de forma orientada a objetos.
