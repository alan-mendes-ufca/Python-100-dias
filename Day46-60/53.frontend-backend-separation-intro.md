## Separação Frontend-Backend

No desenvolvimento tradicional de aplicações Web, a maioria dos programadores trata o navegador como a linha divisória entre frontend e backend. A parte do navegador que mostra as páginas aos usuários é chamada de frontend, e todo o código executado no servidor que fornece lógica de negócios e prepara dados para o frontend é chamado de backend. A separação front-end-backend significa que os engenheiros front-end e back-end concordam com as APIs de interação de dados e, em seguida, fazem o desenvolvimento e os testes em paralelo. O back-end apenas fornece dados e não os renderiza nas páginas. O frontend obtém dados por meio de solicitações HTTP e é responsável por renderizá-los nas páginas. Este trabalho é feito por código JavaScript no navegador.

Usar a separação front-end-backend traz muitos benefícios. Vamos falar brevemente sobre esses benefícios abaixo:

1. **Melhore a eficiência do desenvolvimento**. Depois que o front-end e o back-end são separados, o código do front-end e do back-end pode ser desacoplado. Contanto que o front-end e o back-end concordem com as interfaces e os parâmetros de interface necessários ao aplicativo, eles podem iniciar o desenvolvimento paralelo sem esperar que o outro lado termine. Nesse caso, os engenheiros de front-end e back-end podem se concentrar apenas em seu próprio trabalho de desenvolvimento, o que ajuda a construir uma equipe melhor. Além disso, no modelo de desenvolvimento de separação front-end-backend, mesmo que os requisitos mudem, desde que a interface e o formato dos dados não mudem, os desenvolvedores de back-end não precisam alterar o código, e apenas o front-end precisa mudar.
2. **Melhore a capacidade de manutenção do código**. Após a separação frontend-backend, o código do aplicativo não é mais uma mistura de código frontend e backend. Existe apenas uma dependência de chamada durante o tempo de execução. Dessa forma, manter o código fica muito mais fácil e agradável, e mudar uma parte não afetará mais o corpo inteiro. Quando seu código se torna claro e limpo, a legibilidade e a capacidade de manutenção do código melhoram bastante.
3. **Suporta vários terminais e arquitetura de serviço**. Após a separação frontend-backend, o mesmo conjunto de interfaces de dados pode servir terminais diferentes, o que é mais útil para construir aplicativos multiterminais. Além disso, como as interfaces fornecidas pelo back-end podem chamar umas às outras por meio de HTTP(S), isso ajuda a construir uma arquitetura de serviço, incluindo microsserviços.

A seguir, vamos reescrever o aplicativo de votação anterior no método de separação frontend-backend.

### Retornando dados no formato JSON

Como acabamos de dizer, no modelo de desenvolvimento de separação frontend-backend, o backend precisa fornecer APIs de dados para o frontend. Essas APIs geralmente retornam dados no formato JSON. Em um projeto Django, podemos primeiro transformar objetos em dicionários e, em seguida, usar o `JsonResponse` empacotado do Django para retornar dados no formato JSON para o navegador. A maneira específica é mostrada abaixo.

```python
# poll2/views.py
from django.http import JsonResponse

from .models import Subject


def show_subjects(request):
    queryset = Subject.objects.all()
    subjects = []
    for subject in queryset:
        subjects.append({
            'no': subject.no,
            'name': subject.name,
            'intro': subject.intro,
            'isHot': subject.is_hot
        })
    return JsonResponse(subjects, safe=False)
```

No código acima, percorremos o objeto `QuerySet` retornado pelos assuntos de consulta, transformamos os dados de cada assunto em um dicionário, salvamos os dicionários em um contêiner de lista chamado `subjects` e, finalmente, usamos `JsonResponse` para serializar a lista e retornar dados no formato JSON para o navegador. Como `JsonResponse` está serializando uma lista em vez de um dicionário, precisamos definir o valor do parâmetro `safe` como `False` para que possa serializar `subjects`; caso contrário, ocorrerá uma exceção `TypeError`.

Mantenha a rota da API no arquivo de URLs da aplicação.

```python
# poll2/urls.py
from django.urls import path

from .views import show_subjects


urlpatterns = [
    path('api/subjects/', show_subjects, name='show-subjects'),
]
```

Quando a API precisar representar modelos mais complexos, os serializadores do Django REST framework (DRF) tornam explícitos os campos publicados e evitam a conversão manual. A instalação e o uso do DRF são abordados na próxima aula; até lá, mantenha a implementação com `JsonResponse` para que este exemplo não introduza uma dependência antecipada.

### Usando Vue.js para renderizar páginas

A seguir, usamos o framework frontend Vue.js para renderizar a página. Para estudar o framework, consulte o [guia oficial do Vue](https://pt.vuejs.org/guide/introduction).

Reescreva a página `subjects.html` e use Vue.js para renderizar a página.

```html
<!DOCTYPE html>
<html lang="pt-BR">
<head>
    <meta charset="UTF-8">
    <title>Informações sobre assuntos</title>
    <style>
        /* Cascading style sheet content is omitted here */
    </style>
</head>
<body>
    <div id="container">
        <h1>Todos os assuntos</h1>
        <hr>
        <div id="main">
            <dl v-for="subject in subjects" :key="subject.no">
                <dt>
                    <a :href="'/static/html/teachers.html?sno=' + subject.no">
                        {{ subject.name }}
                    </a>
                    <span v-if="subject.isHot">Destaque</span>
                </dt>
                <dd>{{ subject.intro }}</dd>
            </dl>
        </div>
    </div>
    <script src="https://unpkg.com/vue@3.5.39/dist/vue.global.prod.js"></script>
    <script>
        const app = Vue.createApp({
            data() {
                return {
                    subjects: []
                }
            },
            created() {
                fetch('/api/subjects/')
                    .then(resp => {
                        if (!resp.ok) throw new Error('Falha ao buscar assuntos')
                        return resp.json()
                    })
                    .then(json => {
                        this.subjects = json
                    })
            }
        })
        app.mount('#main')
    </script>
</body>
</html>
```

O desenvolvimento da separação frontend-backend precisa implantar páginas frontend como recursos estáticos. Quando o projeto realmente estiver online, separaremos as partes estáticas e dinâmicas de toda a aplicação Web. Os recursos estáticos são implantados por meio de um servidor Nginx ou Apache. O programa Python que gera conteúdo dinâmico é implantado em um servidor uWSGI ou Gunicorn. As solicitações de conteúdo dinâmico são roteadas pelo Nginx ou Apache para o servidor uWSGI ou Gunicorn.

Durante o estágio de desenvolvimento, geralmente usamos o servidor de teste integrado do Django. Para experimentar a separação frontend-backend, mantenha as páginas estáticas no diretório de recursos estáticos do seu projeto e exponha apenas as rotas de API necessárias. A [próxima aula](./54.restful-architecture-and-drf-intro.md) apresenta a instalação e os fundamentos do DRF.
