## Índices

Os índices são a forma mais importante de melhorar o desempenho de consultas em bancos de dados relacionais. Um índice em um banco de dados relacional é como o índice de um livro. Podemos imaginar que se quisermos encontrar algum ponto de conhecimento em um livro, mas o livro não tiver índice, isso seria terrível. Provavelmente teremos que virar página por página para saber onde está esse ponto de conhecimento. A criação de um índice trará sobrecarga de armazenamento, assim como o índice de um livro ocupa algum espaço, mas depois de abrir mão de algum espaço, a redução no tempo de consulta também é muito óbvia.

Colunas de todos os tipos de dados no MySQL podem ser indexadas. Para o mecanismo de armazenamento InnoDB no MySQL 8.0, ele suporta três tipos de índices: índices de árvore B+, índices de texto completo e índices de árvore R. Aqui, apresentamos apenas o índice de árvore B+ mais amplamente utilizado. A razão para usar a árvore B+ é muito simples, porque é a estrutura de dados mais eficiente atualmente para armazenamento massivo de dados e classificação baseada em disco. A árvore B+ é uma [árvore equilibrada](https://en.wikipedia.org/wiki/Self-balancing_binary_search_tree). A altura da árvore geralmente é 3 ou 4, mas pode salvar dados de milhões a bilhões. Para consultar uma linha de dados desses dados, são necessárias apenas 3 ou 4 operações de E/S.

A árvore B+ é composta por um nó raiz, nós intermediários e nós folha. Os nós folha são usados para salvar dados classificados. Como os registros no índice são classificados, ao encontrar dados dentro de um nó folha, a pesquisa binária pode ser usada. Essa forma de pesquisa é muito eficiente. Quando a quantidade de dados é pequena, uma árvore B+ possui apenas um nó raiz e os dados também são salvos no nó raiz. À medida que os registros aumentam cada vez mais, a árvore B+ se divide. O nó raiz não salva mais dados, mas fornece ponteiros para os nós do próximo nível, o que ajuda a decidir rapidamente em qual nó folha os dados estão.

Ao criar uma tabela bidimensional, geralmente especificamos uma coluna de chave primária para a tabela. Um índice é criado na coluna de chave primária por padrão. Para o mecanismo de armazenamento MySQL InnoDB, por usar a estrutura de armazenamento de tabela organizada por índice, o índice na chave primária são os dados de toda a tabela, e esse tipo de índice também é chamado de **índice clusterizado**. Obviamente, uma tabela pode ter apenas um índice clusterizado, caso contrário os dados da tabela teriam que ser salvos várias vezes. Os índices que criamos por nós mesmos são todos índices secundários, e um nome mais comum é **índices não clusterizados**. Através de nossos índices personalizados não clusterizados, só podemos localizar a chave primária do registro. Ao obter dados, ainda pode ser necessário consultar novamente através do índice clusterizado na chave primária. Este fenômeno é denominado “de volta à mesa”. Portanto, usar um índice não clusterizado para obter dados geralmente é mais lento do que usar um índice clusterizado.

A seguir, vamos usar um exemplo simples para explicar o significado dos índices. Por exemplo, queremos encontrar um aluno pelo nome do aluno. Essa cena deve ser muito comum no desenvolvimento real, assim como encontrar produtos pelo nome do produto. Podemos usar a palavra-chave `explain` do MySQL para visualizar o plano de execução do SQL, ou seja, as etapas específicas para o banco de dados executar a instrução SQL.

```SQL
explain select * from tb_student where stuname='Lin Zhennan'\G
```

```
*************************** 1. row ***************************
           id: 1
  select_type: SIMPLE
        table: tb_student
   partitions: NULL
         type: ALL
possible_keys: NULL
          key: NULL
      key_len: NULL
          ref: NULL
         rows: 11
     filtered: 10.00
        Extra: Using where
1 row in set, 1 warning (0.00 sec)
```

No plano de execução SQL acima, vários itens merecem nossa atenção:

1. `select_type`: o tipo de consulta.
   - `SIMPLE`: `SELECT` simples, não há necessidade de usar `UNION` ou subconsulta.
   - `PRIMARY`: se a consulta contiver uma subconsulta, o `SELECT` mais externo será marcado como `PRIMARY`.
   - `UNION`: a segunda ou posterior instrução `SELECT` em uma operação `UNION`.
   - `SUBQUERY`: o primeiro `SELECT` em uma subconsulta.
   - `DERIVED`: a subconsulta `SELECT` de uma tabela derivada.
2. `table`: tabela correspondente à consulta.
3. `type`: a forma como o MySQL encontra linhas que satisfazem a condição na tabela, também chamada de tipo de acesso. Inclui `ALL` (varredura de tabela completa), `index` (varredura de índice completo, apenas percorre a árvore de índice), `range` (varredura de intervalo de índice), `ref` (varredura de índice não exclusivo), `eq_ref` (varredura de índice exclusivo), `const` / `system` (consulta em nível constante) e `NULL` (sem necessidade de acessar a tabela ou índice). Entre todos os tipos de acesso, `ALL` é obviamente o pior em desempenho. A verificação completa da tabela que representa significa que cada linha da tabela deve ser verificada para encontrar linhas correspondentes.
4. `possible_keys`: índices que o MySQL pode escolher, mas **não pode utilizá-los**.
5. `key`: o índice que o MySQL realmente usa. Se for `NULL`, significa que nenhum índice foi usado.
6. `key_len`: o comprimento do índice utilizado. Claro, quando a consulta não é afetada, quanto menor o comprimento, melhor.
7. `rows`: o número de linhas que a consulta precisa verificar. Este é um **valor estimado**.
8. `extra`: informações extras sobre a consulta.
   - `Using filesort`: O MySQL não pode usar o índice para finalizar a operação de classificação.
   - `Using index`: são utilizadas apenas informações do índice, não sendo necessário consultar novamente a tabela para obter mais informações.
   - `Using temporary`: O MySQL precisa usar uma tabela temporária para salvar o conjunto de resultados. Isso geralmente é usado para agrupar e classificar.
   - `Impossible where`: a cláusula `where` não leva a nenhuma linha que satisfaça a condição.
   - `Distinct`: depois que o MySQL encontra a primeira linha correspondente, ele para de procurar mais linhas para a combinação de linhas atual.
   - `Using where`: as colunas consultadas não são cobertas pelo índice e a condição de filtro não é a coluna inicial do índice.

A partir do plano de execução acima, podemos ver que quando consultamos um aluno pelo nome do aluno, na verdade ele está fazendo uma varredura completa da tabela. Escusado será dizer que o desempenho dessa consulta deve ser muito ruim, especialmente quando há muitas linhas na tabela. Se precisarmos consultar frequentemente os alunos pelo nome do aluno, devemos criar um índice na coluna correspondente ao nome do aluno e usar o índice para agilizar a consulta.

```SQL
create index idx_student_name on tb_student(stuname);
```

Verifique o plano de execução do SQL agora mesmo novamente.

```SQL
explain select * from tb_student where stuname='Lin Zhennan'\G
```

```
*************************** 1. row ***************************
           id: 1
  select_type: SIMPLE
        table: tb_student
   partitions: NULL
         type: ref
possible_keys: idx_student_name
          key: idx_student_name
      key_len: 62
          ref: const
         rows: 1
     filtered: 100.00
        Extra: NULL
1 row in set, 1 warning (0.00 sec)
```

Você pode notar que depois de criar um índice no nome do aluno, a consulta agora não é mais uma verificação completa da tabela, mas uma consulta baseada em índice, e o número de linhas verificadas é apenas uma linha única. Obviamente, isso melhora muito o desempenho da consulta. O MySQL também permite criar um índice de prefixo, ou seja, criar um índice nos primeiros N caracteres do campo indexado. Desta forma, o espaço ocupado pelo índice pode ser reduzido, mas economizar espaço pode muito provavelmente ser uma perda de tempo. **Tempo e espaço são contradições irreconciliáveis**, conforme mostrado abaixo.

```SQL
create index idx_student_name_1 on tb_student(stuname(1));
```

O índice acima equivale a criar um índice de acordo com o primeiro caractere do nome do aluno. Vejamos novamente o plano de execução SQL.

```SQL
explain select * from tb_student where stuname='Lin Zhennan'\G
```

```
*************************** 1. row ***************************
           id: 1
  select_type: SIMPLE
        table: tb_student
   partitions: NULL
         type: ref
possible_keys: idx_student_name
          key: idx_student_name
      key_len: 5
          ref: const
         rows: 2
     filtered: 100.00
        Extra: Using where
1 row in set, 1 warning (0.00 sec)
```

Não sei se todos perceberam que desta vez as linhas digitalizadas viraram 2 linhas, pois há dois alunos cujo sobrenome é “Lin” na tabela de alunos. Se usarmos apenas o primeiro caractere do nome como índice, então, por meio do índice, a consulta encontrará essas duas linhas.

Se quiser excluir um índice, você pode usar o SQL abaixo.

```SQL
alter table tb_student drop index idx_student_name;
```

ou

```SQL
drop index idx_student_name on tb_student;
```

Ao criar índices, também podemos usar índices compostos e índices funcionais. No MySQL, as partes funcionais de chave foram introduzidas no MySQL 8.0.13, não no MySQL 5.7. Em versões 5.7, uma alternativa é criar uma coluna gerada para a expressão e indexar essa coluna. Fazer bom uso de índices compostos para obter a **cobertura do índice** pode reduzir a classificação desnecessária e as operações de retorno à tabela, e isso permitirá que o desempenho da consulta melhore em múltiplos. Os leitores interessados podem estudá-lo por conta própria.

Simplesmente resumimos as regras de design de índices para todos:

1. As colunas **mais adequadas** para índices são as colunas que aparecem na cláusula **`WHERE`** e nas cláusulas de junção.
2. Quanto maior for a cardinalidade da coluna indexada, ou seja, quanto mais valores e menos valores repetidos, melhor será o efeito do índice.
3. Usar um **índice de prefixo** pode reduzir o espaço ocupado pelo índice e mais índices podem ser armazenados em cache na memória.
4. **Quanto mais índices, melhor** está errado. Embora os índices acelerem as operações de leitura, ou seja, consultas, operações de gravação, incluindo inserção, exclusão e atualização, todas se tornam mais lentas, porque alterações nos dados levam a atualizações de índices, assim como adicionar ou excluir capítulos de um livro precisa atualizar o índice analítico.
5. Ao usar o mecanismo de armazenamento InnoDB, o índice normal de uma tabela salva o valor da chave primária, portanto **a chave primária deve escolher um tipo de dados o mais curto possível**. Isso pode efetivamente reduzir o espaço ocupado pelo índice e melhorar o efeito de cache do índice.

Por fim, mais um ponto precisa ser explicado. Para o índice de árvore B usado pelo InnoDB, para colunas numéricas, além da comparação de igualdade, o índice ainda funciona ao usar `>`, `<`, `>=`, `<=`, `BETWEEN...AND...` e `<>`; para colunas de string, se a consulta difusa for usada e o curinga não estiver no início, o índice também funciona, mas em outras situações o índice se torna inválido, o que significa que é muito provável que uma consulta completa da tabela seja feita.
