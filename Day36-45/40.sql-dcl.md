## DCL em detalhes

Os servidores de banco de dados geralmente contêm dados muito importantes. Podemos usar o controle de acesso para garantir que esses dados estejam seguros, e o DCL é o que resolve esse problema. Ele pode conceder permissões de acesso a um usuário específico ou recuperar permissões específicas de um usuário específico. DCL é muito importante para administradores de banco de dados, porque o gerenciamento de permissões de usuários está relacionado à segurança do banco de dados. Simplificando, através do DCL podemos permitir que usuários confiáveis ​​acessem o banco de dados, impedir que usuários não confiáveis ​​acessem o banco de dados e, ao mesmo tempo, minimizar as permissões de cada visitante, para que as permissões do visitante sejam apenas suficientes.

### Criando usuários

Podemos usar o SQL abaixo para criar um usuário e especificar uma senha de acesso.

```sql
CREATE USER 'wangdachui'@'%' IDENTIFIED BY 'Wang.618';
```

O SQL acima cria um usuário chamado `wangdachui`. Sua senha é `Wang.618`. Este usuário pode acessar o servidor de banco de dados de qualquer host, pois após `@` utilizamos o curinga `%`, que pode representar qualquer quantidade de caracteres. Se quisermos limitar o usuário `wangdachui` para que ele só possa acessar o servidor de banco de dados a partir de hosts no segmento de rede `192.168.0.x`, podemos modificar a instrução SQL da seguinte maneira.

```sql
DROP USER IF EXISTS 'wangdachui'@'%';

CREATE USER 'wangdachui'@'192.168.0.%' IDENTIFIED BY 'Wang.618';
```

Neste momento, se utilizarmos a conta `wangdachui` para acessar o servidor de banco de dados, dificilmente poderemos fazer alguma operação, pois esta conta não possui nenhuma permissão de operação.

### Concessão de privilégios

Usamos a instrução abaixo para conceder a `wangdachui` a permissão para consultar a tabela de faculdades (`tb_college`) no banco de dados `school`.

```sql
GRANT SELECT ON `school`.`tb_college` TO 'wangdachui'@'192.168.0.%';
```

Também podemos permitir que `wangdachui` tenha permissão de consulta em todos os objetos no banco de dados `school`, conforme mostrado abaixo.

```sql
GRANT SELECT ON `school`.* TO 'wangdachui'@'192.168.0.%';
```

Se quisermos que `wangdachui` também tenha as permissões `insert`, `delete` e `update`, podemos fazer isso da seguinte maneira.

```sql
GRANT INSERT, DELETE, UPDATE ON `school`.* TO 'wangdachui'@'192.168.0.%';
```

Se também quisermos conceder permissão a `wangdachui` para executar DDL, podemos usar o SQL mostrado abaixo.

```sql
GRANT CREATE, DROP, ALTER ON `school`.* TO 'wangdachui'@'192.168.0.%';
```

Se quisermos que a conta `wangdachui` tenha todas as permissões de operação em todos os objetos de todos os bancos de dados, podemos executar a operação mostrada abaixo. Mas, em geral, não faríamos isso, porque como dissemos antes, as permissões só precisam ser suficientes. Uma conta comum não deveria ter permissões tão grandes.

```sql
GRANT ALL PRIVILEGES ON *.* TO 'wangdachui'@'192.168.0.%';
```

### Revogando privilégios

Se quisermos recuperar as permissões `insert`, `delete` e `update` que `wangdachui` possui no banco de dados `school`, podemos usar a operação abaixo.

```sql
REVOKE INSERT, DELETE, UPDATE ON `school`.* FROM 'wangdachui'@'192.168.0.%';
```

Se quisermos recuperar todas as permissões, podemos fazê-lo da seguinte maneira.

```sql
REVOKE ALL PRIVILEGES ON *.* FROM 'wangdachui'@'192.168.0.%';
```

Deve ser explicado que, como o banco de dados pode armazenar em cache as permissões do usuário, após conceder ou retirar as permissões, podemos executar a seguinte instrução para fazer com que as novas permissões entrem em vigor imediatamente.

```sql
FLUSH PRIVILEGES;
```
