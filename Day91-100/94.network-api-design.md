## Network API Design

Many web applications and mobile applications now use a frontend-backend separation model. To say it simply, the frontend or mobile client interacts with the backend through network APIs, gets the data provided by the interface, and is responsible for rendering the user interface. API is short for Application Programming Interface. A network API usually refers to a resource that can be accessed through a URL. Through that URL, a client can ask the server to operate on a resource and return the result. You can think about it this way: a network API is also a kind of encapsulation. To say it simply, it hides complex business logic behind a simple API.

In that sense, a network API is also a kind of encapsulation: complex business logic is hidden behind a simple interface.

The general format of a URL is:

```text
protocol://username:password@host:port/path1/.../pathN/resource
```

> **Note**: The username, password, port, and path in a URL are not required parts. They can be set according to actual need.

Network APIs are usually accessed over HTTP or HTTPS. One of the biggest advantages of HTTP-based APIs is that they are simple to access and easy to use across languages and applications.

### Design Principles

#### The Key Issue

An important principle when designing APIs for mobile or PC clients is this: **design the API around business entities, not around user-interface operations**.

If the API is designed directly around user actions or screen functions, then as requirements change and the interface changes, the backend developer has to keep rewriting the API or maintaining multiple versions of it. That increases both development cost and maintenance cost.

A better approach is to treat business entities as resources provided by the server, and URLs as the identifiers of those resources. This is simpler and more natural. For relatively complex user actions, a “facade” can be provided to combine several interfaces together when necessary.

The lesson gives an example of comment-related APIs from a website:

| API | Meaning |
| --- | --- |
| `comments/show` | Get the comment list for a post |
| `comments/by_me` | Get my own comments |
| `comments/to_me` | Get comments sent to me |
| `comments/mentions` | Get comments that mention me |
| `comments/create` | Create a comment |
| `comments/destroy` | Delete a comment |
| `comments/reply` | Reply to a comment |

It should be explained that these interfaces are **not REST-style APIs**. REST is an architectural style that is often considered very suitable for distributed network applications, but it should still be read critically, because some views are not completely correct and some contents may even contradict each other.

API responses are usually returned in **JSON** or **XML**, though XML has mostly fallen out of favor. For JSON responses, the lesson recommends avoiding `null` values whenever possible, because once such values are handled badly, they can bring unnecessary trouble to frontend and mobile developers, especially when they use strongly typed languages. One way to reduce this problem is to enforce `not null` constraints or sensible default values in database design.

#### Other Issues

The lesson also mentions several practical issues:

1. **Update prompts**. Design an API that clients call first so the server can tell them whether an app update is available.
2. **Version upgrades**. API versioning should consider backward compatibility. New and old versions should both remain accessible for a while. Version information can be placed in the URL or in HTTP headers.
3. **Image sizing**. Mobile clients may need several sizes for the same image. One approach is to pass size parameters when fetching an image. A better approach is often to use cloud storage or a CDN that already supports image resizing.

### Writing API Documentation

The lesson then uses a comment API as an example to show how interface documentation can be written.

First, it defines global status codes:

| Code | Message | Meaning |
| --- | --- | --- |
| `10000` | Get comments successfully |  |
| `10001` | Create comment successfully |  |
| `10002` | Cannot create comment | Failed content review |
| `10003` | Comment has been deleted | The comment was removed before viewing |

#### 1. Get article comments

**GET** `/articles/{article-id}/comments/`

- Developer: Wang Dachui
- Last updated: August 10, 2018
- Tag: `v1.0`
- Description: get all comments for the specified article
- Usage note: returns 20 records by default and requires an identity key in the request header

Request parameters:

| Parameter | Type | Required | Location | Meaning |
| --- | --- | --- | --- | --- |
| `page` | integer | no | query | page number, default `1` |
| `size` | integer | no | query | number of comments per request (`10~100`), default `20` |
| `key` | string | yes | header | user identity key |

Example response:

```json
{
    "code": 10000,
    "message": "Comments retrieved successfully",
    "page": 1,
    "size": 10,
    "totalPage": 35,
    "contents": [
        {
            "userId": 1700095,
            "nickname": "Wang Dachui",
            "pubDate": "July 31, 2018",
            "content": "Is the editor sick or what?"
        },
        {
            "userId": 1995322,
            "nickname": "Bai Yuanfang",
            "pubDate": "August 2, 2018",
            "content": "The person upstairs said it well."
        }
    ]
}
```

#### 2. Add a new article comment

**POST** `/articles/{article-id}/comments`

- Developer: Wang Dachui
- Last updated: August 10, 2018
- Tag: `v1.0`
- Description: create a comment for the specified article

Request parameters:

| Parameter | Type | Required | Location | Meaning |
| --- | --- | --- | --- | --- |
| `userId` | string | yes | body | user ID |
| `key` | string | yes | header | user token |
| `content` | string | yes | body | comment content |

Example response:

```json
{
    "code": 10001,
    "message": "Comment created successfully",
    "comment": {
        "pubDate": "July 31, 2018",
        "content": "Is the editor sick or what?"
    }
}
```

The final practical advice in the lesson is that if you do not already have experience writing API documentation, online platforms such as RAP2 or YAPI can help you write and manage it.

### Summary

The core point of this lesson is straightforward: an API should be designed around stable business resources, not fragile interface actions. Good network API design also includes practical details such as versioning, update prompts, image handling, response conventions, and careful documentation. The documentation is not a side note after the interface; it is part of the interface itself.
