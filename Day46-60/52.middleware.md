## Usando Middleware

Antes disso, já estabelecíamos uma regra que o usuário deveria fazer login antes de votar, mas agora surge um novo problema. Se muitas funções em nosso aplicativo precisam que o usuário faça login primeiro, por exemplo, se a exportação do relatório Excel e a visualização do gráfico estatístico são limitadas a usuários logados, precisamos adicionar código em cada função de visualização para verificar se `userid` está na sessão? A resposta é não. Se fizermos isso, nossas funções de visualização certamente estarão cheias de códigos repetidos. O mestre de programação *Martin Fowler* disse uma vez: **o código tem muitos maus cheiros, e a duplicação é o pior deles**. Em programas Python, podemos usar decoradores para dar habilidades extras às funções. Em projetos Django, podemos colocar código repetido, como verificar se o usuário está logado no **middleware**.

### Visão geral do middleware Django

Middleware é um componente colocado entre o processo de solicitação e o processo de resposta de um aplicativo da Web. Ele desempenha o papel de interceptador e filtro em toda a aplicação Web. Através do middleware, podemos interceptar solicitações e respostas e filtrá-las. Simplesmente falando, isso significa fazer processamento extra. Normalmente, um componente de middleware concentra-se apenas em uma coisa específica. Por exemplo, a estrutura Django usa `SessionMiddleware` para suportar sessão e usa `AuthenticationMiddleware` para implementar autenticação de solicitação baseada em sessão. Ao combinar vários componentes de middleware, podemos concluir tarefas mais complexas. Isso é exatamente o que o Django faz.

O arquivo de configuração de um projeto Django inclui configuração de middleware, conforme mostrado abaixo.

```python
MIDDLEWARE = [
    'django.middleware.security.SecurityMiddleware',
    'django.contrib.sessions.middleware.SessionMiddleware',
    'django.middleware.common.CommonMiddleware',
    'django.middleware.csrf.CsrfViewMiddleware',
    'django.contrib.auth.middleware.AuthenticationMiddleware',
    'django.contrib.messages.middleware.MessageMiddleware',
    'django.middleware.clickjacking.XFrameOptionsMiddleware',
]
```

Vamos explicar um pouco o papel desses middlewares:

1. `CommonMiddleware` - configuração básica de middleware. Ele pode lidar com alguns parâmetros de configuração abaixo.
`DISALLOWED_USER_AGENTS` - agentes de usuário (navegadores) não permitidos `APPEND_SLASH` - se deseja anexar `/` `USE_ETAG` - relacionado ao cache do navegador
2. `SecurityMiddleware` – middleware relacionado à segurança. Ele pode lidar com itens de configuração relacionados à segurança.
`SECURE_HSTS_SECONDS` - hora de forçar HTTPS `SECURE_HSTS_INCLUDE_SUBDOMAINS` - se HTTPS cobre subdomínios `SECURE_CONTENT_TYPE_NOSNIFF` - se o navegador tem permissão para adivinhar o tipo de conteúdo `SECURE_BROWSER_XSS_FILTER` - se deve ativar o filtro de ataque de script entre sites `SECURE_SSL_REDIRECT` - se deve redirecionar para HTTPS `SECURE_REDIRECT_EXEMPT` - isento de redirecionamento para HTTPS
3. `SessionMiddleware` - middleware de sessão.
4. `CsrfViewMiddleware` - middleware que evita a falsificação de solicitações entre sites por meio da geração de tokens.
5. `XFrameOptionsMiddleware` - middleware que evita ataques de clickjacking definindo parâmetros de cabeçalho de resposta.

Durante o processo de solicitação, o middleware acima é executado de cima para baixo na ordem em que são escritos. Em seguida, ocorre a análise da URL e, finalmente, a solicitação chega à função de visualização. Durante o processo de resposta, o middleware acima é executado de baixo para cima na ordem em que são escritos, que é exatamente o oposto da ordem usada durante o processo de solicitação.

### Middleware personalizado

Existem duas maneiras de implementar middleware no Django: implementação baseada em classes e implementação baseada em funções. O segundo está mais próximo da forma como os decoradores são escritos. Um decorador é na verdade uma aplicação do padrão proxy. Ele coloca no proxy preocupações transversais, funções que não têm uma conexão inevitável com a lógica de negócios normal, como autenticação de identidade, gravação de log e conversão de codificação. O objeto proxy completa o comportamento do objeto proxy e adiciona funções extras. O middleware intercepta e filtra as solicitações e respostas dos usuários e adiciona processamento extra. Neste ponto, é exatamente o mesmo que um decorador, portanto implementar middleware de forma baseada em funções é quase exatamente o mesmo que escrever um decorador. Abaixo usamos middleware customizado para implementar a validação de login do usuário.

```python
"""
middlewares.py
"""
from django.http import JsonResponse
from django.shortcuts import redirect

# Resource paths that can only be accessed after login
LOGIN_REQUIRED_URLS = {'/praise/', '/criticize/', '/excel/', '/teachers_data/'}


def check_login_middleware(get_resp):

    def wrapper(request, *args, **kwargs):
        # The requested resource path is in the set above
        if request.path in LOGIN_REQUIRED_URLS:
            # If the session contains userid, treat the user as logged in
            if 'userid' not in request.session:
                # Check whether it is an Ajax request
                if request.is_ajax():
                    # Ajax request returns JSON data to tell the user to log in
                    return JsonResponse({'code': 10003, 'hint': 'Please log in first'})
                else:
                    backurl = request.get_full_path()
                    # Non-Ajax request is directly redirected to the login page
                    return redirect(f'/login/?backurl={backurl}')
        return get_resp(request, *args, **kwargs)

    return wrapper
```

Claro, também podemos definir uma classe para atuar como decoradora. Se a classe possui o método mágico `__call__`, os objetos desta classe podem ser chamados como funções. Portanto, a seguir está outra maneira de implementar middleware. O princípio é exatamente o mesmo do código acima.

Existe outra maneira baseada em classe de implementar middleware. Essa forma não é mais recomendada em versões mais recentes do Django, mas no código que você encontrar, você ainda poderá encontrar esse estilo. O código aproximado é mostrado abaixo.

```python
from django.utils.deprecation import MiddlewareMixin


class MyMiddleware(MiddlewareMixin):

    def process_request(self, request):
        pass

    def process_view(self, request, view_func, view_args, view_kwargs):
        pass

    def process_template_response(self, request, response):
        pass

    def process_response(self, request, response):
        pass

    def process_exception(self, request, exception):
        pass
```

Os cinco métodos da classe acima são todos funções de gancho de middleware. Eles são chamados de volta quando uma solicitação do usuário é recebida, antes de entrar na função de visualização, ao renderizar o modelo, ao retornar a resposta e quando ocorre uma exceção. É claro que escrever esses métodos depende das necessidades do middleware. Nem toda situação precisa que todos os cinco métodos sejam substituídos. Depois de escrever o código do middleware, precisamos alterar o arquivo de configuração para ativar o middleware e fazê-lo funcionar.

```python
MIDDLEWARE = [
    'django.middleware.security.SecurityMiddleware',
    'django.contrib.sessions.middleware.SessionMiddleware',
    'django.middleware.common.CommonMiddleware',
    'django.middleware.csrf.CsrfViewMiddleware',
    'django.contrib.auth.middleware.AuthenticationMiddleware',
    'django.contrib.messages.middleware.MessageMiddleware',
    'django.middleware.clickjacking.XFrameOptionsMiddleware',
    'debug_toolbar.middleware.DebugToolbarMiddleware',
    'vote.middlewares.check_login_middleware',
]
```

Observe a ordem dos elementos na lista de middleware acima. Quando uma solicitação do usuário é recebida, o middleware é executado um por um, de cima para baixo. Somente após a conclusão desses middlewares a solicitação finalmente chega à função de visualização. É claro que durante esse processo a solicitação do usuário pode ser interceptada, assim como no middleware customizado acima. Se o usuário acessar recursos protegidos sem fazer login, o middleware redirecionará a solicitação diretamente para a página de login, e o middleware posterior e a função de visualização não continuarão a ser executados. Durante o processo de resposta, o middleware acima é executado um por um, de baixo para cima. Dessa forma, também podemos fazer processamento adicional da resposta.

A ordem de execução do middleware é muito importante. Para middleware que depende um do outro, devemos garantir que o middleware do qual depende seja colocado antes do middleware que depende dele. Assim como o middleware customizado que escrevemos acima, ele deve ser colocado após `SessionMiddleware`, pois precisamos depender do objeto `session` que esse middleware vincula à solicitação para saber se o usuário efetuou login.
