## Hive in Practice

### Hive Overview

[Hive](https://hive.apache.org/) is a data warehouse tool based on Hadoop that was open-sourced by Facebook and is now maintained by the Apache Software Foundation. It is the most widely used big-data solution.

Simply speaking, Hive can turn SQL queries into MapReduce or Spark tasks. It gives perfect support to SQL, and can very conveniently do big-data statistics, so that people who cannot use Java or Scala can also use the big-data platform and distributed file system to store and process data.

<img src="res/what_hive_can_do.png" style="zoom:50%;">

<img src="res/HADOOP-ECOSYSTEM-Edureka.png">

Hive's main roles include:

1. Mapping structured data files to tables.
2. Providing HiveQL, a SQL-like query language for large-scale analysis.
3. Running on top of HDFS and distributed execution engines such as MapReduce, Spark, and Tez.

Hive's characteristics are shown below:

1. SQL-like syntax lowers the learning barrier.
2. Good scalability for PB-level data.
3. Supports multiple storage formats such as Text, ORC, Parquet, and Avro.
4. Works with different execution engines.
5. High latency compared with OLTP databases, so it is better for offline analytics than low-latency transactional workloads.

### Basic Hive Syntax

1. Delete a database.

```hive
DROP DATABASE IF EXISTS eshop CASCADE;
```

2. Create a database.

```hive
CREATE DATABASE IF NOT EXISTS eshop;
```

3. Switch context.

```hive
USE eshop;
```

4. Create an external table.

```hive
CREATE EXTERNAL TABLE IF NOT EXISTS dim_user_info 
(
user_id           STRING,
user_name         STRING, 
sex               STRING,
age               INT,
city              STRING,
first_active_time STRING,
level             INT,
extra1            STRING,
extra2            MAP<STRING, STRING>
)
ROW FORMAT DELIMITED 
FIELDS TERMINATED BY '\t'
COLLECTION ITEMS TERMINATED BY ','
MAP KEYS TERMINATED BY ':'
LINES TERMINATED BY '\n'
STORED AS TEXTFILE;
```

5. Load data.

```hive
LOAD DATA LOCAL INPATH '/home/hadoop/data/user_info/user_info.txt' OVERWRITE INTO TABLE dim_user_info;
LOAD DATA INPATH '/user/data/user_info.txt' OVERWRITE INTO TABLE dim_user_info;
```

6. Create a partitioned table.

```hive
CREATE TABLE IF NOT EXISTS fact_user_trade 
(
user_name      STRING,
piece          INT,
price          DOUBLE,
pay_amount     DOUBLE,
goods_category STRING,
pay_time       BIGINT
)  
PARTITIONED BY (dt STRING)
ROW FORMAT DELIMITED
FIELDS TERMINATED BY '\t'
LINES TERMINATED BY '\n'
STORED AS TEXTFILE;
```

Enable dynamic partitions:

```hive
SET hive.exec.dynamic.partition=true;
SET hive.exec.dynamic.partition.mode=nonstrict;
SET hive.exec.max.dynamic.partitions=1000;
SET hive.exec.max.dynamic.partitions.pernode=10000;
```

Repair partitions:

```hive
MSCK REPAIR TABLE fact_user_trade;
```

### Query Examples

```hive
-- Turn on local mode auto judgment
SET hive.exec.mode.local.auto=true;
-- Set the number of MAPREDUCE files
SET hive.exec.mode.local.auto.input.files.max=128;
-- Set the size of MAPREDUCE files
SET hive.exec.mode.local.auto.input.bytes.max=134217728;
-- Set the number of Reduce tasks to 1
SET mapreduce.job.reduces=1;

-- Query the first 10 female users in Beijing
SELECT user_name 
  FROM dim_user_info
 WHERE city='beijing'
       AND sex='female'
 LIMIT 10;

-- Query the user name, purchase quantity, and payment amount of users who bought food goods on 2019-03-24
SELECT user_name
     , piece
     , pay_amount
  FROM fact_user_trade
 WHERE dt='2019-03-24'
       AND goods_category='food';

-- Count the total payment amount of user ELLA in 2018 and the number of days between the most recent and earliest consumption
SELECT SUM(pay_amount) AS total
     , DATEDIFF(
           MAX(FROM_UNIXTIME(pay_time, 'yyyy-MM-dd')),
           MIN(FROM_UNIXTIME(pay_time, 'yyyy-MM-dd'))
       ) AS gap_days
  FROM fact_user_trade
 WHERE YEAR(dt)='2018'
       AND user_name='ELLA';
```

### Grouping and Aggregation

```hive
SELECT goods_category
     , COUNT(DISTINCT user_name) AS total_user
     , SUM(pay_amount) AS total_pay
  FROM fact_user_trade
 WHERE dt BETWEEN '2019-01-01' AND '2019-04-30'
 GROUP BY goods_category;

SELECT user_name
     , SUM(pay_amount) AS total
  FROM fact_user_trade
 WHERE dt BETWEEN '2019-04-01' AND '2019-04-30'
 GROUP BY user_name
HAVING SUM(pay_amount) > 50000;

SELECT COUNT(t.user_name)
  FROM (SELECT user_name
             , COUNT(DISTINCT goods_category) AS total
          FROM fact_user_trade
         WHERE YEAR(dt)='2018'
         GROUP BY user_name
        HAVING COUNT(DISTINCT goods_category)>2) AS t;

SELECT user_name
     , SUM(pay_amount) AS total
  FROM fact_user_trade
 WHERE dt BETWEEN '2019-04-01' AND '2019-04-30'
 GROUP BY user_name
 ORDER BY total DESC
 LIMIT 5;
```

Age-segment statistics:

```hive
SELECT CASE WHEN age < 20 THEN 'Under 20' 
            WHEN age < 30 THEN 'Under 30' 
            WHEN age < 40 THEN 'Under 40' 
            ELSE '40 and above' 
        END AS age_seg, 
       COUNT(DISTINCT user_id) AS total 
  FROM dim_user_info 
 GROUP BY CASE WHEN age < 20 THEN 'Under 20' 
               WHEN age < 30 THEN 'Under 30' 
               WHEN age < 40 THEN 'Under 40' 
               ELSE '40 and above' 
           END;

SELECT age_seg,
       IF(marriage_status = 1, 'Married', 'Unmarried') AS marriage_status,
       COUNT(*) AS total
  FROM (SELECT CASE WHEN age < 20 THEN 'Under 20'
                    WHEN age < 30 THEN '20-30'
                    WHEN age < 40 THEN '30-40'
                    ELSE '40 and above'
                END AS age_seg,
               extra2['marriage_status'] AS marriage_status
          FROM dim_user_info
         WHERE TO_DATE(first_active_time) BETWEEN '2018-01-01' AND '2018-12-31'
       ) AS t
 WHERE age_seg in ('20-30', '30-40')
 GROUP BY age_seg, IF(marriage_status = 1, 'Married', 'Unmarried');
```

Collect categories into arrays or maps:

```hive
SELECT user_name,
       COLLECT_SET(goods_category) AS categories
  FROM fact_user_trade
 GROUP BY user_name;

SELECT user_name
     , CONCAT_WS(', ', COLLECT_SET(goods_category)) AS categories
  FROM fact_user_trade
 GROUP BY user_name;

SELECT user_name
     , STR_TO_MAP(CONCAT_WS(',', COLLECT_LIST(CONCAT(goods_category, ':', cnt)))) AS category_cnt_map
  FROM (SELECT user_name
             , goods_category
             , COUNT(*) AS cnt
          FROM fact_user_trade
         GROUP BY user_name, goods_category) AS t
 GROUP BY user_name;
```

### Lateral View / Explode

```hive
CREATE OR REPLACE VIEW v_user_categories
AS
SELECT user_name
     , COLLECT_SET(goods_category) AS categories
  FROM fact_user_trade
 WHERE dt BETWEEN '2019-04-01' and '2019-04-30' 
 GROUP BY user_name;

CREATE OR REPLACE VIEW v_user_categories_map
AS
SELECT user_name
     , STR_TO_MAP(CONCAT_WS(',', COLLECT_LIST(CONCAT(goods_category, ':', cnt)))) AS category_cnt_map
  FROM (SELECT user_name
             , goods_category
             , COUNT(*) AS cnt
          FROM fact_user_trade
         WHERE dt BETWEEN '2019-04-01' AND '2019-04-30' 
         GROUP BY user_name, goods_category) AS t
 GROUP BY user_name;

SELECT user_name
     , category
  FROM v_user_categories
LATERAL VIEW EXPLODE(categories) t AS category;

SELECT user_name
     , category
     , cnt
  FROM v_user_categories_map
LATERAL VIEW EXPLODE(category_cnt_map) t AS category, cnt;
```

#### Data Types

Hive data types generally fall into primitive and complex categories.

| Type | Description | Typical use |
| --- | --- | --- |
| `TINYINT` | 1-byte signed integer | |
| `SMALLINT` | 2-byte signed integer | |
| `INT` | 4-byte signed integer | |
| `BIGINT` | 8-byte signed integer | IDs, counters |
| `BOOLEAN` | `TRUE` / `FALSE` | |
| `FLOAT` | Single-precision float | |
| `DOUBLE` | Double-precision float | |
| `DECIMAL(P,S)` | Fixed-precision decimal | Money |
| `STRING` | Variable-length string | Text, JSON |
| `CHAR` | Fixed-length string | |
| `VARCHAR` | Length-limited string | |
| `BINARY` | Binary data | |
| `TIMESTAMP` | Timestamp | |
| `DATE` | Date | |
| `INTERVAL` | Time interval | |
| `STRUCT` | Composite object | Event payloads |
| `MAP` | Key-value collection | Attributes |
| `ARRAY` | Ordered same-type collection | Lists |

Example:

```hive
CREATE TABLE complex
(
  c1 ARRAY<INT>,
  c2 MAP<STRING, INT>,
  c3 STRUCT<a:STRING, b:INT, c:DOUBLE>,
  c4 STRUCT<name:STRING, addrs:ARRAY<STRUCT<city:STRING, zipcode:STRING>>>
);

SELECT c1[0]
     , c2['key']
     , c3.b
     , c4.addrs[0].city
  FROM complex;
```

Hive supports both implicit and explicit type conversion:

```hive
SELECT CAST('123' AS INT);
SELECT CAST(3.14159 AS DECIMAL(5,2));
```

#### Table Types

| Table type | Meaning |
| --- | --- |
| Managed table | Hive manages both metadata and data |
| External table | Hive manages metadata only; data remains where it is |
| Partitioned table | Data is split into directory-based partitions |
| Bucketed table | Data is hashed into fixed buckets |
| Temporary table | Session-scoped temporary data |
| View | Virtual table based on a query |

#### Common Modifiers

| Modifier | Purpose |
| --- | --- |
| `EXTERNAL` | Create an external table |
| `PARTITIONED BY` | Define partition columns |
| `CLUSTERED BY ... INTO N BUCKETS` | Define buckets |
| `STORED AS` | Choose storage format |
| `ROW FORMAT` | Define row parsing / SerDe |
| `LOCATION` | Set custom HDFS path |
| `TBLPROPERTIES` / `DBPROPERTIES` | Store metadata properties |
| `COMMENT` | Add documentation |

#### `ROW FORMAT`

Common options include delimited text, `OpenCSVSerde`, `RegexSerDe`, `JsonSerDe`, `AvroSerDe`, `ParquetHiveSerDe`, and `OrcSerde`.

> **Note**: You usually specify a SerDe explicitly only when using a non-default format such as CSV, JSON, or Regex-based parsing.

### Data Writing Patterns

Common writing patterns include:

- `INSERT INTO`: append data
- `INSERT OVERWRITE`: replace target data
- `LOAD DATA`: move existing files into the Hive table location
- `LOAD DATA LOCAL`: load from the local filesystem
- `CREATE TABLE AS SELECT`: create a table from a query result
- `INSERT OVERWRITE DIRECTORY`: export query results
- dynamic partition insert

| Function | Description | Example |

#### Application Examples

1. `FROM_UNIXTIME`: convert a timestamp into a date string.

```hive
SELECT FROM_UNIXTIME(pay_time, 'yyyy-MM-dd hh:mm:ss')
  FROM fact_user_trade LIMIT 10;
```

2. `DATEDIFF`: compute the difference between two dates.

```hive
SELECT user_name, DATEDIFF('2019-4-1', to_date(firstactivetime))
  FROM dim_user_info LIMIT 10;
```

3. `IF`: return different values based on a condition.

```hive
SELECT sex
     , IF(level > 5, 'high', 'low') AS level_type
     , COUNT(DISTINCT user_id) AS total
  FROM dim_user_info
 GROUP BY sex, IF(level > 5, 'high', 'low');
```

4. `SUBSTR`: extract a substring.

```hive
SELECT SUBSTR(first_active_time, 1, 7) AS month
     , COUNT(DISTINCT user_id) AS total
  FROM dim_user_info
 GROUP BY substr(first_active_time, 1, 7);
```

5. `GET_JSON_OBJECT`: extract the value for a specified key from a JSON string, such as `GET_JSON_OBJECT(info, '$.first_name')`.

```hive
SELECT GET_JSON_OBJECT(extra1, '$.phonebrand') AS phone
     , COUNT(DISTINCT user_id) AS total
  FROM user_info
 GROUP BY GET_JSON_OBJECT(extra1, '$.phonebrand');
```

> **Note**: The corresponding MySQL function is named `JSON_EXTRACT`.

| Operation | Example syntax | Description | Result characteristics | Typical use case |
| --- | --- | --- | --- | --- |
| `GROUP BY` | `SELECT region, product, SUM(amount) FROM sales GROUP BY region, product;` | Group by the specified columns | Outputs aggregation for one dimension combination | Single-dimension or fixed-combination summaries |
| `GROUPING SETS` | `SELECT region, product, SUM(amount) FROM sales GROUP BY GROUPING SETS ((region, product), (region), (product), ());` | Specify multiple grouping combinations at once | Outputs several chosen combinations and can include totals | Reports where only selected combinations matter |
| `CUBE` | `SELECT region, product, SUM(amount) FROM sales GROUP BY CUBE(region, product);` | Automatically generate all dimension combinations | Outputs all dimension combinations (`2^n`) | Full OLAP-style multidimensional analysis |

### Sampling Operations

| Sampling style | Example syntax | Core meaning | Random | Repeatable | Accuracy / bias | Requirements | Performance and notes | Typical use |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| Physical bucket sampling | `SELECT * FROM t TABLESAMPLE(BUCKET 2 OUT OF 8);` | Choose the second physical bucket out of eight | No | Stable | Follows table bucketing | Table must be bucketed | Reads only some files, so it is fast | Proportional sampling on large tables |
| Logical bucket sampling | `SELECT * FROM t TABLESAMPLE(BUCKET 1 OUT OF 4 ON user_id);` | Hash `user_id` into four logical buckets and take bucket one | Pseudo-random | Stable for the same key | Approximately one quarter, affected by skew | No bucketed table required | Distribution depends on key skew | Consistent train/test splits |
| Bernoulli sampling | `SELECT * FROM t WHERE RAND() < 0.1;` | Independently sample each row with 10% probability | Yes | Can be fixed with a seed | Expected proportion only | None | Requires a full scan | Exploratory down-sampling |
| Fixed-size per group | `SELECT * FROM (SELECT *, ROW_NUMBER() OVER(PARTITION BY gid ORDER BY RAND()) rn FROM t) x WHERE rn <= 100;` | Randomly choose a fixed number of rows per group | Yes | Can be fixed with a seed | Equal sample count per group | Window function support | Two scans plus sorting | Balanced sampling |
| Per-group rate sampling | `WHERE RAND() < CASE WHEN gid='A' THEN 0.2 ELSE 0.05 END` | Assign different sampling rates to each group | Yes | Can be fixed with a seed | Expected proportion with intra-group fluctuation | None | Full scan | Weighted sampling |
| `LIMIT` truncation | `SELECT * FROM t LIMIT 1000;` | Take the first `N` rows, not random | No | Stable | Biased | None | Fastest, but only for previews | Development and field inspection |

### Sorting Operations

| Sorting style | Example syntax | Core meaning | Reducers | Globally ordered | Shuffle / sorting behavior | Performance and notes | Typical use |
| --- | --- | --- | --- | --- | --- | --- | --- |
| `ORDER BY` | `SELECT * FROM t ORDER BY ts DESC;` | Global sort | 1 | Yes | All data goes to one reducer | Slowest and becomes a bottleneck on large results | Small result sets, final display |
| `ORDER BY ... LIMIT` | `SELECT * FROM t ORDER BY score DESC LIMIT 1000;` | Global Top-N | 1 | Yes | Still one reducer, but `LIMIT` helps | Acceptable for Top-N queries | Global Top-N |
| `SORT BY` | `SELECT * FROM t SORT BY ts DESC;` | Sort only inside each reducer | Multiple | No | Each reducer sorts its own partition | Faster than `ORDER BY`, but not globally sorted | Parallel export and sharded sorting |
| `DISTRIBUTE BY` | `SELECT * FROM t DISTRIBUTE BY key;` | Control how rows are sent to reducers | Multiple | No | Same keys go to the same reducer, no sort | Often paired with `SORT BY` | Partitioning for later work |
| `DISTRIBUTE BY + SORT BY` | `SELECT * FROM t DISTRIBUTE BY key SORT BY key, ts DESC;` | Same key in one reducer and sorted within that reducer | Multiple | No, but ordered within key | Same-key rows stay together and are sorted | Common and practical pattern | Time-series ordering by user or product |
| `CLUSTER BY` | `SELECT * FROM t CLUSTER BY key;` | Shortcut for `DISTRIBUTE BY key` plus ascending `SORT BY key` | Multiple | No | Syntax sugar; cannot choose direction | Simple but less flexible | Pre-sorting for bucketed writes or joins |

### Lateral View

`LATERAL VIEW` is a very important Hive syntax feature. It is mainly used together with table-generating functions such as `EXPLODE` to split one row into multiple rows. The most common use cases involve `ARRAY`, `MAP`, and JSON-like fields, and it can also be used to turn a wide table into a narrow one.

| Usage | Example syntax | Description | Typical use case |
| --- | --- | --- | --- |
| `LATERAL VIEW explode(array)` | `SELECT order_id, item FROM orders LATERAL VIEW explode(items) t AS item;` | Split an array into multiple rows, one element per row | Order details, event logs |
| `LATERAL VIEW posexplode(array)` | `SELECT order_id, pos, item FROM orders LATERAL VIEW posexplode(items) t AS pos, item;` | Split an array and keep element indexes | Clickstream analysis with order retained |
| `LATERAL VIEW explode(map)` | `SELECT user_id, k, v FROM user_tags LATERAL VIEW explode(tags) t AS k, v;` | Split a map into `(key, value)` rows | User tags, attribute key-value pairs |
| `LATERAL VIEW json_tuple(json, …)` | `SELECT log_id, ip, device FROM event_log LATERAL VIEW json_tuple(log_json,'ip','device') t AS ip, device;` | Extract fields from JSON into columns | Log parsing, semi-structured data |
| `LATERAL VIEW inline(array)` | `SELECT order_id, col1, col2 FROM orders LATERAL VIEW inline(item_structs) t AS col1,col2;` | Expand an array of structs into multiple rows and columns | Multi-attribute embedded rows |
| `LATERAL VIEW stack(n, …)` | `SELECT user_id, subject, score FROM user_score LATERAL VIEW stack(3,'math',math,'eng',english,'phy',physics) t AS subject, score;` | Turn multiple columns into multiple rows | Wide-to-long transformations |

### HiveSQL vs. MySQL

| Feature | Hive SQL | MySQL SQL | Difference |
| --- | --- | --- | --- |
| Table partitioning | `PARTITIONED BY` | No direct equivalent in this course context | Hive partitions are important for large data filtering |
| Table bucketing | `CLUSTERED BY` | Not supported | Hive can bucket data to optimize large-table operations |
| Storage formats | `STORED AS` with ORC, Parquet, and others | Row-oriented engines such as InnoDB | Hive supports columnar formats for analytics |
| Query engine | MapReduce, Tez, Spark | Traditional database engine execution | Hive can switch execution engines |
| Complex data handling | `LATERAL VIEW`, `MAP`, `ARRAY`, `STRUCT` | No comparable native support | Hive handles nested and semi-structured data better |
| Join optimization | Supports strategies such as MapJoin | Standard relational joins | Hive includes big-data-specific join optimizations |
| UDF support | UDF and UDTF support | UDF support and database-side features like triggers | Hive emphasizes analytical extensibility |
| Query optimization | Partition pruning, column pruning, dynamic partition insert | Mainly index-driven optimization | Hive optimization is designed for large-scale analytics |
| Indexes | Supported but usually not encouraged | Core performance feature | Indexes matter far more in MySQL |
| Data types | Includes `ARRAY`, `MAP`, `STRUCT` | Mostly scalar types | Hive supports richer structured types |
| Materialized views | Supported | No native support in typical MySQL usage | Hive can precompute and persist results |

### Performance Optimization

There are many ways to optimize Hive SQL query performance, including table design, query tuning, and execution engine choices.

| Optimization method | Description | Typical scenario |
| --- | --- | --- |
| Partitioning | Partition a table by a field such as date, then use partition pruning to scan less data | Large tables often filtered by date, area, and similar fields |
| Bucketing | Split data into fixed buckets using a hash value | Frequent joins or aggregations on a specific column |
| Column pruning | Query only the columns you need | Queries that touch only part of a wide table |
| Partition pruning | Filter partition columns in queries | Partitioned tables with partition filters |
| Columnar formats such as ORC / Parquet | Use compressed, analytics-friendly formats | Large-scale scanning and aggregation |
| MapJoin | Load a small table into memory and complete the join on the map side | Small-table-to-large-table joins |
| Choosing the right join type | Use sort-merge join or bucketed join when appropriate | Both tables are bucketed or sorted on the same keys |
| Choosing the right execution engine | Use MapReduce, Tez, or Spark according to the workload | Varies by query type and data volume |
| Dynamic partition insert | Enable `hive.exec.dynamic.partition=true` | Data that needs to be inserted into partitions dynamically |
| Reduce scanned data early | Use `WHERE` to filter as early as possible | Any query over very large data sets |
| Run in batches | Use `LIMIT` or `TABLESAMPLE` when full accuracy is not required | Exploration over very large data volumes |
| Materialized views | Precompute and store repeated query results | Queries that are executed repeatedly |
| Parallel execution | Enable `hive.exec.parallel=true` when appropriate | Multiple tasks that can run together |
