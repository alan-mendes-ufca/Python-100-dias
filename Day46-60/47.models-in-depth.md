## Uma análise mais aprofundada dos modelos

Na lição anterior, mencionamos que Django é um framework Web baseado na arquitetura MVC, e MVC busca a dissociação de "modelo" e "visão". Para ser mais direto, "modelo" significa dados, por isso também é comumente chamado de "modelo de dados". Em projetos reais, os modelos de dados geralmente são persistidos por meio de um banco de dados, e os bancos de dados relacionais foram e ainda são a primeira escolha para persistência. Abaixo, utilizamos um projeto de votação para explicar pontos de conhecimento relacionados aos modelos.

O projeto de votação funciona assim:

- a página inicial mostra todas as disciplinas em uma plataforma educacional online
- clicar em uma disciplina exibe seus professores
- usuários logados podem votar a favor ou contra um professor
- usuários não registrados podem se inscrever
- páginas de login e cadastro também fazem parte do projeto

MySQL é usado como banco de dados de apoio.

Primeiro criamos um projeto Django chamado `vote` e adicionamos um ambiente virtual e dependências para ele. A seguir, no projeto, crie um aplicativo chamado `polls` e uma pasta chamada `templates` para salvar páginas de modelo.

Posteriormente, transformaremos essas páginas estáticas em páginas de modelo necessárias ao projeto Django.

2. Crie as tabelas de disciplinas e professores:

```SQL
use vote;

create table `tb_subject`
(
    `no` integer auto_increment comment 'Subject ID',
    `name` varchar(50) not null comment 'Subject name',
    `intro` varchar(1000) not null default '' comment 'Subject introduction',
    `is_hot` boolean not null default 0 comment 'Whether it is a popular subject',
    primary key (`no`)
);

create table `tb_teacher`
(
    `no` integer auto_increment comment 'Teacher ID',
    `name` varchar(20) not null comment 'Teacher name',
    `sex` boolean not null default 1 comment 'Gender',
    `birth` date not null comment 'Date of birth',
    `intro` varchar(1000) not null default '' comment 'Introduction',
    `photo` varchar(255) not null default '' comment 'Photo',
    `gcount` integer not null default 0 comment 'Positive votes',
    `bcount` integer not null default 0 comment 'Negative votes',
    `sno` integer not null comment 'Subject ID',
    primary key (`no`),
    foreign key (`sno`) references `tb_subject` (`no`)
);
```

5. Django fornece ORM para resolver o problema de persistência. ORM é traduzido para o chinês como "mapeamento objeto-relacional". Como Python é uma linguagem de programação orientada a objetos, usamos modelos de objetos em programas Python para salvar dados, enquanto bancos de dados relacionais usam modelos relacionais e salvam dados com tabelas bidimensionais. Esses dois modelos não combinam. ORM é usado para obter **conversão bidirecional** entre o modelo de objeto e o modelo de relacionamento, portanto, não precisamos escrever instruções SQL e operações de cursor no código Python, porque o ORM as conclui automaticamente. Usando o ORM do Django, podemos transformar diretamente a tabela de assuntos e a tabela de professores criadas agora em classes modelo no Django.

Você pode gerar classes de modelo a partir de tabelas existentes com:

```bash
python manage.py inspectdb > polls/models.py
```

Podemos ajustar ligeiramente as classes de modelo geradas automaticamente. O código é mostrado abaixo.

```python
from django.db import models


class Subject(models.Model):
    no = models.AutoField(primary_key=True, verbose_name='ID')
    name = models.CharField(max_length=50, verbose_name='Name')
    intro = models.CharField(max_length=1000, verbose_name='Introduction')
    is_hot = models.BooleanField(verbose_name='Is Popular')

    class Meta:
        managed = False
        db_table = 'tb_subject'


class Teacher(models.Model):
    no = models.AutoField(primary_key=True, verbose_name='ID')
    name = models.CharField(max_length=20, verbose_name='Name')
    sex = models.BooleanField(default=True, verbose_name='Gender')
    birth = models.DateField(verbose_name='Date of Birth')
    intro = models.CharField(max_length=1000, verbose_name='Introduction')
    photo = models.ImageField(max_length=255, verbose_name='Photo')
    good_count = models.IntegerField(default=0, db_column='gcount', verbose_name='Positive Votes')
    bad_count = models.IntegerField(default=0, db_column='bcount', verbose_name='Negative Votes')
    subject = models.ForeignKey(Subject, models.DO_NOTHING, db_column='sno')

    class Meta:
        managed = False
        db_table = 'tb_teacher'
```

> **Observação**: As classes de modelo herdam direta ou indiretamente da classe `Model`. As classes de modelo correspondem a tabelas bidimensionais em bancos de dados relacionais, os objetos de modelo correspondem a registros na tabela e os atributos de objetos de modelo correspondem a campos na tabela. Se você não compreender completamente as definições de atributos nas classes de modelo acima, poderá consultar o conteúdo "referência de definição de modelo" fornecido posteriormente neste artigo.

### Use ORM para concluir operações CRUD em modelos

Abra um shell do Django:

```bash
python manage.py shell
```

#### Criar

```python
from polls.models import Subject

subject1 = Subject(name='Python Full-Stack Development', intro='A very popular subject', is_hot=True)
subject1.save()
subject2 = Subject(name='Full-Stack Software Testing', intro='A subject about automated testing', is_hot=False)
subject2.save()
subject3 = Subject(name='JavaEE Distributed Development', intro='Server-side development based on Java', is_hot=True)
subject3.save()
```

#### Excluir

```python
subject = Subject.objects.get(no=2)
subject.delete()
```

#### Atualizar

```python
subject = Subject.objects.get(no=1)
subject.name = 'Python and AI'
subject.save()
```

#### Consulta

```python
Subject.objects.all()
Subject.objects.filter(is_hot=True)
Subject.objects.get(no=1)
```

O ORM do Django também suporta ordenação, filtragem, agregação e consultas de objetos relacionados. No trabalho real, a maior parte do trabalho CRUD em um aplicativo Django é escrito na camada de modelo por meio dessas APIs, em vez de SQL bruto.

Para consultas de múltiplos objetos, o Django retorna um `QuerySet`. `QuerySet` usa consulta lenta, o que significa que a criação do próprio `QuerySet` não aciona imediatamente uma consulta ao banco de dados. Somente quando o queryset for realmente usado o Django enviará SQL e buscará o resultado. Isso é algo que você precisa lembrar no desenvolvimento real.

Se desejar atualizar várias linhas, não será necessário obter os objetos do modelo um por um e modificar os atributos um por um. Você pode usar o método `update()` em um `QuerySet` para atualizar várias linhas de uma vez.

### Gerenciando modelos com Django Admin

Depois de criar classes de modelo, você pode gerenciá-las através do aplicativo de administração integrado do Django. Em projetos do mundo real, o site de administração pode não satisfazer totalmente suas necessidades, mas ao aprender Django ele é uma ferramenta muito útil e também lhe dá uma noção do que um backend de gerenciamento normalmente precisa fornecer.

1. Migre as tabelas necessárias para a aplicação `admin`.

```bash
python manage.py migrate
```

2. Crie uma conta de superusuário para o site de administração. Você será solicitado a fornecer um nome de usuário, e-mail e senha.

```bash
python manage.py createsuperuser
```

> **Nota**: Ao inserir a senha, não há eco de terminal e você não pode retroceder da maneira visível usual, então digite com cuidado.

3. Execute o projeto e visite `http://127.0.0.1:8000/admin` em um navegador. Faça login com a conta de superusuário que você acabou de criar.

6. Registre classes de administração de modelo.

```python
from django.contrib import admin

from polls.models import Subject, Teacher


class SubjectModelAdmin(admin.ModelAdmin):
    list_display = ('no', 'name', 'intro', 'is_hot')
    search_fields = ('name', )
    ordering = ('no', )


class TeacherModelAdmin(admin.ModelAdmin):
    list_display = ('no', 'name', 'sex', 'birth', 'good_count', 'bad_count', 'subject')
    search_fields = ('name', )
    ordering = ('no', )


admin.site.register(Subject, SubjectModelAdmin)
admin.site.register(Teacher, TeacherModelAdmin)
```

### Implemente as páginas do assunto e do professor

1. Modifique `polls/views.py` e grave funções de visualização para renderizar a página da disciplina e a página do professor.

```python
from django.shortcuts import render, redirect

from polls.models import Subject, Teacher


def show_subjects(request):
    subjects = Subject.objects.all().order_by('no')
    return render(request, 'subjects.html', {'subjects': subjects})


def show_teachers(request):
    try:
        sno = int(request.GET.get('sno'))
        teachers = []
        if sno:
            subject = Subject.objects.only('name').get(no=sno)
            teachers = Teacher.objects.filter(subject=subject).order_by('no')
        return render(request, 'teachers.html', {
            'subject': subject,
            'teachers': teachers
        })
    except (ValueError, Subject.DoesNotExist):
        return redirect('/')
```
