## Principais pontos técnicos de sites de comércio eletrônico

### Modelos de negócios

1. B2B: empresa para empresa. Ambos os lados da transação são empresas (negócios). O caso mais típico é o Alibaba.
2. C2C: pessoa a pessoa, por exemplo: Taobao e Renrenche.
3. B2C: business to person, por exemplo: Vipshop e Jumei.
4. C2B: pessoa para empresa. Os consumidores primeiro apresentam a demanda e depois as empresas organizam a produção de acordo com essa demanda, por exemplo: Shangpin Home Collection.
5. O2O: on-line para off-line. Combina oportunidades de negócios offline com a internet, de modo que a internet se torna uma plataforma para transações offline, por exemplo: Meituan Waimai e Ele.me.
6. B2B2C: business to business to person, por exemplo: Tmall e JD.

### Destaques de requisitos

1. Lado do usuário
   - Página inicial: categorias de produtos, carrossel de banners, notícias de rolagem, carregamento em cascata, recomendações, descontos, produtos mais vendidos e assim por diante.
   - Usuário: login (login de terceiros), registro, logout, funções de autoatendimento, como informações pessoais, histórico de navegação, endereço de entrega e assim por diante.
   - Produto: categoria, lista, detalhes, pesquisa, pesquisa quente, histórico de pesquisa, adicionar ao carrinho, favorito, seguir, comentar e assim por diante.
   - Carrinho de compras: visualizar, editar (alterar quantidade, excluir mercadorias, limpar tudo).
   - Pedido: enviar pedido (pagamento), histórico de pedidos, detalhes do pedido, comentários do pedido e assim por diante.
2. Lado do administrador
   - CRUD das principais entidades comerciais.
   - Tarefas agendadas, periódicas e não periódicas, como processamento de pedidos não pagos e emissão de alarmes para eventos anormais encontrados durante a coleta de dados.
   - Funções de relatório, como importar e exportar arquivos Excel e PDF e mostrar estatísticas de ECharts no frontend.
   - Controle de permissão, como RBAC, lista branca, lista negra e assim por diante.
   - Fluxo de negócios, como iniciar um processo de reembolso. Mecanismos de fluxo de trabalho comuns incluem Activity, Airflow e Spiff.
   - Serviços de terceiros, como mapas, SMS, logística, pagamento, verificação de nome real, previsão do tempo, monitoramento e armazenamento em nuvem.

### Login de terceiros

Login de terceiros significa usar uma conta de um site de terceiros, geralmente um site social bem conhecido, para fazer a verificação de login. O objetivo principal é obter informações relacionadas ao usuário desse site de terceiros. Exemplos na China são QQ e Weibo, e exemplos fora da China são Google e Facebook. A maioria dos sistemas de login de terceiros usa o protocolo [OAuth](https://en.wikipedia.org/wiki/OAuth). É um padrão aberto sobre autorização. O proprietário dos dados informa ao sistema que concorda em permitir que um aplicativo de terceiros entre no sistema e obtenha os dados. O sistema então cria um token de acesso de curto prazo para substituir a senha do aplicativo de terceiros. É amplamente utilizado e a versão 2.0 geralmente é usada agora.

Sobre a diferença entre um **token** e uma **senha**, podemos simplesmente resumir três diferenças:

1. Um token é de curto prazo. Ele expirará automaticamente e o usuário não poderá alterá-lo sozinho. Uma senha geralmente é válida por um longo período. Se o usuário não alterar, isso não mudará.
2. Um token pode ser revogado pelo proprietário dos dados e torna-se inválido imediatamente. Uma senha geralmente não pode ser revogada por outras pessoas.
3. Um token possui um escopo de permissão. Para serviços de rede, um token somente leitura é mais seguro do que um token de leitura e gravação. Uma senha geralmente tem permissão total.

Assim, com um token, um aplicativo de terceiros pode obter permissão, mas essa permissão ainda poderá ser controlada a qualquer momento e não colocará em risco a segurança do sistema. Esta é a vantagem do protocolo OAuth.

#### Fluxo de autorização do OAuth 2.0

1. Depois que o usuário abre o cliente, o cliente solicita ao usuário, que é o proprietário do recurso, que dê autorização.
2. O usuário concorda em dar autorização ao cliente.
3. O cliente usa a autorização da etapa anterior para solicitar um token de acesso do servidor de autenticação.
4. Após autenticar o cliente, o servidor de autenticação emite o token de acesso.
5. O cliente usa o token de acesso para solicitar recursos ao servidor de recursos.
6. O servidor de recursos confirma que o token de acesso está correto e concorda em abrir o recurso para o cliente.

### Aquecimento do cache e consulta ao cache

#### Aquecimento do cache

O aquecimento do cache significa carregar dados no cache antecipadamente quando o servidor é iniciado. Para isso, podemos escrever uma subclasse de `AppConfig` no módulo `apps.py` da aplicação Django e substituir o método `ready()`, conforme mostrado abaixo.

```python
import pymysql

from django.apps import AppConfig
from django.core.cache import cache

SELECT_PROVINCE_SQL = 'select distid, name from tb_district where pid is null'


class CommonConfig(AppConfig):
    name = 'common'

    def ready(self):
        conn = pymysql.connect(host='1.2.3.4', port=3306,
                               user='root', password='pass',
                               database='db', charset='utf8',
                               cursorclass=pymysql.cursors.DictCursor)
        try:
            with conn.cursor() as cursor:
                cursor.execute(SELECT_PROVINCE_SQL)
                provinces = cursor.fetchall()
                cache.set('provinces', provinces)
        finally:
            conn.close()
```

Em seguida, você também precisa escrever o seguinte código no `__init__.py` do aplicativo.

```python
default_app_config = 'common.apps.CommonConfig'
```

Ou cadastre a aplicação no `settings.py` do projeto.

```python
INSTALLED_APPS = [
    ...
    'common.apps.CommonConfig',
    ...
]
```

#### Cache de consulta

Use um decorador personalizado para armazenar em cache os resultados da consulta.

```python
from pickle import dumps, loads

from django.core.cache import caches

MODEL_CACHE_KEY = 'project:modelcache:%s'


def my_model_cache(key, section='default', timeout=None):
    """Decorator that implements model caching."""

    def wrapper1(func):

        def wrapper2(*args, **kwargs):
            real_key = '%s:%s' % (MODEL_CACHE_KEY % key, ':'.join(map(str, args)))
            serialized_data = caches[section].get(real_key)
            if serialized_data:
                data = loads(serialized_data)
            else:
                data = func(*args, **kwargs)
                cache.set(real_key, dumps(data), timeout=timeout)
            return data

        return wrapper2

    return wrapper1
```

```python
@my_model_cache(key='provinces')
def get_all_provinces():
    return list(Province.objects.all())
```

class CartItem(object):
    """An item in the shopping cart."""

    def __init__(self, sku, amount=1, selected=False):
        self.sku = sku
        self.amount = amount
        self.selected = selected

    def total(self):

class ShoppingCart(object):
    """Shopping cart."""

    def __init__(self):
        self.items = {}
        self.index = 0

    def add_item(self, item):
        if item.sku.id in self.items:
            self.items[item.sku.id].amount += item.amount
        else:
            self.items[item.sku.id] = item

    def remove_item(self, sku_id):
        if sku_id in self.items:
            self.items.remove(sku_id)

    def clear_all_items(self):
        self.items.clear()

    @property
    def cart_items(self):
        return self.items.values()

    @property
    def cart_total(self):
        total = 0
        for item in self.items.values():
            total += item.total
        return total

### Venda relâmpago e vendas exageradas

1. Venda relâmpago: uma venda relâmpago geralmente significa que uma simultaneidade muito alta deve ser tratada em um tempo muito curto. O sistema precisa suportar um tráfego que pode ser cem vezes maior que o tráfego normal em um curto espaço de tempo. Portanto, a arquitetura de venda relâmpago é um problema relativamente complexo. Suas ideias centrais são controle de tráfego e otimização de desempenho, e precisa da cooperação de todas as partes, do frontend ao backend. No frontend, o JavaScript pode ser usado para fazer uma contagem regressiva, evitar envios duplicados e limitar atualizações frequentes. O controle de tráfego significa principalmente permitir que apenas uma pequena parte do tráfego entre no serviço de back-end, porque apenas um pequeno número de usuários pode finalmente ter sucesso na venda relâmpago. Na arquitetura física, cache e filas de mensagens podem ser usados ​​para otimizar o sistema. O cache é usado porque há muitas leituras e poucas gravações. O inventário pode ser colocado no Redis e a primitiva `DECR` pode ser usada para reduzir o estoque. Redis também pode ser usado para limitação de taxa. O motivo é semelhante à limitação da frequência com que os códigos de verificação por telefone podem ser enviados. A função mais importante da fila de mensagens é suavizar os picos de tráfego e desacoplar os nós upstream e downstream. Além disso, deve ser utilizado um design de serviço sem estado, para que o dimensionamento horizontal seja mais fácil.
2. Venda excessiva: suponha que o estoque de um produto seja 1. Ao mesmo tempo, o usuário 1 e o usuário 2 compram esse produto simultaneamente. Após o usuário 1 enviar o pedido, o estoque é alterado para 0. Mas o usuário 2, sem saber disso, também envia o pedido, e o estoque é alterado para -1. Este é o problema do exagero. Existem três ideias comuns para resolvê-lo:
   - Bloqueio pessimista: ao consultar a quantidade do produto, utilize `select ... for update` para bloquear os dados. Desta forma, quando o usuário 1 verifica o estoque, o usuário 2 fica bloqueado e não consegue ler a quantidade do estoque. O usuário 2 poderá continuar somente depois que o usuário 1 confirmar ou reverter a atualização do inventário. Isso resolve o problema do exagero. Mas para produtos com acesso simultâneo muito alto, esse método tem um desempenho muito ruim. No desenvolvimento real, o bloqueio só pode ser considerado quando o estoque é menor que um determinado valor, mas em geral esse método não é uma escolha muito boa.
   - Bloqueio otimista: não bloqueie ao consultar a quantidade. Ao atualizar o estoque, exija que a quantidade do produto ainda seja igual à quantidade lida anteriormente. Caso contrário, significa que outra transação já atualizou o inventário e a solicitação deverá ser enviada novamente.
   - Tente reduzir o estoque diretamente: combine a consulta (`select`) e a atualização (`update`) em uma operação SQL. Ao atualizar o inventário, adicione uma condição como `inventory >= purchase_quantity` ou `inventory - purchase_quantity >= 0` à condição `where`. Este método requer que o nível de isolamento da transação seja confirmado para leitura.

### Pesquisa de texto completo

#### Escolhendo uma solução

1. Use a função de consulta difusa do banco de dados: a eficiência é baixa, sempre é necessária uma varredura completa da tabela e a segmentação de palavras não é suportada.
2. Use a função de pesquisa de texto completo do banco de dados: antes do MySQL 5.6, ele era adequado apenas para o mecanismo MyISAM. A operação de pesquisa e outras operações DML são acopladas dentro do banco de dados, o que pode tornar a operação de pesquisa muito lenta. Quando a quantidade de dados atinge o nível de um milhão, o desempenho cai muito e o tempo de consulta torna-se longo.
3. Use um mecanismo de pesquisa de código aberto: os dados do índice e os dados originais são separados. ElasticSearch ou Solr podem ser usados ​​para fornecer um serviço de indexação externo. Se a pesquisa de texto completo de alta simultaneidade não for necessária, o Python Whoosh puro também pode ser considerado.

#### ElasticSearch

ElasticSearch é um banco de dados de documentos distribuído e um mecanismo de pesquisa e análise de texto completo de código aberto altamente escalonável. Permite o armazenamento, pesquisa e análise de grandes quantidades de dados, e esse processo ocorre quase em tempo real. Geralmente é usado como mecanismo e tecnologia subjacente que alimenta funções e requisitos de pesquisa complexos. Sites conhecidos como Wikipedia, Stack Overflow e GitHub usam ElasticSearch.

O mecanismo subjacente do ElasticSearch é o mecanismo de pesquisa de código aberto [Lucena](https://lucene.apache.org/). Mas usar o Lucene diretamente é muito problemático. Você mesmo deve escrever o código para chamar suas interfaces, e ele suporta apenas Java. ElasticSearch é como um pacote completo em torno do Lucene. Ele fornece interfaces API estilo REST e oculta diferenças de linguagem de programação por meio de acesso HTTP. ElasticSearch constrói um [índice invertido](https://zh.wikipedia.org/zh-hans/%E5%80%92%E6%8E%92%E7%B4%A2%E5%BC%95). Mas seu tokenizador integrado quase não oferece suporte para segmentação de palavras chinesas, portanto, o plug-in `elasticsearch-analysis-ik` precisa ser instalado para fornecer tokenização chinesa.

Além do ElasticSearch, Solr e Whoosh também podem ser usados ​​para fornecer serviços de mecanismo de busca. Num projeto Django, as seguintes soluções podem ser consideradas:

- `haystack` (`django-haystack` / `drf-haystack`) + `whoosh` + `Jieba`
- `haystack` (`django-haystack` / `drf-haystack`) + `elasticsearch`
- `requests` + `elasticsearch`
- `django-elasticsearch-dsl`
